## BUG-088 · 书籍 epub 内容从不上传到云端（磁盘无 .epub，`epubPath` 是纯文件名恒不存在）
- **报告**：2026-06-06（用户：开了「同步书籍文件」但云盘书文件夹只有 .json、无 epub）。真机 Drive API + 本地磁盘取证定位。
- **真实性**：✅ **真 bug（沿真实代码 + 真机取证）**。取证：真调 Drive API——4 个书文件夹**只有进度/统计 .json、零个 .epub**；本地磁盘——所有书目录 `original.epub` 数量=0（书是**解压成目录**存的，`C:\Users\…\Documents\hoshi_books\<id>\` 含 mimetype/META-INF/OPF/text/images）。根因：`SyncManager._exportContentIfMissing`（`sync_manager.dart`）做 `File(book.epubPath).existsSync()` 才上传，但 `epub_books.epub_path` 存的是**纯原始文件名**（`EpubImporter` 第 175 行 `epubPath: fileName`，如 `謎解き… - 東川篤哉.epub`），`File('纯文件名')` 解析到进程 CWD、**永远不存在** → `existsSync()=false` → **静默跳过 epub 上传**（开不开开关都一样）。对称地 `_importContentIfMissing` 也用 `!File(book.epubPath).existsSync()`（恒 true）→ 把 epub 下载到那个纯文件名路径（污染 CWD 且从不重解压）。即磁盘上**根本没有单个 .epub 可传**，内容同步对 epub 完全失效。
- **[x] ① 已修复** — 本提交（自含修复+测试+本条目）。**根因修（重打包解压目录成 epub 再传）**：新增可测纯函数 `repackageExtractedEpub(extractDir, outputPath)`——用 `ZipFileEncoder.addDirectory(dir, includeDirName: false)` 把 `book.extractDir`（合法 epub 布局）打成根目录对齐的 .epub（mimetype/META-INF 在 zip 根，对端 `EpubImporter` 能直接重导）。`_exportContentIfMissing` 改为：`extractDir` 存在 → 重打包到系统临时 .epub → `uploadContentFile` 上传 → finally 删临时（不再看恒假的 `epubPath`）。`_importContentIfMissing` 的 epub 分支改为按 `extractDir` 是否存在判断「内容是否已在本地」：在=跳过；不在=记日志跳过（远端独有书由 `importRemoteBookFolder` 负责重导），**消除原来下载到纯文件名污染 CWD 的死路径**。音频分支用真实绝对路径，本就正确，不动。
- **[x] ② 已加自动化测试** — `hibiki/test/sync/epub_content_repackage_test.dart`：构造假解压目录(mimetype+META-INF/container.xml+OEBPS/content.opf+chapter)→ `repackageExtractedEpub` → `ZipDecoder` 解出，断言 ① mimetype/META-INF/container.xml/OEBPS/* 都在**zip 根**（无 `extracted/` 包裹前缀）② mimetype 内容正确 ③ 空/缺目录返回 false 不写文件。`test/sync/` 全量 314 绿，analyze 0。
- **备注**：sync 内容类。代码正确 + 无回归；**真机复测待用户**：开「同步书籍文件」后云盘书文件夹出现 `<书名>.epub`，且另一台设备能下载并正确导入该书（重打包→重导往返）。配合 [[BUG-087]] 的分块续传，大 epub 也能传完。注：重打包用默认 deflate（mimetype 未严格 STORE-first），对 Hibiki 自身 `EpubImporter`（ZipDecoder，顺序无关）无碍；如需对外部 epub 校验器严格合规可后续让 mimetype STORE-first。
