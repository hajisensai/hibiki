## BUG-380 · 滚动模式阅读进度只在滑动停下才更新(JS纯尾沿200ms去抖)
- **报告**：2026-06-21（用户：滚动模式下阅读进度只在滑动停下才更新，应该边滑边实时变）
- **真实性**：✅ 真 bug。根因在 setup 脚本的 scroll reporter：`_onReaderScrollEvent`（`hibiki/lib/src/pages/implementations/reader_hibiki/webview.part.dart:923`，BUG-213 引入）是**纯尾沿去抖**——每个 `scroll` 事件都 rAF 后 `clearTimeout` 把 200ms `setTimeout` 推后，滑动期间 `_reportReaderScroll()` 永不执行，只在滑动停下 200ms 后才回传一次 `callHandler('onReaderScroll')`。Dart 端 `_handleReaderScroll`→`_refreshProgress`（`navigation.part.dart`）本身不节流、UI 实时读 `_progressCurrentChars/Total`，瓶颈唯一在 JS 这 200ms 纯尾沿去抖。跨章能更新是另一条通道（`_onRestoreComplete`→`_refreshProgress`），与此无关。
- **[x] ① 已修复** — ①JS 改为「rAF 节流 + 尾沿补一发」（`webview.part.dart:923`）：滑动中每个动画帧最多回传一次（约 16ms/次，肉眼连续），滑停 120ms 后短尾沿再补发一次最终静止位置。②Dart 端新增 `_refreshProgressFromScroll`（`navigation.part.dart`）的「在飞 + 待重跑」coalesce 守卫（字段 `_scrollProgressInFlight`/`_scrollProgressPending`，`reader_hibiki_page.dart:620` 附近）：一次 `_refreshProgress` 在途（含较重的 `hoshiProgressDetails` 遍历全章 TextNode + caretRangeFromPoint）时再来的滚动只置 pending，飞完补跑一次，避免高频上报堆积 evaluateJavascript。`_handleReaderScroll` 门控通过后改调 `_refreshProgressFromScroll()`。轮询/恢复路径仍直接调 `_refreshProgress` 不受守卫影响。
- **[x] ② 已加自动化测试** — 更新 `hibiki/test/reader/reader_inchapter_progress_scroll_test.dart`：源码守卫锁住「rAF 节流（`if (!_progressScrollRaf)`，rAF 回调直接 `_reportReaderScroll()`）+ 尾沿 120ms 补发 + 不得回退 `}, 200);` 纯尾沿」「`_handleReaderScroll` 门控后走 `_refreshProgressFromScroll()`」「`_refreshProgressFromScroll` 是 inFlight/pending/whenComplete coalesce」；既有 `readerScrollProgressRefreshAllowed` 纯函数真值表保留。撤销修复 → 4 个断言转红，已验证 red→green。
- **备注**：性能：rAF 节流约 16ms/次上报 + Dart coalesce 守卫防 evaluateJavascript 堆积；落库仍受 `_debouncedSavePosition` 500ms 去抖保护，DB 写压力不变。**待真机肉眼复测**：滚动模式边滑边看进度条/百分比连续更新（连续模式 window 滚动 + 分页模式 body 滚动两路径）。
