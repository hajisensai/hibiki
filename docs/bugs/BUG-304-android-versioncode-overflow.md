## BUG-304 · Android versionCode 经 ×1,000,000 公式溢出 int32/超 21 亿上限，beta/release 包建不出
- **报告**：2026-06-16（用户：TODO-414）
- **真实性**：✅ 真 bug。根因双层：①`.github/workflows/release.yml`（修复前 ~81）06-13 引入的 `ANDROID_BUILD_NUMBER=$((PUBSPEC_BUILD * 1000000 + RELEASE_SEQUENCE))`，把 `--build-number` 顶到约 6800 万级（`PUBSPEC_BUILD=68` × 1e6 + `RELEASE_SEQUENCE=4980` = `68_004_980`）；②`hibiki/android/app/build.gradle`（修复前 fat ~148 / split ~166）的 `output.versionCodeOverride = 100 * versionCode (+ offset)` 再 ×100，得 `100 × 68_004_980 + 2 = 6_800_498_002`。该 versionCode 约 **66 亿**，既溢出 int32（上限 2_147_483_647）又超 Android 官方 versionCode 上限 2_100_000_000，**beta/release（手动 workflow_dispatch 走 `flutter build apk --release --split-per-abi`）的 Android 包根本建不出来**。历史真实已安装的最高 versionCode 约 6603（旧 `100 × build` 公式），66 亿这档从未成功构建发布过，故无需用户卸载。
- **[x] ① 已修复** — 见本轮 commit `fix(android): unify versionCode to monotonic git-rev-count + base`。统一为单调不溢出来源：(a) `release.yml` 去掉 `* 1000000`，`ANDROID_BUILD_NUMBER=$RELEASE_SEQUENCE`（直接用 `git rev-list --count HEAD` 的提交数；`PUBSPEC_BUILD` 仅留作日志）；(b) `hibiki/android/app/build.gradle` 顶部新增 `def versionCodeBase = 1000000000`（一次性迁移基数，1e9 高于历史最高 ~6603、又留约 1099 万提交头寸，int32/Android 双安全）+ `def maxVersionCode = 2100000000`，两处 `versionCodeOverride` 改 `versionCodeBase + 100 * versionCode + offset`，并加上限断言 `if (computed > maxVersionCode) throw new GradleException(...)` 防未来越界 fail-fast；(c) `.github/workflows/main.yml` 的 `actions/checkout` 加 `fetch-depth: 0`（否则浅 checkout 致 `git rev-list --count` 偏小、versionCode 与 release.yml 不一致），新增「Resolve shared release sequence」步骤把 `RELEASE_SEQUENCE` 写进 `$GITHUB_ENV`，debug/release-verify/release 三处 `flutter build apk` 全加 `--build-number "$RELEASE_SEQUENCE"`（验证构建与 release.yml 同源单调）；(d) `tool/check_release_policy.ps1` 把旧 `* 1000000` 断言更新为新公式 + build.gradle base/ceiling/computed 断言；(e) `docs/agent/build.md` 同步 versionCode 措辞（保留 cross-workflow release sequence / git rev-list --count HEAD / single GitHub Release 三串守卫词）。实测 `--build-number 4969`：fat `1_000_000_000 + 100×4969 = 1_000_496_900`，arm64 `+2 = 1_000_496_902`，均 < 2.1e9；`--build-number 4970` 单调 +100；debug/release 同 build-number 一致。
- **[x] ② 已加自动化测试** — `hibiki/test/build/release_workflow_diagnostics_guard_test.dart` 新增 test「TODO-414: Android versionCode is the monotonic git-rev-count + base, never the overflowing *1000000 formula」：断言 release.yml 不再含 `* 1000000`、含 `ANDROID_BUILD_NUMBER=$RELEASE_SEQUENCE`；main.yml 含 `fetch-depth: 0` + `RELEASE_SEQUENCE=$(git rev-list --count HEAD)` + ≥2 处 `--build-number "$RELEASE_SEQUENCE"`；build.gradle 含 `def versionCodeBase = 1000000000`、`def maxVersionCode = 2100000000`、`output.versionCodeOverride = computed`、≥3 处 `throw new GradleException`。`tool/check_release_policy.ps1` 同步加固（Forbid `PUBSPEC_BUILD \* 1000000` + Require base/ceiling/computed）。
- **备注**：versionCode 单调性现由「提交数（每 commit +1）」保证而非语义版本 `+build`；`pubspec.yaml` 的 `+build` 退化为日志/可读版本标识，不再进 versionCode。两条发布 workflow（Android release.yml + 桌面 release-desktop.yml）继续共享同一 `git rev-list --count HEAD` 序列，符合 cross-workflow release sequence 规则。真机/真 CI release 构建出包待用户在发布环节验证。
