## BUG-423 · 调试日志框选拖拽未响应卡死
- **报告**：2026-06-25（用户：「我在框选调试日志的时候未响应卡死了」）
- **真实性**：✅ 真 bug。同族 TODO-762/BUG-119/BUG-413/TODO-806 延续。根因 `hibiki/lib/src/utils/components/hibiki_material_components.dart`（`_HibikiLogPanelState`/`logSelectionScrollDecision` 边缘自动滚动放行分支，旧 :2366-2369）。
  - 链路：日志正文（调试/错误日志页共用 `HibikiLogPanel`）TODO-762 起改为 `SelectionArea` + `ListView.builder` 懒构造行。拖拽框选贴边时，旧判据放行「边缘自动滚动」→ `ListView` 持续构造视口外新行 → `SelectionArea` 注册的 `Selectable` 集合随拖拽时长**单调膨胀** → 每次 `onPointerMove`（:2282）对全部已注册 `Selectable` 做命中/边界重算占满 UI 线程 → 未响应卡死。
  - 放大器：日志行 `Text(softWrap: true)`（旧 :2305）把一行拆成多视觉行，`SelectionArea` 单行选区命中要对每段 wrap 子矩形逐一求交，命中成本随行长放大（也是 TODO-806 框选坐标错位来源）。
- **[x] ① 已修复** — 根因修复：拖拽选区激活且有实质位移时一律拦掉程序化滚动，**删掉「边缘自动滚动」放行分支**，`Selectable` 集合钉在视口内有限行，膨胀链路从根上断掉；判据 `logSelectionScrollDecision` 简化为仅 `pointerSelectionActive`/`delta` 两参（消除按指针位置/方向的特殊情况）。`_LogSelectionScrollController` 同步删除不再使用的指针几何 / 手动滚动状态（`_pointerY`/`_viewportHeight`/`_userScrolledDuringSelection`/`_markUserScroll`），`_LogSelectionScrollPosition` 删 `applyUserOffset`/`pointerScroll` override（手动滚动本就不该拦，走默认）。日志行 `softWrap: true → false`（降低单行命中成本，兼修 TODO-806 坐标错位）。提交 6f9f47ee5。
  - **BUG-119 行为契约取舍**：拖拽期边缘自动滚动被禁用。Never break userspace 的边界——滚到视口外去框选本就不可靠（`SelectionArea` 不为滚出视口的行保留选区），「复制整段日志去排障」这一真实用途已由常驻「复制全部」按钮（绕开 `SelectionArea` 走 `widget.log` 全量，:2217/:2314）完整覆盖，禁用边缘自动滚动不损失任何可用能力。BUG-119「拖拽选区不被程序化 jumpTo/animateTo 拽回」的不变式仍由同一 gate 守住（拖拽期一律拦，是旧拦截规则的超集）。手动滚动（拖滚动条 / 滚轮）不经判据，不受影响。
- **[x] ② 已加自动化测试** — `hibiki/test/widgets/log_panel_scroll_select_guard_test.dart`：①纯函数真值表 `logSelectionScrollDecision`（拖拽未激活→放行；位移可忽略→放行；拖拽激活 + 实质位移上/下/小位移→一律 BLOCK，钉死「边缘自动滚动放行分支不复活」）；②源码守卫追加 `softWrap: false` 在场、`softWrap: true` 不在场断言。提交 6f9f47ee5。
- **备注**：需真机/真渲染复测原始失败路径——在真实运行的 app「调试日志」页对大日志按住鼠标拖拽框选（含拖到顶/底边缘）不再卡死、UI 持续响应；「复制全部」仍能拿整段日志。headless 难以稳定复现真实选区+滚动几何，故在最强可落地层（纯函数真值表 + 源码守卫 + widget 结构）守住不变式。

## TODO-934 · 续记：BUG-423 一刀切的交互回归——拖到边区不响应（已根因修复并恢复边缘自动滚动）
- **报告**：2026-06-28（用户：「框选拖到边区外不响应」）。这是 BUG-423 修复的**交互回归**，不是新 bug：BUG-423 为止卡死「一刀切禁用拖拽期所有程序化滚动（含边缘自动滚动）」，代价就是拖到边区不再滚动延伸选区。
- **BUG-423 当年根因诊断不准确（已沿 SDK 源码核实）**：BUG-423 把「`SelectionArea` 的 `Selectable` 集合单调膨胀」当根因，但 Flutter 3.44.0 SDK 证据表明 `selectables` **大小有界**——`ListView.builder` 离屏行被 `RenderSliverMultiBoxAdaptor.collectGarbage`（`sliver_multi_box_adaptor.dart:586`）回收，`RenderParagraph.dispose`（`paragraph.dart:562`）经 `_removeSelectionRegistrarSubscription`（`paragraph.dart:501`）把该行 `Selectable` 从 `MultiSelectableSelectionContainerDelegate.remove`（`selectable_region.dart:2411`）注销，集合被钉在「视口 + cacheExtent」内、不随滚动距离膨胀（widget 守卫实测滚过 5000 行渲染行数恒为 23）。真正的卡死放大器是 **`softWrap:true` 长行**：其 `getBoxesForSelection`（`text_painter.dart:1671` / `paragraph.dart:1521`）成本随该行换行成的视觉行数 O(N) 放大，而边缘自动滚动每帧持续重算选区几何。BUG-423 的 `softWrap:false` + BUG-448 的 `ConstrainedBox`+`ClipRect`（把每行布局宽度钉死在视口内）**已经把每帧几何成本压成 O(视口可见内容)、与滚动距离无关**——卡死链路已从根上断掉。
- **根因修复（TODO-934）**：恢复边缘自动滚动但切断膨胀链路（其实链路已断，恢复是安全的）。`hibiki/lib/src/utils/components/hibiki_material_components.dart`：`logSelectionScrollDecision` 增加 `animated` 维度——拖拽激活 + 实质位移时，**动画滚动（`animateTo` = `EdgeDraggingAutoScroller` 边缘自动滚动，每帧 ≤20px 有界一小步，`scrollable_helpers.dart:247/305`）放行**；**瞬跳滚动（`jumpTo` = 键盘 granular/directional 扩展的 `_jumpToEdge` 拽回，`scrollable.dart:1419`）仍拦**（BUG-119 不变式）。`_allowProgrammaticScroll` 加 `animated` 参，controller/position 两对 `animateTo` 传 `animated:true`、两对 `jumpTo` 传 `animated:false`。纯 `SelectionArea`+`Text`（无 `EditableText`）结构下，鼠标拖拽框选期间唯一的程序化滚动来源就是边缘自动滚动的 `animateTo`，`bringIntoView` 是空实现（`selectable_region.dart:1871`），故放行 `animateTo` 不会引入任何拽回。
