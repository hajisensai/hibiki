## BUG-334 · TODO-572: 视频内封字幕首次打开常加载不出来，需重开一次

- **报告**：2026-06-19（用户 B03 验收第 10 条：`这个加载很怪，总会有没加载出来的情况，需要重新打开一次才行`）——内封字幕（视频容器里的字幕轨）首次打开视频时经常没出来，退出重开一次才加载出来。
- **真实性**：✅ 真 bug，时序竞态。根因在 `hibiki/lib/src/media/video/video_player_controller.dart`：`load()` 在 `player.open()` 之后立即 `unawaited(_loadEmbeddedSubtitleIfNeeded(...))`（原 `:808`）后台抽内封文本字幕成 cue，但它**只跑一次、且过期判据是弱的 `_videoPath != videoPath`**。文本内封字幕的枚举不读 libmpv 轨道，而是走 ffmpeg 旁路 `ffmpeg -i`（`probeEmbeddedSubtitleTracks` `video_subtitle_source.dart:395`）。首次打开大容器（冷页缓存）时，该枚举与 `_applyLoad` 里同时 fire 的 `prewarmEmbeddedSubtitleCache`（`video_hibiki_page.dart:1773`）整轨抽取（`ffmpeg -map`）以及 libmpv 正在 demux 播放**三方争用磁盘 IO**，`ffmpeg -i` 探测可能超时（`returnCode==null`）→ 返回 0 轨 → `enumerationTimeout` → 整条链路当「无内封字幕」终态处理。这是个**瞬态争用失败被当终态**：退出重开时抽取缓存已命中、OS 页缓存让枚举瞬时、cue 可能已落库，三重冗余让字幕出来。BUG-303（TODO-412）只放宽了枚举超时窗口，未消除时序，自己备注「若真机仍空看 `embedded enumeration timed out` 日志」——本单即放宽超时后真机首开仍偶发空的续单。与 BUG-313（TODO-521 章节首载竞态）同构，但章节用 libmpv duration-ready 信号重读，文本字幕走 ffmpeg 旁路没有等价的「争用消退后重枚举」。
- **[x] ① 已修复** — 把「枚举加载 + 瞬态失败就绪后重试一次」抽成 player-agnostic 纯协调器 `loadDefaultTextEmbeddedSubtitleCuesWithReadinessRetry`（`hibiki/lib/src/media/video/video_subtitle_source.dart`）：首次加载若是**瞬态失败**（`enumerationTimeout` / `enumerationFailed` / `emptyCues`——「本该有字幕但这次没拿到」，由纯谓词 `isTransientDefaultEmbeddedSubtitleLoad` 判定）就 `await waitForReady()` 等就绪信号后**重试一次**；终态（`noEmbeddedTracks` / `noTextTrack` / `missingFile` / `loaded`）不重试；每个 await 边界用注入的 `isStillCurrent()` 丢弃过期结果。`VideoPlayerController._loadEmbeddedSubtitleIfNeeded` 改为接收 `player` + `loadToken`，调协调器时注入 `waitForReady: () => _waitUntilSubtitleTracksReady(player)`（监听 `player.stream.tracks`，等容器字幕区 demux 进页缓存、三方争用消退，5s 超时）与 `isStillCurrent: () => _isCurrentLoad(player, loadToken)`（player identity + load token 双判据，替换原弱判据 `_videoPath != videoPath`，复用同一 player 换集时也能丢弃旧片字幕）。`load()` 触发点改传 `player` + `loadToken`（`video_player_controller.dart` `load()` 末尾）。复用既有 `_waitUntilSubtitleTracksReady` helper、既有 `prewarmEmbeddedSubtitleCache` 单趟全轨缓存（TODO-104）与外挂字幕路径均不动。
- **[x] ② 已加自动化测试** — `hibiki/test/media/video/video_subtitle_source_test.dart` group「首开瞬态争用重试（TODO-572）」7 例行为测试（注入 `loadOnce` 桩，不依赖真 Player）：`isTransientDefaultEmbeddedSubtitleLoad` 全状态分类、首次枚举超时→等就绪重试一次→第二次拿到 cue、首次即 loaded 不等就绪不重试、终态 `noEmbeddedTracks` 不重试、重试后仍瞬态空只重试一次（有界、不无限）、等就绪期间换片（`isStillCurrent` 变假）不重试丢弃旧结果、首次加载后即换片在第一个 await 边界丢弃。`hibiki/test/pages/video_subtitle_loading_overlay_guard_test.dart` 加源码契约守卫，断言 controller `load()` 传 `player`+`loadToken`、`_loadEmbeddedSubtitleIfNeeded` 经协调器加载、就绪信号是 `_waitUntilSubtitleTracksReady(player)`（非固定延迟）、过期判据是 `_isCurrentLoad(player, loadToken)` 双判据、不再退回单次直调。
- **备注**：未用固定延迟/无限重试修竞态——重试有界（最多一次）、仅在明确识别的瞬态 IO 争用状态触发、由 libmpv 字幕轨真实就绪信号触发、loadToken 防过期。属对「磁盘 IO 争用 + ffmpeg 探测耗时」这类不可控外部时序的就绪后重读，符合根因修复原则。
