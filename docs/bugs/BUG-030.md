## BUG-030 · 「管理音频来源」对话框里光标停在 URL 输入框时按方向键上下移不动焦点
- **报告**：2026-06-04（用户，附 Windows「管理音频来源」对话框截图：URL 输入框获得焦点，「这里按方向键上下动不了」）。
- **真实性**：✅ **真 bug（聚焦的单行文本框吞掉上下方向键，焦点被困在输入框里）**。沿真实路径定位：app 用键盘方向键做焦点导航。`home_page._handleKeyEvent`（`home_page.dart:163`）把方向键转成 `gamepadMoveFocusInDirection`，但**这是 home 页 `Focus` 节点上的处理器**；对话框是压在 home 之上的独立路由，其焦点作用域**不是** home 那个 `Focus` 的后代，所以对话框里走的是 Flutter 框架默认的方向焦点。默认路径下，焦点落在 `AudioSourcesDialog` 的 URL `AdaptiveSettingsTextField`（单行）时，框架的 `DefaultTextEditingShortcuts` 把每个方向键都映射成插入符意图、`EditableText` 一律消费——包括单行框里**插入符无处可去的上/下键**（消费成 no-op），于是上下键既不移插入符也不让焦点离开输入框 = 焦点被困死。证据：widget 探针在真实对话框里聚焦输入框后连按上/下，`primaryFocus` 恒为该 `EditableText`（行送其余控件正常：scope→chevron→switch→switch→输入框，唯独到输入框就卡住）。根因 = 聚焦文本框对「插入符用不到的方向键」也照单全收，把焦点导航锁死，而对话框缺一个能在框架文本快捷键之前抢下这些键的兜底处理器。
- **[x] ① 已修复** — 在 app 级 `wrapWithGlobalNavigation`（`global_navigation.dart`，位于 Navigator 之上、能收到从任意路由冒泡上来的按键，已处理手柄 B / 全局 Escape）补一个**外科式**方向键处理器 `_handleGlobalArrowFocus`：它**只在文本框聚焦时**介入，其余方向键一律 `ignored`、不碰任何现有行为（home 方向导航、阅读器翻页、滑块/下拉、对话框里框架默认方向 traversal 本就能走非文本框控件）。它在按键冒泡链上比 `DefaultTextEditingShortcuts` 更靠近焦点（本 wrapper 在 builder 里包 Navigator，框架文本快捷键在 WidgetsApp 更外层），故能在插入符之前抢下「插入符用不到的方向键」：左/右恒留插入符；上/下仅在**多行**框留插入符（换行导航），**单行**框上/下经 `gamepadMoveFocusInDirection`（与手柄 D-pad 同一 bootstrap + reading-order 兜底）放出去 = 让焦点逃离输入框。新增共享纯函数 `arrowTraversalDirection` / `focusedEditableText`（`gamepad_service.dart`）；`focusedEditableText` 用祖先查找解决 `EditableText` 把焦点托管给内层 `Focus(debugLabel:'EditableText')`、`primaryFocus.context.widget` 其实是那个 `Focus` 而非 `EditableText` 的坑（裸 `is EditableText` 永远 miss）。**附带（opus 审查 W1）**：`home_page._handleKeyEvent` 原用私有 `_arrowDirection` + `_isEditableFocused`（后者正是裸 `is EditableText`、恒 miss → home 搜索框 ←/→ 被当焦点移动而非光标移动的潜伏 bug）；统一改用上述两个共享 helper，删私有副本，既消重又顺手修掉 home 搜索框光标潜伏 bug。消除「单行文本框困死方向焦点」+「两份方向键解释器」两处特殊情况，非补丁式绕过。`flutter analyze` 0 issue、`dart format` 0 改动。
- **[x] ② 已加自动化测试** — `test/shortcuts/global_navigation_test.dart` 新增 4 条真 widget 行为测试（`wrapWithGlobalNavigation` 包 `MaterialApp`，[按钮]/[单行或多行文本框]/[按钮] 布局，点聚焦文本框）：① 单行框按 ↑ → 焦点离开文本框并落到上方按钮；② 单行框按 ↓ → 焦点离开并落到下方按钮（修复前两条都因焦点困在框内而红，实测探针证实）；③ 单行框按 ←/→ → 焦点**仍在**文本框（插入符不被劫持）；④ 多行框按 ↓ → 焦点**仍在**文本框（换行导航保留）。`test/pages/home_page_gamepad_static_test.dart` 的顺序守卫随 `_arrowDirection`→`arrowTraversalDirection` 改名同步更新（断言不变）。`test/shortcuts` + `test/widgets`（focus traversal / gamepad / theme swatch）+ `audio_sources_dialog_page_test` 全量绿无回归。
- **备注**：桌面键盘焦点导航类。代码 + 单测绿、`flutter analyze` 0 issue、`dart format` 0 改动。修复对全 app 生效（任何路由里聚焦单行框，上/下都能逃离；更深层真正消费方向键的控件——滑块/下拉/阅读器翻页/多行框插入符——因更早处理仍优先）。opus 审查 🟢 可合并（W1 已落地，W2 reader popup-header ←/→ 因改外科式只碰文本框而**自然消解**——非文本框方向键完全不经本处理器）。**真机肉眼复测原始失败路径**（Windows 上开「管理音频来源」→ 点 URL 输入框 → 按 ↑ 焦点上移到来源行、按 ↓ 下移到「添加本地音频数据库」/底栏按钮；←/→ 仍在框内移光标）待用户后补。**并发 agent 注记**：本工作树有并发 agent 已占 BUG-028/029 + 未提交的有声书/backup 改动（`audiobook_controller.dart` 等）；全量 `flutter test` 偶现 `test/sync/backup_service_test.dart`（每跑失败用例不同）与 `app_model_audio_sources_test`（顺序依赖）红，单独跑 audio 文件全绿、backup 为其固有 flaky，均与本键盘改动无代码经路交集。本次只 stage 本 bug 的 6 个文件（`global_navigation.dart` / `gamepad_service.dart` / `home_page.dart` / `global_navigation_test.dart` / `home_page_gamepad_static_test.dart` / 本文档）。
