## BUG-381 · 书籍图片右键复制图片菜单位置不跟界面缩放(坐标未经Overlay变换链映射)
- **报告**：2026-06-21（用户：书里图片右键「复制图片」按钮位置没根据界面大小修改，缩放后错位）
- **真实性**：✅ 真 bug（BUG-261 视频右键菜单的同型 bug，在阅读器图片菜单路径未修）。根因：`_showReaderImageContextMenuAtGlobalPosition`（`hibiki/lib/src/pages/implementations/reader_hibiki/chrome.part.dart:106`）拿到的 `globalPosition` 是真实屏幕坐标（右键路径 `_showReaderImageContextMenu` 用阅读器 State RenderBox `localToGlobal`，State 被 `HibikiAppUiScaleNeutralizer` 中和回净缩放=1 的真实视口空间；放大图路径传 `details.globalPosition`），但 `showMenu` 的 `RelativeRect` 落在它路由 Overlay 的坐标系，而该 Overlay 在全局 `HibikiAppUiScale` 的 FittedBox 之内（缩放后画布空间）。直接把真实屏幕坐标当画布坐标喂 `showMenu`，界面大小≠100% 时菜单偏离图片 factor≈scale。`_readerImageMenuScale` 只缩放菜单内容尺寸，不参与锚点。
- **[x] ① 已修复** — `_showReaderImageContextMenuAtGlobalPosition`（`chrome.part.dart:106`）改为先 `final Offset anchor = overlay.globalToLocal(globalPosition);` 把锚点从真实屏幕坐标沿真实渲染变换链映射到菜单宿主 Overlay 的本地坐标系（FittedBox 缩放被 render transform 自动吸收，对任意 scale 含自动模式自洽无残差；scale=1 时变换为单位阵，逐像素等价向后兼容），`RelativeRect` 改用 `anchor.dx/dy`。与 BUG-129/261 同范式（不读 scale 数值逆算）。菜单内容缩放 `menuScale` 不动。
- **[x] ② 已加自动化测试** — 更新 `hibiki/test/pages/reader_image_actions_guard_static_test.dart` 的「reader image context menu scales with reader chrome only」用例：断言菜单锚点必须经 `overlay.globalToLocal(globalPosition)` 映射、`Rect.fromLTWH` 用 `anchor.dx/anchor.dy`（不得用裸 globalPosition），且锚点不得乘 menuScale。撤销修复（改回 `globalPosition.dx/dy`）→ 转红，已验证 red→green。
- **备注**：右键菜单仅 Windows 平台（`isWindowsPlatform` 门控，移动端长按走系统分享面板不弹定位菜单）。**待真机肉眼复测**：Windows 调界面大小≠100% 后，书内图片右键 + 放大图右键的菜单都对准鼠标/图片。
