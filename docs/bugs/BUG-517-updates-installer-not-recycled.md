## BUG-517 · 更新安装包安装成功后未回收
- **报告**：2026-07-01（用户：）TODO-1089。现象：`%AppData%\Hibiki\Hibiki\updates` 里自动更新下载的安装包（Windows setup.exe）升级后一直残留，"还是没自动删除"。
- **真实性**：✅ 真 bug。根因是**安装成功缺少即刻回收回调**：Windows 更新经 Inno Setup handoff 安装成功、app 以新版重启后，`UpdateChecker.reconcilePendingWindowsInstallerHandoff`（`hibiki/lib/src/utils/misc/update_checker_release.dart:925`）只在 `WindowsUpdateHandoff.reconcile`（`hibiki/lib/src/utils/misc/update_handoff.dart:585,603`）里**删掉 `update-handoff.json` 标记**，被它安装的 setup.exe **从不在安装成功那一刻回收**。唯一的兜底是 `_cleanupOldApks`→`selectStaleUpdateArtifacts` 的 **7 天 GC**（`update_checker_release.dart:103` cutoff = 7 天前），而这个 GC 只在 `_check`（即 `scheduleCheck` 触发的更新检查）里跑（`update_checker_release.dart:237`）——关闭自动检查 / `neverRemind` 短路（`update_checker_release.dart:232`）/ `!supportsUpdateCheck` 时**永不跑**。于是刚装完的几百 MB 安装包至少残留 7 天，检查被关时无限残留。
- **[x] ① 已修复** — 在安装成功的确定生命周期终点即刻回收安装包。新增纯函数 `installerToDeleteAfterSuccessfulHandoff`（`hibiki/lib/src/utils/misc/update_checker_download.dart:238`）：仅当 `installed==true`、`installerPath` 非空、且归一化后该文件位于 `updatesDir` **根目录下的直属文件**时返回可删路径（越界/更深子目录/空根一律返 null，绝不误删任意路径）；`reconcilePendingWindowsInstallerHandoff`（`update_checker_release.dart:946` 附近）在拿到 `result` 后按此纯函数结果 `File(...).delete()`（best-effort，删失败记日志、由 7 天 GC 兜底，不影响成功提示）。7 天 GC 保留作失败/未完成安装的兜底。提交哈希：见分支 fix-1089-updates-cleanup。
- **[x] ② 已加自动化测试** — `hibiki/test/utils/misc/update_checker_cleanup_test.dart` 新增 group「installerToDeleteAfterSuccessfulHandoff」（8 例）：回归复现「安装成功应立刻删安装包」、未成功不删、installerPath 空/null 不删、updates 目录外的任意路径不删（防误删 System32）、更深子目录不删、反斜杠归一判定目录内、尾部斜杠根不影响、空根不删。红→绿全过。
- **备注**：跨平台安全——路径比较用 `_normalizeUpdatePathForCompare`，仅 Windows 小写化（其余平台保持大小写敏感）。Android updates 目录走 `getTemporaryDirectory` 由系统回收，且非 handoff 安装模型；此修复针对 Windows handoff 安装成功路径。与 BUG-473（TODO-1010，7 天 GC 回收旧完整包）互补：BUG-473 管「历史旧包按 mtime 回收」，本 bug 管「刚装的包安装成功即回收」，消除「等一周」的残留窗口。
