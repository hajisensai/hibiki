## BUG-066 · 一次性导入大量词典（~50 本）整个 app 崩溃
- **报告**：2026-06-06（用户：「导入词典的时候一下导50本，崩溃了修复一下」；平台 Windows 桌面，用户转述、无崩溃日志、不可在本机复现）。采番注：动手前本 worktree 基线最大号 BUG-062，合并时 develop 已被并发分支占用至 BUG-065（063=Drive token / 064=备份 / 065=AnkiConnect），故本条改取 066。
- **真实性**：✅ **真 bug，但当前不可静态指认罪魁**。沿真实代码路径排除误解：批量导入**已是串行**（`dictionary_dialog_page.dart:325` `for`+`await`，同时只跑一本；每本导完即 `persistDictionary` 落库，崩溃不丢已导完的），**不是 50 本并发**。崩溃机制定位到 native 进程级：每本经 `Isolate.run`（`hoshidicts.dart:351`）→ `hoshidicts_import` 开 32MB 栈工作线程（`hoshidicts_ffi.cpp:222`）→ 内部 `std::async` 开 `CPU+4` 线程（`importer.cpp:359/424`）。native 处理中的 `std::bad_alloc` 被 `import_thread_fn` try/catch 接住（→ 干净失败提示，非崩溃）；**能绕过 try/catch 直接带崩整个进程**的只有：① 某本畸形词典触发**深递归栈溢出 / 访问违例（SEH，非 C++ 异常）**；② OS 级 OOM-kill（Win x64 桌面少见）。`Isolate.run` 不是 OS 进程级隔离 → 任一本 native 硬崩 = 整个 app 崩。**因此 Win 桌面最可能是 50 本里某一本词典的解析路径硬崩**。附带确证缺陷：用户「低内存模式」开关是**死的**——`importer.hpp:24` `import(...,bool low_ram=false)` 默认 false，FFI 入口 `hoshidicts_ffi.cpp:186` 调用时只传两参，Dart 一路传的 `lowMemoryMode` 在 FFI 边界被丢弃，永远到不了 native。
- **[ ] ① 根因修复（待用户复现指认罪魁词典后再修该解析路径）** — 本轮**先接入能存活进程级崩溃的诊断**，非最终根因修：现有 `ErrorLogService` 错误日志是**异步落盘**（`_appendToFile` `await writeAsString`），native 硬崩时缓冲来不及刷盘 → 崩后日志里没有任何「崩在哪本」的线索，这正是无法定位的根因。新增**同步落盘的导入面包屑**：`error_log_service.dart` 加 `markImportStart`（每本调 FFI 前 `writeAsStringSync(flush:true)` 写 `import_crash_breadcrumb.txt`）/ `markImportEnd`（返回后删）/ `readAndClearBreadcrumb`（启动恢复）；`init()` 启动时若有残留面包屑 → 折进错误日志记一条「上次词典导入疑似让 app 崩溃：<文件路径>」。`dictionary_import_manager.dart` 两处 FFI 调用点（`importFromFile` 单本 + `importFromDirectory` 目录单本）用 `try/finally` 包住 start/end。用户复现一次 → 桌面日志页「另存为」导出 → 残留行即罪魁词典，再对该词典的 native 解析路径做根因修复并钩 ①。
- **[x] ② 已加自动化测试** — `hibiki/test/utils/misc/import_crash_breadcrumb_test.dart`（4 例）：文件不存在→null（正常路径不误报）；空/纯空白→null；崩溃残留→读出 trim 内容并删文件（恢复后不重复报警）；start 写入→end 清除后无残留。`flutter analyze` 3 文件 0 issue；全绿。
- **备注**：词典导入 / native 进程级崩溃类。崩溃在 native 进程级，Dart try/catch 与 `Isolate.run` 均拦不住，故本轮交付定位手段而非盲改 native（盲改需重编 5 平台库且很可能不对症，违反根因纪律）。**待用户**：在 Windows 复现「导 ~50 本崩溃」一次 → 导出日志发回 → 据罪魁词典做最终根因修复。后续可选硬化：把死掉的 `low_ram` 契约接通 FFI / 把每本导入放独立子进程做崩溃隔离（路线 C，较大改动）。
