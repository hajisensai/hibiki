## BUG-311 · TODO-518: video episode switch resumes near end then immediately auto-advances
- **报告**：2026-06-18（用户：视频上一集点了以后，直接又跳转回来了，是因为播放完毕了吗）
- **真实性**：✅ 真 bug。根因定位：
  - `hibiki/lib/src/pages/implementations/video_hibiki_page.dart:1101` 初次打开播放列表时直接把目标集保存位置传给 `_loadEpisode`，没有区分“恢复进度”与“明确从头”。
  - `hibiki/lib/src/pages/implementations/video_hibiki_page.dart:1606` EOF 自动连播、`video_hibiki_page.dart:1873` `_switchEpisode`、`video_hibiki_page.dart:1916/4831/4839` 列表/上一集/下一集入口原本共享无 intent 切集路径。
  - `hibiki/lib/src/media/video/video_player_controller.dart:687` 的 seek 阶段原本只要 `initialPositionMs > 0` 就恢复到该位置；目标集保存位置接近片尾时，autoPlay 会立刻触发 completed，再跳到下一集，表现为“上一集点了以后又跳回来”。
- **[x] ① 已修复** — 新增 `EpisodeStartIntent` 与 `resolveEpisodeStart`，在 `VideoPlayerController.load` 的 `player.open(play:false)` 后、autoPlay 前等待 duration 并按 intent 决定实际起播点：上一集/自动连播从头；下一集/列表/初次打开在目标保存位置接近结尾时从头，非近结尾恢复；duration 未知时保留下一集/列表/初次打开的保存进度，避免误清。`explicitCue` 专门表示收藏、字幕锚点等“用户明确要打开这一毫秒”的入口，它不参与近片尾归零，避免把用户点选的精确目标缩小成普通恢复进度。提交：本提交。
- **[x] ② 已加自动化测试** — `hibiki/test/media/video/video_episode_start_policy_test.dart` 覆盖 duration 已知/未知、近结尾/非近结尾、90% 边界、3 秒边界、负值 clamp、五个入口 intent；`hibiki/test/pages/video_playlist_auto_advance_guard_static_test.dart` 反转旧断言，要求 EOF/上一集/下一集/列表都显式传 `EpisodeStartIntent`，并要求初次打开接入 start policy；`hibiki/test/media/video/video_lifecycle_static_test.dart` 同步守卫 `load()` 用 `resolvedStartMs` 初始化字幕，避免重新钉回旧的 `initialPositionMs` 不变量。
