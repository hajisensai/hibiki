  1. 先 `Close()` capture **session**（`session_closable->Close()`，停止产生新帧，同步）。
  2. **`frame_pool_->remove_FrameArrived(on_frame_arrived_token_)`** 同步 revoke，把 token 置空。返回后 WGC 不再向本 token 投递新的 FirePresentEvent；此刻帧池仍存活、delegate 表内存有效——revoke 只把我们的项从一张有效表里摘掉（standard WinRT event 在 remove 时安全修改列表、fire 期用快照），不留野指针。这正是 `rcx=0` null-delegate 崩溃的根除点：null 槽只在 Close 撤销整张表后才出现，而我们把 Close 推迟到在途事件全部 fire 之后。
  3. 把 `frame_pool_` / `frame_arrived_handler_` / `callback_state` 整组移交 `PendingCaptureTeardown` holder（保持存活），经同一 `dispatcher_queue_` 用 **Normal 优先级 `TryEnqueue`**（不再用第五修的 `TryEnqueueWithPriority(Low)` 押注）排 `kCaptureTeardownDrainHops=2` 个 drain 跳让出泵，使 revoke 之前已排进 CoreMessaging 的在途 deferred FirePresentEvent 先 fire 完（遍历有效表 + `active=false` 安全返回），收敛后才在 `FinalizeCaptureTeardown` 里 `pool_closable->Close()` 撤销 delegate 表。
  与第五修的确切差异：断源不再依赖优先级仲裁的精确性，而靠同步 `remove_FrameArrived`；即便 drain 跳判定早一跳，多 fire 的事件也因「已 revoke + 帧池存活」而安全。运行期路径与退出路径都安全：退出时消息泵已停，`TryEnqueue` 返回 false（队列 shutdown）→ 内联 `FinalizeCaptureTeardown`，且 session 已 Close、FrameArrived 已 revoke，无在途事件来源，直接 Close 不泄漏、不崩。
