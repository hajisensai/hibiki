## BUG-163 · 桌面制卡闪退：WebView2 捕获帧迟到事件打进已拆除 delegate
- **报告**：2026-06-09（飞书 TODO-013：「我电脑制卡闪退了，你修复一下」）。与 TODO-012「视频制卡整句音频/画面不动」区分：本条崩溃证据落在 Windows WebView2/GraphicsCapture 捕获帧 teardown，不是字幕/视频帧抽取或整句音频范围。
- **真实性**：✅ **真 bug（本机 WER + dump 栈确认）**。`C:\Users\wrds\AppData\Local\CrashDumps\hibiki.exe.68244.dmp`（2026-06-09 19:15:23，`hibiki.exe` 0.5.0.33）对应事件日志 `GraphicsCapture.dll` `0xc0000005`，`cdb` 栈落在 `GraphicsCapture!TypedEventHandler<Direct3D11CaptureFramePool,...>::operator()+0x15`，`rcx=0`，下方是 `Direct3D11CaptureFramePool::FirePresentEvent` 和 CoreMessaging UI 线程消息派发。根因在 `packages/flutter_inappwebview_windows/windows/custom_platform_view/texture_bridge.cc:79` 注册 FrameArrived 回调时捕获裸 `this`，且旧 `StopInternal()` 在 teardown 中先 `remove_FrameArrived` 再 `Close()` frame pool；已经排到 UI 消息循环里的 `FirePresentEvent` 仍会执行，remove 后事件源内部可命中空 `TypedEventHandler` delegate，或迟到回调碰到已释放 bridge。
- **[x] ① 已修复** — 本提交。`TextureBridge` 新增 `FrameArrivedCallbackState`，FrameArrived lambda 改为捕获 `callback_state` 而不是裸 `this`；`Stop()`/析构先 `InvalidateFrameArrivedCallback()`，把 state 置为 inactive/null，再关闭 `capture_session_` 和 `frame_pool_`。同时停止在 teardown 期 `remove_FrameArrived`，避免已排队 `FirePresentEvent` 走到空 delegate；`frame_pool_` 仍 `Close()` 并置空，切断后续事件源。
- **[x] ② 已加自动化测试** — `hibiki/test/widgets/texture_bridge_stop_guard_test.dart` 扩展源码守卫：断言存在 `FrameArrivedCallbackState` / `InvalidateFrameArrivedCallback` / `[callback_state]`，禁止 FrameArrived lambda 捕获裸 `this`，禁止 `remove_FrameArrived` 回到 teardown 路径，并保留 `pool_closable` + `frame_pool_ = nullptr`。
