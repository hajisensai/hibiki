## BUG-093 · 阅读器/视频查词弹窗每次都先「白屏一下」再出内容（每次查词都冷启动 WebView）
- **报告**：2026-06-06（用户：「查词的时候查词弹窗要先白一下，等一会才会显示，能不能优化」「复用热 webview，视频也要复用，干脆默认启动一个一直留着吧」「开启的时候」）。采番注：初记为 092，rebase 到 develop 时发现 090/091/092 均已占用（BUG-090 系统主题强调色 a0c29e1d5、BUG-092 宽屏设置顶对齐 9b44c5c13），改取 093。
- **真实性**：✅ **真 bug（沿真实代码路径定位，非「弹窗慢」而是每次查词都冷启动 WebView）**。划词弹窗内容由 `DictionaryPopupWebView`（`InAppWebView`）渲染，冷加载 popup.html + JS + CSS 要时间 → 白屏。本有两套规避都对阅读器/视频/有声书（共享基类 `BaseSourcePageState`）失效：① **搜索期预挂载（[[BUG-080]]）**只在 `DictionaryPopupLayer` 的 `isSearching` 分支挂 WebView，但 `base_source_page._buildPopupLayer` **从不传 `isSearching`**，搜索期只画纯 Flutter 的 `_buildLoadingPlaceholder`（无 WebView）→ 该机制对 base_source_page 整体未覆盖。② **关闭后保留热 WebView 复用**（`_dismissPopupAt(0)` 留隐藏热槽，`_reusableHiddenTopPopup` 条件 `stack.length==1 && !visible`）本可避免冷加载，**但阅读器/视频每次查词前都调 `prunePopupStack(0)`**（`reader_hibiki_page.dart:3263/3282` 文本选择 + 歌词路径），其 keepCount=0 把 `_popupStack.value` 直接清成 `[]` → 丢掉那个热 WebView → 复用条件永不成立 → `_buildSearchPopupItem` 每次新建 fresh `webViewKey` → **每次查词都冷加载 → 每次都白屏**。独立「查词」页走另一套 `dictionary_page_mixin`（搜索前就 push `isSearching=true` 的 entry）故不白屏，只有 base_source_page 三表面白。
- **[x] ① 已修复** — 本提交（自含修复+测试+本条目）。**根因修（按用户决定：默认常驻一个热 WebView 一直复用，不再每次丢弃）**：① `DictionaryPopupLayer` 新增 `keepWebViewWarm` 标志，`_buildBody` 的挂载条件由 `hasEntries || isSearching` 改为 `hasEntries || isSearching || keepWebViewWarm` —— 让槽位在空闲/隐藏时也挂着 WebView（透明、不绘内容）冷加载一次。② `BaseSourcePageState` 在 `initState` 的 postFrame 里 `_seedWarmPopup()`：开书/开视频时就 seed 一个 `isWarmSlot=true`、`visible=false` 的持久隐藏槽（result 用既有空哨兵 `kPopupSearchingPlaceholderResult`，无需改 nullable），WebView 在页面空闲期就把 popup.html/JS/CSS 加载好。③ `prunePopupStack(keepCount<=0)` 改为**保留**该热槽（折叠掉嵌套子弹窗 + 隐藏热槽 + clearSelection），不再清成 `[]`；低内存模式无热槽，仍清空（沿用其 close 即销毁的预算）。④ `_buildPopupLayer` 传 `keepWebViewWarm: item.isWarmSlot`。效果：阅读器/视频/有声书全程共用**同一个**热 WebView，`_reusableHiddenTopPopup` 每次都命中它 → 查词只走 `didUpdateWidget` 推结果（WebView 早就 ready）→ **无冷加载、无白屏**；连第一次查词也因「开启时就预热」而是热的。`_dismissPopupAt(0)` 原本就保留隐藏 top（非低内存），与热槽天然一致。reader/video 零改（base 基类一处覆盖三表面）。
- **[x] ② 已加自动化测试** — `hibiki/test/pages/base_source_page_warm_popup_test.dart`：① 开页即 seed 一个 `isWarmSlot`、隐藏的槽（新增 `@visibleForTesting debugPopupStack` 钩子断言）；② **reader 式 `prunePopupStack(0)` 前置的连续两次查词复用同一 `webViewKey`**（核心回归——锁住「不再每次冷加载」）；③ 低内存模式不 seed、prune(0) 清空；④ 源码守卫锁 `keepWebViewWarm: item.isWarmSlot` 接线 + `prunePopupStack` 保留热槽 + layer 挂载条件含 `keepWebViewWarm`。新增可复用测试 helper `test/helpers/fake_inappwebview_platform.dart`（注册 no-op `InAppWebViewPlatform`，让含 `InAppWebView` 的 widget 树能在单测构建——热槽现会真的实例化 WebView）；既有 `base_source_page_hot_popup_test.dart` 的 setUp 接入该 helper。验证：`flutter analyze lib` 0、全量 **+2793 绿 0 失败**（~2 跳过）。
- **备注**：阅读器/视频/WebView 类。代码正确 + 无回归；**真机复测待用户**（CLAUDE.md 验证纪律）：打开书/视频后划词查词，弹窗应直接出内容、不再先白屏；连续多次查词都不白。内存代价 = 每个打开的 source page 常驻一个隐藏 popup WebView（用户明确要「默认启动一个一直留着」），低内存模式不启用。
