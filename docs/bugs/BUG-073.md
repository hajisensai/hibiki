## BUG-073 · TrueHD 音轨的电影（及有声书）播放无声音（全平台 media_kit libmpv 缺 TrueHD 解码器）
- **报告**：2026-06-06（用户：「视频播放支持 5.1 音频吗，我弄了电影结果没声音」「TrueHD 这类影院音轨也要支持」「为什么用旧的版本，不用新的 mpv」）。采番注：动手时基线最大号 BUG-071、原取 072，收尾核对 develop 已被并发分支 `420983bfd`（视频查词暂停后关窗不续播）占用 072，故顺延取 073（仍可能被并发抢，以提交为准）。
- **真实性**：✅ **真 bug（捆绑 libmpv 缺 TrueHD 解码器，沿真实代码路径 + 真实文件 ctypes 实证）**。视频走 media_kit、有声书走 just_audio_media_kit，桌面端**共用同一个 `libmpv-2.dll`**。用户的真实文件 `...TrueHD.5.1-ManualMux.mkv`（默认音轨=truehd 6ch 5.1，另有一条 flac 2.0）：标准 `mpv.exe`（新 libmpv）有声；hibiki 无声。逐层证伪后定位：
  - 字节扫描捆绑 DLL 有 `ac3/eac3/dca/truehd/mlp` 字符串 → 一度误判「缺解码器」**不成立**（字符串在≠解码器能用）；
  - 用 ctypes 直接加载 hibiki 捆绑的 `hibiki/build/windows/x64/runner/Release/libmpv-2.dll`（构建日期 **2023-09-24**）复刻 media_kit 配置加载真文件：`[ad] Failed to initialize a decoder for codec 'truehd'.` → **运行时 init 失败**；
  - 用 ffmpeg 造 ac3/eac3/dts/truehd/aac 小样本逐个测**同一捆绑 DLL**：aac/ac3/eac3/**dca(DTS) 全 OK，唯独 truehd FAIL** → 缺的只有 TrueHD/MLP（无损，被精简 FFmpeg 砍掉），AC3/EAC3/DTS 本就能解（用户最初「AC3/DTS 也要支持」的前提源于我的错误假设，已澄清）；
  - 失声链：默认音轨=TrueHD → 解码器 init 失败 → libmpv **不自动回退**到那条 flac 2.0 轨 → 整轨静音。与下混/声道协商/media_kit 配置**全无关**（裸 libmpv `--no-config` 在用户真实立体声设备上把 5.1→stereo 下混、AO 正常打开，已实证）。
  - 「为什么用旧的」根因：media_kit 的 libmpv 来自 `media-kit/libmpv-win32-video-build`，该仓库**最后一版就是 2023-09-24，两年多停更**，而 `media_kit_libs_windows_video` 1.0.11（已是最新）在 `windows/CMakeLists.txt` 硬编码 URL+MD5 钉死它。升级 media_kit 版本无效（无新版），唯有换 libmpv 源。根因 `media_kit_libs_windows_video-1.0.11/windows/CMakeLists.txt:67-71`（钉死 stale libmpv）。
  - **全平台同根（opus 研究 agent 读 media-kit 构建脚本实证，configure 级证据）**：不止 Windows。media-kit 的 **`default` 风味** FFmpeg 在编译期 `--enable-demuxer=truehd` 但**没有 `--enable-decoder=truehd/mlp`**（Android `flavors/default.sh` 白名单只列 demuxer；darwin `meson.build` 同源镜像），故 **macOS/iOS/Android 用的 `-default` 产物同样能识别 TrueHD 流但解不出声**——与 Windows 同一根因。media-kit 的 **`full` 风味**（`flavors/full.sh` / `audio_full_options` = `--enable-decoders`）才含 truehd/mlp 解码器。Linux 走系统 libmpv（发行版完整 FFmpeg）本就有。
- **[x] ① 已修复** — 本提交（自含修复+守卫+本条目；并发抢号致哈希随 rebase 变动故不内联记录）。**根因修（换依赖产物，非补丁），全 5 平台收敛到「full-codec libmpv」一套维护范式**：vendor 4 个 `media_kit_libs_*_video` 到 `third_party/` + `pubspec.yaml` `dependency_overrides` 指过去——
  - **Windows**：media-kit win32 构建已归档（无 full 变体），故 CMake 重指向仍维护的 **zhongfly/mpv-winbuild 2026-06-05**（完整 FFmpeg）。换前 ctypes 实测新 DLL：TrueHD 小样本与用户真电影均 `OK: Opening decoder truehd`，render/client API 导出齐、7z 布局与解压步骤一致。`libmpv-2.dll` 30MB→118MB。
  - **macOS/iOS**：Makefile 由 darwin **v0.6.0 `-video-default`** 升 **v0.7.0 `-video-full`** + 新 sha256（下载实算）。
  - **Android**：build.gradle 4 条 jar 由 **v1.1.7 `default-*`** 升 **v1.1.11 `full-*`** + 新 md5（下载实算）。佐证：full 的 `libmpv.so` 比 default 大 ~4MB（多编的解码器）。
  - **Linux**：系统 libmpv，无需 override。
  - 单一全平台源不存在（Windows 与 darwin/android 是两条独立 libmpv 构建谱系，media-kit 自己都没统一、还把 Windows 仓归档了）；现实最优=mac/iOS/Android 统一到 media-kit full 系列（同构建体系，升级=改版本号+校验和）+ Windows 单列 zhongfly + Linux 系统。
- **[x] ② 已加自动化测试** — `hibiki/test/build/libmpv_truehd_guard_test.dart`（源码扫描守卫，全平台——真构建/解码无法在 host 单测复现）：断言 ① pubspec 把 **4 个 `media_kit_libs_*_video`** 全 override 到 vendored fork、② Windows CMake 指 `zhongfly/mpv-winbuild` 非归档上游、③ macOS/iOS Makefile 用 `-video-full` 非 `-video-default`、④ Android gradle 用 `v1.1.11/full-*` 非 `v1.1.7/default-*`，各自校验和仍钉。
- **备注**：构建依赖/native 解码类。**比常规更强的待验证警示**：此改动是全平台 build-infra 改动，我**没在本机实际构建/运行任何一个平台**（后台跑不了 GUI 构建）。各平台证据强度：Windows 已 ctypes 实测新 DLL 解 TrueHD（最强）；macOS/iOS/Android 是 **configure 级证据**（media-kit full 风味 `--enable-decoders` 明确含 truehd/mlp）+ Android full .so 比 default 大 4MB 佐证，但**未真机 init 实测**（字符串/体积≠100% 能 init，虽 configure 已从机制保证）。`pub get` 在本机已过（4 个 override 正常解析）。落地需用户**逐平台**复测：① Windows `flutter build windows`（首次走代理 `127.0.0.1:34151` 下 118MB libmpv）；② macOS/iOS 用 CLAUDE.local.md 的远程 Mac `flutter build macos`/`ios --simulator`（Makefile 会下 v0.7.0 full xcframework 验 sha256）；③ Android `flutter build apk`（gradle 下 v1.1.11 full jar 验 md5）。每端用 TrueHD 5.1 样片复测「有声」+ 回归 AAC/AC3/DTS 视频与有声书。媒体包变大（各端完整 FFmpeg）。
