## BUG-275 · Bundled ffmpeg.exe invalid 时字幕枚举静默失败(TODO-336)
- **报告**：2026-06-14（用户：Windows 包程序目录里有 ffmpeg.exe，但内嵌/内封字幕轨加载仍失败）
- **真实性**：✅ 真 bug（BUG-233 的回退覆盖不全的续集）。沿真实代码路径：字幕枚举 `listEmbeddedSubtitleTracks`（`hibiki/lib/src/media/video/video_subtitle_source.dart:303`）经 `resolveFfmpegBackend().run(['-hide_banner','-i',video])` → `CliFfmpegBackend.run`（`hibiki/lib/src/media/video/ffmpeg_backend.dart:194`）→ `_runCliFfmpeg`（同文件 `:128`）。用户程序目录里的 bundled `ffmpeg.exe` 是 BUG-233 实证的损坏/架构不匹配文件（本机 `ffmpeg.exe -version` 返 STATUS_INVALID_IMAGE_FORMAT）。
- **根因**：`_runCliFfmpeg` 的 bundled→PATH 回退太窄，死盯单一错误码：① 退出码只认 `-1073741701`/`0xC000007B`；② `ProcessException` 只认 `errorCode==193`（ERROR_BAD_EXE_FORMAT）。实测一个无效 PE 在 `Process.start` 阶段抛 `ProcessException`，errorCode 随损坏方式而异（ERROR_FILE_NOT_FOUND=2 / ERROR_BAD_EXE_FORMAT=193 / ERROR_EXE_MACHINE_TYPE_MISMATCH=216 等，非 193 的居多）。errorCode≠193 时 `_runCliFfmpeg` **rethrow** 而不回退 PATH → 异常冒泡到 `listEmbeddedSubtitleTracks` 的 `on ProcessException { return [] }`（`video_subtitle_source.dart:308`）→ **静默返空** → 字幕菜单无任何内封轨、所有内嵌/内封加载都失败。stderr 解析正则、路径解析、`Platform.resolvedExecutable` 与 Inno Setup 打包（ffmpeg.exe 随 `{app}\*` 落 hibiki.exe 同目录）经真 ffmpeg(2026 N-123122)实测均正确，**不是**症结。
- **[x] ① 已修复**：`_runCliFfmpeg`（`hibiki/lib/src/media/video/ffmpeg_backend.dart`）改为「bundled 这个文件跑不起来就回退 PATH」，而非死盯单一错误码：bundled exe 的**任何 `ProcessException`**（其路径已过 `_bundledFfmpegPath()` 的 `existsSync()`，文件确在磁盘上却启动失败 = 损坏/架构不匹配/无执行权限）都记 debug 日志并回退 PATH 的 `ffmpeg`；跨平台一致（移动端不走 CLI 后端，桌面三端都受益）。bundled 退出码 STATUS_INVALID_IMAGE_FORMAT 的回退保留。显式 `HIBIKI_FFMPEG` 覆盖走第一分支、不进 bundled 分支，旧契约不变（如实报错、不悄悄换 PATH）。删去仅 BUG-233 用的 `_windowsBadExeFormatErrorCode` / `_isWindowsBadExeFormatProcessException`。该回退被所有 ffmpeg 消费者共用（字幕枚举/抽取、cue 动图、句子音频、视频帧/封面），一处修全链路受益。
- **[x] ② 已加自动化测试**：`hibiki/test/media/video/ffmpeg_executable_resolve_test.dart` 新增：① bundled 启动 `ProcessException`（errorCode 2/216/193/5 多码）一律回退 PATH 取到字幕枚举输出；② 非 Windows（errorCode 13 权限）也回退；③ 显式 `HIBIKI_FFMPEG` 启动失败仍如实传播 `ProcessException`（不回退）；④ 无 bundled 无覆盖时 PATH 的 `ProcessException` 向上传播（调用方据此降级无字幕）。改前 ①② 红、改后全 9 例绿。端到端验证（worktree 临时脚本，不入库）：构造无效 bundled `ffmpeg.exe`（启动抛 errorCode=216）+ PATH 真 ffmpeg + 真 mkv（1 条 ass 内封轨），修后 `_runCliFfmpeg` 回退 PATH 后字幕枚举恢复出 1 条轨（改前会 rethrow→静默返空）。
- **备注**：相邻验证 `flutter test test/media/video/ffmpeg_executable_resolve_test.dart ffmpeg_backend_test.dart ffmpeg_backend_selection_test.dart video_subtitle_source_test.dart test/utils/desktop_audio_clipper_test.dart test/reader/reader_mining_audio_guard_test.dart` 全绿（1 跳过：本机有 ffmpeg，无 ffmpeg 的 ProcessException 真跑用例按设计跳过）。`flutter analyze` 改动文件 No issues。**仍需真机确认**：本机无法复现「用户那台机器上 bundled ffmpeg.exe 究竟是哪种损坏」，需用户用其失败的 Windows 安装包（程序目录里的 ffmpeg.exe + 一个有内封字幕的真实视频）复测字幕轨现在能枚举/加载；若用户系统 PATH 上也没有可用 ffmpeg，则回退到 PATH 仍取不到 ffmpeg → 需进一步定位 bundled ffmpeg 为何损坏（打包阶段 ffmpeg-min 产物是否被杀软/CDN 破坏），那是打包侧问题不属本运行时修复范围。
