## BUG-048 · 设置页文本框（Anki 设置 Host 等）聚焦后按「↓」焦点向上跳而非到下一行（设置文本框没注册成方向导航锚点）
- **报告**：2026-06-05（用户截图：Anki 设置页 Host 文本框聚焦，按 ↓「为什么不会去下面，而是往上走了」）。采番注：动手前查 `docs/BUGS.md` 当前最大号为 BUG-047，故本条取 048。
- **真实性**：✅ **真 bug（键盘/手柄方向导航，桌面/iOS 专有——AnkiConnect 段 `if (!Platform.isAndroid)`）**。沿真实代码路径定位：Host/Port/API 密钥三框走 `_AnkiConnectionField` → `AdaptiveSettingsTextField` → `HibikiTextField`。`HibikiTextField` **只有在 `focusId != null` 时**才用 `HibikiFocusRegistration` 把焦点节点注册进方向焦点控制器（`hibiki_material_components.dart:448`：`if (widget.focusId == null) return textField;`），而 `AdaptiveSettingsTextField` 既没传 focusId 也没暴露该参数 → **三个文本框对几何焦点控制器完全不可见**。聚焦 Host 后按 ↓：单行框逃逸由 `global_navigation.dart:_handleGlobalArrowFocus` → `gamepadMoveFocusInDirection` → `HibikiFocusController.move(down)` 处理；但 `_currentEntry()`（`hibiki_focus_controller.dart:253`）拿当前 primaryFocus（Host 的 EditableText 节点）去比对已注册条目，没有任何 entry 的 focusNode 与之 `identical` → 返回 null → `active == null` → 走 `_moveByReadingOrder(currentIndex: -1)` → `_nextIndex(-1)` 返回 0 → 聚焦 `targets[0]`（**第一个**已注册行 = Host 上方的「从 AnkiDroid 获取」行）。于是 ↓ 跳上去了。同段 Tags 输入框、词典设置对话框文本框是同一潜伏 bug。根因点 = `settings_shared.dart` 的 `AdaptiveSettingsTextField` 不给文本框注册 focusId，致其不是几何导航锚点。
- **[x] ① 已修复** — 根因修：消除「设置文本框忘了传 focusId 就不可导航」这一整类失败（非给单个调用点打补丁）。`AdaptiveSettingsTextField` 由 `StatelessWidget` 改 `StatefulWidget`，State 持有稳定的 `_fallbackFocusId = HibikiFocusId('settings-textfield-${identityHashCode(this)}')`（跨重建稳定，镜像同文件 `_SettingsRowFocusTarget` 的做法），始终向 `HibikiTextField` 传 `focusId: widget.focusId ?? _fallbackFocusId`；并暴露可选 `focusId` 参数给需要显式 id 的调用方。这样每个设置文本框都成为方向导航锚点（既能被 `_currentEntry()` 命中作几何起点，也能作为相邻框的目标），故 Host ↓→Port→API 密钥、↑ 反向皆正确。`HibikiTextField` 自身的注册仍受「无 HibikiFocusRoot 时返回裸框」闸门保护，无焦点根的上下文（部分测试）不受影响。`flutter analyze`（改动文件）0 issue、`dart format` 0 改动。
- **[x] ② 已加自动化测试** — `hibiki/test/widgets/adaptive_settings_textfield_focus_test.dart`（widget 行为，最强可落地层）：HibikiFocusRoot 下一个可点击行（模拟「从 AnkiDroid 获取」）+ 三个**不传 focusId** 的 `AdaptiveSettingsTextField`（各自 focusNode），聚焦第一个框后 `controller.move(down)` 断言落到**第二个框**（修复前落到上方那行 → 红，已实测 stash 掉修复即红、恢复即绿）、再 ↓ 到第三个、↑ 回第二个，且上方行 0 次点击。`flutter test test/focus/ test/widgets/ test/settings/ test/pages/anki_settings_page_test.dart` 仅 `settings_schema_coverage_test` 的 5 个**门控 switch 行**（音量按钮/查词暂停/句导航/调试日志，与文本框无关）预存红——已 `git stash` 剥离本修复实测基线同为这 5 个红，证明本改动**零新增失败**；其余 +332 全绿。
- **备注**：焦点/方向导航/设置文本框类。代码 + 单测绿、`flutter analyze` 0 issue。**真机肉眼复测原始失败路径待用户**（Windows/桌面 + 键盘或手柄）：Anki 设置 → 聚焦 Host 框按 ↓ 应到 Port、再 ↓ 到 API 密钥、↑ 反向；同段 Tags 框、词典设置对话框文本框同理可上下导航。
