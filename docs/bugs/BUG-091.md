## BUG-091 · 制卡偶发「Write failed errno 10053」失败（Anki 正常）：陈旧 keep-alive 连接上 addNote 不重试
- **报告**：2026-06-06（用户：Anki 页面一切正常，但制卡偶尔失败；经 [[BUG-089]] 修复后错误日志页捕获到完整堆栈 = `ClientException with SocketException: Write failed (errno=10053)` 于 `addNote`）。采番注：本 worktree 基线最大号 BUG-089，但 BUG-090 已被并发 worktree（fix/win-system-theme-accent · Windows 系统主题强调色，未合 develop）占用，故取 091。
- **真实性**：✅ **真 bug（沿真实堆栈 + 代码定位，非环境问题）**。根因正是 [[BUG-065]] 描述的陈旧 keep-alive 连接：AnkiConnect 的极简 HTTP server 会关闭空闲 keep-alive 连接，持久 `http.Client` 连接池把请求交给一个已被对端关闭的 socket → 首次使用即报连接断（Win errno 10053 WSAECONNABORTED「Write failed」）。`AnkiConnectService._postWithStaleConnectionRetry`（`ankiconnect_service.dart`）对**幂等**动作会在新连接上重试一次，但 `addNote`/`createModel` 被 `_nonIdempotentActions` 完全排除、**从不重试**（原始 BUG-065 修复怕「写成功后读响应阶段断连」导致重发出重复卡）。后果：用户闲置一段时间后**第一次制卡**复用陈旧连接 → 写失败 → 不重试 → 直接失败，而 Anki 其实完全正常。关键洞察：用户的错误是 **`Write failed`（写系统调用失败）**，证明请求字节**根本没送达**服务器（write() 失败＝未交付，partial write 也是不完整 HTTP 请求、Anki 不会建卡）→ 此情形重发**可证明不会产生重复卡**；原作者担心的是「写成功、读响应阶段 reset」（消息为 "Connection reset/closed"，不含 "Write failed"）那种可能已被处理的情况，二者由消息文本可区分。
- **[x] ① 已修复** — 本提交（自含修复+测试+本条目）。**根因修（把 addNote 重试收窄到可证明 dup-safe 的写阶段失败，而非一刀切禁止）**：新增 `_isPreDeliveryWriteFailure(ClientException)`——仅当消息含 `write failed` 或 `broken pipe`（EPIPE，对端关读端时写失败）才判定为「请求未交付」；`_postWithStaleConnectionRetry` 改为：幂等动作 → 任何连接断都重试（`_isConnectionDrop`，不变）；非幂等动作（addNote/createModel）→ 仅 `_isPreDeliveryWriteFailure` 时重试一次（在新连接上，陈旧 socket 已被踢出池）。响应阶段 reset（可能已交付）仍照旧上抛给用户，杜绝重复卡。errno 单独无法区分读/写阶段，故非幂等路径靠消息文本判定（唯一可靠的「交付前/后」判别信号）。
- **[x] ② 已加自动化测试** — `packages/hibiki_anki/test/ankiconnect_service_test.dart`「stale keep-alive connection」组：① 新增「retries addNote on a pre-delivery write failure (request not sent)」= 用真实形态的 `Write failed errno 10053` 异常，断言 addNote 重试一次（attempts==2）并返回 note id；② 新增「does NOT retry addNote on a response-phase reset (may be post-delivery)」= `Connection reset by peer` 异常，断言只试一次（attempts==1）后上抛（dup-safe）；替换原「NEVER retries addNote」单测。包全量 33 绿，analyze 0。
- **备注**：Anki 网络 IPC 类。代码正确 + 无回归；**真机复测待用户**：app/Anki 闲置一段时间后第一次制卡应成功（自动在新连接重发），不再偶发「Write failed」。仍保留「写成功后响应阶段断连不重试」的防重复卡保证。与 [[BUG-089]]（失败原因可见）配套——正是 089 让此 errno 浮出水面才得以定位 090。
