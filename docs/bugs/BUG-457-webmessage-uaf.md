## BUG-457 · WebView2 事件 handler 析构后回调 UAF
- **报告**：2026-06-29（用户：）· TODO-964 · 同模式 BUG-450/TODO-931（第三处独立 UAF）
- **真实性**：✅ 真 bug（cdb 两个 dump 确证）。崩点 = WebView2 事件 handler 在 `InAppWebView` 析构后被迟到回调触发的 use-after-free：dump 的 `rcx` 直接是 `{ "name" ...` 的 JSON 消息字节（已释放、被复用为 WebView2 JSON 消息缓冲的内存被当 `this` 解引用）→ 主嫌 `add_WebMessageReceived`。
- **根因**（`packages/flutter_inappwebview_windows/windows/in_app_webview/`）：
  - TODO-931 只为 `add_WebResourceRequested` 做了正确范式（保存 `webResourceRequestedToken_`、lambda 捕获 `alive_` 拷贝、`~InAppWebView()` 里 `*alive_=false`+`remove_WebResourceRequested`），**没推广到其余约 13 个裸 `[this]` 的 handler**。
  - 这些 handler 全部裸捕获 `[this]`、不保存 `EventRegistrationToken`、`~InAppWebView()` 从不 `remove_`、回调入口无存活守卫。WebView2 的 `Stop()`/`Close()` 不同步冲刷在途事件 → 析构后迟到回调把已释放内存当 `this`/`channelDelegate`/`cursorChangedCallback_` 解引用 → 崩。
  - 涉及（`in_app_webview.cpp`，修复前行号近似）：`add_NavigationStarting`:474 / `add_ContentLoading`:576 / `add_NavigationCompleted`:587 / `add_DocumentTitleChanged`:653 / `add_HistoryChanged`:665 / `add_WebMessageReceived`:679（主崩点） / `add_NewWindowRequested`:776 / `add_WindowCloseRequested`:835 / `add_PermissionRequested`:845 / `add_DOMContentLoaded`:1013（在 `ICoreWebView2_2` 接口上） / `add_CursorChanged`:1055（在 `webViewCompositionController` 上，触碰 `cursorChangedCallback_`） / 静态 DevTools 接收器 `Fetch.requestPaused`:337 与 `Runtime.consoleAPICalled`:733（局部 receiver，registration 不随对象销毁而注销） / 动态 `addDevToolsProtocolEventListener`:1665（receiver 存进 `devToolsProtocolEventListener_` 但 dtor 只 clear 不 remove）。`add_ServerCertificateErrorDetected` 在注释块内（死代码），无需处理。
- **[x] ① 已修复** —
  - `in_app_webview.h`：为每个 handler 新增配套 `EventRegistrationToken xxxToken_`（含 DOMContentLoaded/CursorChanged），并把两个静态 DevTools receiver（`fetchRequestPausedEventReceiver_`/`consoleMessageEventReceiver_`）+ token 提升为成员以便析构时 remove。
  - `in_app_webview.cpp`：每个 handler 改 `[this, alive = alive_]`（或局部 `auto alive = alive_;` 配 `[this, alive]`），入口 `if (!*alive) return S_OK;` 先于任何 `this`/成员解引用（含原先 `if(!channelDelegate)` 这种本身已解引用 `this` 的检查）；`add_Xxx(..., &xxxToken_)` 保存 token。跨 Dart 桥异步回来的内部回调也加固：WebMessageReceived 的 callHandler resolve/reject、NewWindowRequested/PermissionRequested 的 defaultBehaviour/nonNullSuccess（dead 分支仍 `deferral->Complete()` 释放在途 deferral，COM 引用计数有效，与 live 分支互斥不 double-complete）、Fetch 的 allowRequest/cancelRequest。
  - `~InAppWebView()`：`*alive_=false` 之后、`Stop()`/`Close()` 之前，逐个 `remove_Xxx`（DOMContentLoaded 重新 `QueryInterface(ICoreWebView2_2)`；CursorChanged 在 compositionController；两静态 DevTools receiver + 动态 listener map 全部 remove 后 clear）。`channelDelegate` 仍由 `unique_ptr` 成员在 dtor 末尾自然销毁——**未提前 `reset()`**：在途回调对象由 channelDelegate 的待回调机器持有，提前 reset 可能销毁正在执行的回调，而 alive_ 守卫已是权威屏障，故采保守的根因正确做法。
- **[x] ② 已加自动化测试** —
  - `hibiki/test/pages/webview2_event_handler_alive_guard_static_test.dart`（新增源码守卫，16 用例绿）：钉住每个 `add_<Event>` 都有配套 `remove_<Event>`、DevTools receiver 也 remove、dtor 内 `*alive_=false` 先于 remove、handler 以拷贝捕获 `alive_`、入口含 `if (!*alive)` 守卫。
  - **为何用源码守卫**：这是只有真 WebView2 native 层运行才会崩的 C++ 生命周期 bug，本仓无可跑的 headless C++ 单测框架（931 当年也只加了 Dart 守卫），故在最强可落地层（源码扫描契约）锁定。C++ 改动以「编译意图自洽 + 头文件成员声明匹配」验收，真机连点不崩留用户 AMD 设备验。
- **备注**：把 931 范式系统性推广到全部裸 `[this]` 的 WebView2 handler；所有重建 WebView 的表面（reader/video/首页查词）都受益。
