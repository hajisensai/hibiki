## BUG-206 · 手机查词高亮多选/少选字（错位）
- **报告**：2026-06-11（用户：附图 5126df20…uyauyashiku）
- **真实性**：✅ 真 bug — 根因 `hibiki/lib/src/pages/base_source_page.dart`（旧第 199-201 行）：被查词高亮字数取自词典词条 **headword**（`dictionaryResult.entries.first.word.runes.length`，即去屈折后的词典形/汉字写法），而非正文里真正出现的**活用形 matched source 长度**。
- **现象**：查正文假名活用形「うやうやしく」(6 字) 时，词条 headword 是汉字词典形「恭しい」(3 runes)，高亮只取 3 字 → 只盖前半「うやう」（少选字）；当该词在 Android 分页 WebView 里跨多个 DOM 文本节点时，偏短的覆盖渲染成两段错位高亮带（图里看着像「多选」）。无头 node 复现（注入真 `highlightSelection` 源 + 假 DOM）：count=headword 长度 → 段 `[うやう]`（少选）；count=matched 长度(=`bestLength`) → 单段 `[うやうやしく]`（连续正确）。
- **[x] ① 已修复** — 改用既有 `Language.getFinalHighlightLength(result, searchTerm)`（日语非空格分词 → `max(1, bestLength)`，`bestLength` = FFI 去屈折 `matched` 源串长度，等同 Yomitan `originalTextLength`），并保留「无词条 → 0 → 不高亮」语义。新增纯函数 `lookupHighlightCharCount`（`base_source_page.dart`）封装该决策。`searchDictionaryResult` 返回值改走此函数。提交 `d6b5b4dac`。
- **[x] ② 已加自动化测试** — `hibiki/test/pages/lookup_highlight_char_count_test.dart`：kana 正文+kanji headword 须返回活用形长度(6 非 3)、活用动词「食べられた」(5 非 3)、无词条→0、直配词「猫」→1；撤修复（改回 headword 长度）实测红（Expected 6 / Actual 3）。提交 `d6b5b4dac`。
- **备注**：根因是「字数来源/口径」错（用 headword 而非 matched），非 `highlightSelection` 的 range 分发逻辑——后者在传入正确 matched 长度时本就产出连续区段（无头复现 Scenario C 证）。未回退 BUG-110/123/125 的 ruby 高亮修复（ruby 节点不进 `::highlight`、composeOpaqueColor 等）。桌面端不受影响：桌面同样经此路径，修复对桌面=用 matched 长度（更准），现有正确行为不退化（直配词测试守）。残留：Android 真机渲染（`::highlight` 跨节点几何）host 不能完全复现，已用无头 Chromium/node 同引擎在逻辑层复现+守；真机最终视觉留用户复测。
