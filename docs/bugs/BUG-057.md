## BUG-057 · wty-ja-en 等 Wiktionary 非词目（non-lemma / alt-of）释义显示成乱码 `时Hyōgai时alt-of时alternative时kanji`（未识别的 `[词, [标签]]` 数组 glossary 被平坦化成无间隔裸文本）
- **报告**：2026-06-05（用户截图：首页「词典」标签查 `時だった`，wty-ja-en 一条结果显示成 `时Hyōgai时alt-of时alternative时kanji`，问「这查词结果怎么回事…这还有乱码」）。采番注：动手前查 `docs/BUGS.md` 当前最大号为 BUG-056，故本条取 057（并发 agent 多，编号可能被抢，以实际提交为准）。
- **真实性**：✅ **真 bug（渲染器未识别 Wiktionary「异形/form-of」数组 glossary，平坦化成无间隔裸文本，被误读为字节乱码）**。先验真伪+定根因：① 当前已安装的 13 部词典里**没有** wty-ja-en（用户截图后已删，见用户确认），故按 `dictionary_downloader.dart` 的 URL 从 HuggingFace 拉了 `wty-ja-en.zip`（15MB）直接读 `term_bank_*.json` 取证。② 「時」共 8 条 term_bank 条目，其中 `defTags='non-lemma'`、reading 为空那条的 glossary 正是 `[["时",["Hyōgai"]],["时",["alt-of"]],["时",["alternative"]],["时",["kanji"]]]`——Yomitan 的「form-of」表示：一串 `[词, [标签...]]` 对（这里指向简体异形「时」，标签 Hyōgai/alt-of/alternative/kanji）。**「时」不是坏字节，就是 U+65F6 简体「时」本身**；用户母语中文，见简体字紧贴英文标签、无任何间隔/样式，遂当成乱码。③ 根因点 `hibiki/assets/popup/popup.js` 的 `renderStructuredContent` 数组分支：该数组既非 string 数组（`isStringArray`=false）、也非 link 数组（`isLinkArray`=false），落到末尾 `node.forEach(renderStructuredContent)` 把每个 `[词,[标签]]` 对再递归——词「时」走 string 分支生成一个裸文本节点、`["Hyōgai"]` 走长度 1 数组再生成一个裸文本节点，**首尾相接无空格无样式** → 渲染成 `时Hyōgai时alt-of时alternative时kanji`（与截图逐字吻合）。`assets/popup/definition.js`（`DictionaryHtmlWidget`/结构化内容页共享渲染）同一缺陷。Yomitan 本身会把该格式渲染成「词 + 标签 chip」，故这是 Hibiki 渲染缺口，非数据损坏、非 Yomitan 等价正确。
- **[x] ① 已修复** — 根因修（Linus 式：识别这一数据形状并按语义渲染，而非给平坦化打补丁）。`popup.js` / `definition.js` 的 `renderStructuredContent` 数组分支**最前面**新增判定：当 `node` 每一项都是 `[string, string[]]` 对（新 helper `isTaggedTermPair` / `isTaggedTermPairDef`）时，交给 `renderTaggedTermPairs` / `renderTaggedTermPairsDef` 渲染——每对一行（block `.form-of-item`）：被引用词（`.form-of-term`，右留 4px 间距）+ 其标签 chip（复用既有 `.glossary-tag` 样式，`.glossary-tags` 行）；`popup.css` 加 `.form-of-list`/`.form-of-item`（flex+baseline+行间距）让多对成读得通的列表而非挤成一团。判定极窄（项必须恰为「字符串 + 字符串数组」二元组），普通结构化内容（对象/字符串/link 数组）一律不命中、零影响。词暂渲染为非链接可读文本（不自动可点查词），避免误点简体「时」（任何日文词典都查不到）触发空弹窗（与 BUG 截图里中央那个「未找到搜索结果」空框同源，见备注）。`node --check` 两文件语法过；相邻 Dart 词典/弹窗测试 66 例无回归。
- **[x] ② 已加自动化测试** — `hibiki/test/utils/misc/popup_asset_behavior_test.js`（JS 行为测试，最强可落地层——直接在 FakeDOM 里跑 `renderStructuredContent`）新增 `testFormOfGlossaryArrayRendersSeparatedTermAndTags`：用截图同款数据 `[["时",["Hyōgai"]],…]` 渲染，断言产出**单个** `.form-of-list`、**4 个** `.form-of-item`（一对一项，不再平坦化）、首项含 `.form-of-term`=「时」+ `.glossary-tag`=「Hyōgai」、后续项各自保留词与标签。red→green 实测：把新分支条件改成 `false` → 断言「expected a single wrapping list」**红**（exit 1）；恢复后**绿**（exit 0）。
- **备注**：词典渲染/结构化内容/Wiktionary 类。**真机肉眼复测原始失败路径待用户**（任意平台，需重新装回 wty-ja-en）：查「時」→ wty-ja-en 那条非词目释义应显示成可读的「时 [Hyōgai] [alt-of] [alternative] [kanji]」列表，不再是 `时Hyōgai时alt-of…` 连串。注：该 JS 测试目前未接入 `flutter test`（独立 node 运行），是 popup.js 渲染的既有测试层；本条沿用之。**关联**：用户截图中央那个空的「未找到搜索结果」大框 = 点结果内的词/汉字后嵌套弹窗查无结果（`dictionary_page_mixin.buildNestedPopupLayer` 固定 `maxHeight:360` 的 `Positioned`，空结果也撑满）；本次未改其尺寸（属低风险但需设备验证的布局改动），单列为后续项。
