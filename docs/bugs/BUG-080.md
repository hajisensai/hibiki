## BUG-080 · 查词弹窗渲染慢（WebView 冷加载串行排在 FFI 查询之后）
- **报告**：2026-06-06（用户：「查词弹窗渲染好慢」）。采番注：动手时 worktree 基线最大号 BUG-078（本轮同时处理 079/081/082），取 080（以提交为准，可能被并发抢）。
- **真实性**：✅ **真 bug（沿真实代码路径定位）**。划词查词链路：选词 → `setState`（压栈，result=null/isSearching=true，仅显 spinner）→ `await searchDictionary`（**同步** FFI `lookup`+`lookupPopupJson`，UI 线程阻塞）→ `setState`（result 就绪）→ 此刻 `DictionaryPopupLayer._buildBody` 才**首次**插入 `DictionaryPopupWebView`（旧 `dictionary_popup_layer.dart:123` 把 WebView 包在 `result!=null&&entries.isNotEmpty` 里）→ InAppWebView 创建 + 加载 popup.html + 4 个 JS + CSS → `onLoadStop` 才 `_pushResults` 注入渲染。后果：**WebView 整段冷加载（解析 popup.html + dict-media.js + selection.js + popup.js(~1869 行) + popup.css(~608 行)）串行排在 FFI 之后**，无法与搜索并行——首次冷查词尤其慢。JS 端 `renderPopup` 已做首条目优先 + 余条延迟渲染（非瓶颈）；`popupJson` 由 C++ 预构建（非瓶颈）。根因 `hibiki/lib/src/pages/implementations/dictionary_popup_layer.dart:123`（WebView 仅在结果就绪后才挂载）。
- **[x] ① 已修复** — 本提交（自含修复+测试+本条目）。**根因修**：`_buildBody` 的挂载条件从 `hasEntries` 放宽为 `hasEntries || isSearching`——**搜索一开始（result 仍 null）就挂载 WebView**，让 popup.html/JS/CSS 冷加载与同步 FFI 查询**并行**；结果到达后经 WebView 既有的 `didUpdateWidget`（result 标识变化 → `_pushResults`）注入，基础设施现成。预挂载用共享空 `kPopupSearchingPlaceholderResult`（单实例，避免搜索态重建间 `didUpdateWidget` 空推）。无闪烁：WebView `transparentBackground:true` + popup.css body `background-color: var(--background-color, transparent)`，主题变量推送前 body 全透明 → 透出主题化的 `HibikiPopupSurface` + spinner 覆盖，观感与改前一致。搜索完成无结果仍走原 placeholder（不留 WebView）。**未动**更激进的「FFI 移出 UI 线程」（句柄跨 isolate 共享有风险）与「main.dart 常驻预热实例」——本改动是最低风险的最大头收益，其余留待真机 `[dict-perf]` 埋点量化后再定。
- **[x] ② 已加自动化测试** — `hibiki/test/pages/dictionary_popup_eager_mount_guard_test.dart`（源码守卫 2 例：存在 `kPopupSearchingPlaceholderResult`；`_buildBody` 含 `hasEntries || isSearching` 且 `result ?? kPopupSearchingPlaceholderResult`。真 InAppWebView/平台视图 headless 不可用故守卫调用点不变式）。回归：`dictionary_popup_layer` / `base_source_page_hot_popup`（隐藏复用 + 低内存 dispose）/ `dictionary_popup_webview` / `dictionary_popup_zoom` / `dictionary_webview_neutralizer_wired` 共 29 绿，`flutter analyze` 0 issue。
- **覆盖范围（审查 I2，诚实记录）**：传 `isSearching` 给 `DictionaryPopupLayer` 的表面才享受预热——**视频查词（DictionaryPageMixin）/ 首页词典 tab / 悬浮窗 / 独立弹窗页（popup_dictionary_page）**。**阅读器**（`ReaderHibikiPage extends BaseSourcePage`）走的是 base_source_page 的独立 `_buildLoadingPlaceholder`（LinearProgressIndicator），**不传 isSearching**，故本次预热不覆盖阅读器首次冷查词；但 base_source_page 用 `_reusableHiddenTopPopup` + `Visibility(maintainState:true)` 把 WebView **关窗后常驻热复用**，阅读器重复划词本就快、只有每会话首次冷。给阅读器首次也预热需改中枢 base_source_page（pre-warm 复用项的 WebView），风险较高且无法 headless 验证 → **单独评估、未做**。
- **备注**：弹窗渲染**时延**只能在真机/真桌面端用现有 `[dict-perf]`/`[popup-perf]` 埋点量化对比「修前 vs 修后首次冷查词」**待用户**（后台跑不了 GUI 构建）。若真机量化仍偏慢，下一步按埋点决定是否再上 FFI-off-thread / 常驻预热实例 / 阅读器路径预热。
