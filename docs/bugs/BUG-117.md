## BUG-117 · 书内跳转超链接点击「只加遮罩、不跳转」（Windows fork 不触发 shouldOverrideUrlLoading）
- **报告**：2026-06-08（用户：「点击书籍里面的跳转超链接会卡住，没实际跳转……我点击只会加遮罩，不会跳转」；书=Re：ゼロ23 目次页章节链接）。采番注：动手时记 BUG-111，rebase 到 develop 时 111~116 已被并发提交占用，改取 117。
- **真实性**：✅ **真 bug（Windows 设备运行时日志实证）**。在真实 re0 23 上点目次链接，逐点加诊断日志跑 `flutter run -d windows`，铁证：`shouldOverrideUrlLoading` 被调用 **0 次**、`resolveInternalLink` 0 次；点击 `<a href="p-005.xhtml#id-a004">` 后 WebView **直接原生导航**到 p-005，随后 `onLoadStop: url=p-005.xhtml#id-a004 chapter=8` → 因 `_currentChapter` 仍是 8(目次)、`expected=p-toc-001.xhtml` 与 p-005 不符 → 判 **stale 丢弃**，`_onChapterLoadComplete` 不执行 → 新章节不注入分页 setup（还报 `Cannot read properties of undefined (reading 'selectText')`）→ 阅读器卡在半渲染。根因=`flutter_inappwebview_windows` fork **未实现 `shouldOverrideUrlLoading`**（fork lib 内 grep 0 处；`useShouldOverrideUrlLoading:true` 已设也无效），阅读器内链拦截只依赖这个回调，于是 Windows 上链接点击绕过分页导航、`_currentChapter` 不更新（`reader_hibiki_page.dart` 旧 `shouldOverrideUrlLoading`）。「遮罩」=`_readerContentReady=false` 时盖的 `ColoredBox`（:1222）在异常时序下残留。
- **[x] ① 已修复** — 本提交。根因修（消除平台依赖，非补丁）：① 在阅读器 setup 脚本加 JS `click` 捕获监听，命中 `closest('a[href]')` 时 `e.preventDefault()` 阻止原生导航，把浏览器解析后的绝对 `a.href` 经 `callHandler('onInternalLink', a.href)` 转发给 Dart——`click` 事件在每个平台都触发，不依赖平台特定的 `shouldOverrideUrlLoading`。② 新增 Dart `onInternalLink` JS 处理器 + 共享方法 `_handleInternalLinkUrl(url)`（resolve→同章原地跳/异章重载跳/外链兜底，逻辑从旧 `shouldOverrideUrlLoading` 抽出），`shouldOverrideUrlLoading` 改为移动端 fallback 复用同一方法，两路径不分叉。preventDefault 后原生导航不发生，移动端该回调也不再二次触发，无双重处理。设备复测：点目次链接 → `onLoadStop url=p-004.xhtml chapter=12` + `save position section=12`，章节从 8 真切到 12、无 stale/无 8s 超时，遮罩自然掀开、跳转成功。
- **[x] ② 已加自动化测试** — `hibiki/test/pages/reader_internal_link_guard_static_test.dart`（源码扫描守卫，WebView/JS 真行为真机不可单测）：断言 setup 脚本有 `click` 监听 + `closest('a[href]')` + `preventDefault`（在转发 `onInternalLink` 之前）、`onInternalLink` 处理器已注册并调 `_handleInternalLinkUrl`、`_handleInternalLinkUrl` 含 resolve/原地跳/导航/外链且内链先于外链、`shouldOverrideUrlLoading` 复用同一解析方法。`flutter test test/reader test/pages test/epub` 675 例全绿。
- **备注**：阅读器/WebView/平台兼容类。**真机已复测原始失败路径（Windows，re0 23 目次链接，运行时日志验证章节 8→12 切换成功）**。移动端（Android/iOS）`shouldOverrideUrlLoading` 本就触发，新 JS 路径与其行为一致（preventDefault 后原生导航不发生），不破坏移动端；但移动端真机肉眼复测内链跳转待用户。补 `flutter_inappwebview_windows` fork 的 `shouldOverrideUrlLoading` 是更底层的修，但 JS 拦截是跨平台一致的正解，无需动 C++ fork。
