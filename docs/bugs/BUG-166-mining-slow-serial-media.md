## BUG-166 · 制卡慢（约 6 秒）+ 每张卡自动打 hibiki tag
- **报告**：2026-06-11（用户：飞书巡检 TODO-062）
- **真实性**：✅ 真 bug。根因在 `packages/hibiki_anki/lib/src/ankiconnect/ankiconnect_repository.dart:357`（旧代码）与 `packages/hibiki_anki/lib/src/ankidroid/anki_repository.dart:177`（旧代码）：一次制卡的几路独立媒体上传（封面 / 句子(sasayaki)音频 / 单词远程音频 / N 条词典外字）被串成一条 `await` 链，每路一次 AnkiConnect `storeMediaFile` HTTP 往返（或 AnkiDroid `addFileToMedia` 平台通道往返 + 文件读取/纯 Dart SHA256）。一张带封面+句子音频+单词音频+若干外字的卡是 5~8 次串行往返，总耗时 = 各路之和 ≈ 6s。这些上传彼此独立，`storeMediaFile` 是幂等纯写入（文件名由内容 SHA256 决定，不同文件互不冲突），本应并发。
- **[x] ① 已修复** — 两后端 `_mineEntryInner` 把互相独立的几路媒体上传从串行 `await` 改为一次性 `Future.wait` 并发，总耗时从「各路之和」降到「最慢一路」。`isDuplicate` / `addNote` 仍串行在其后（依赖渲染结果，且 addNote 非幂等，见 `_nonIdempotentActions` 的重试保护）。媒体方法内部已各自 try/catch 吞异常返回 null/空 map，`Future.wait` 不会因单路失败而抛。提交：`packages/hibiki_anki/lib/src/ankiconnect/ankiconnect_repository.dart` + `packages/hibiki_anki/lib/src/ankidroid/anki_repository.dart`。计时证据：5 路各 200ms 的独立上传，串行实测 1007ms，`Future.wait` 并行实测 201ms（5× 提速）。
- **[x] ② 已加自动化测试** — `packages/hibiki_anki/test/mining_tag_and_parallel_test.dart`：注入 200ms/store 延迟的录制 service，端到端制一张带封面+句子音频+单词音频+2 外字（共 5 路）的卡，断言总耗时 `< 800ms`（串行下界 5×200=1000ms，并行实测 ≈200ms）——若改回串行此断言变红。
- **备注**：第二诉求「每张 Hibiki 制出的卡打 hibiki tag」在同一改动里落地：新增 `BaseAnkiRepository.buildNoteTags(String userTags)`，把用户配置的 tag **追加** `hibiki`（去重、保序），两后端 `_mineEntryInner` 共用，替换原先各自重复的 `settings.tags.split(...)`。测试覆盖：空配置→`[hibiki]`、保序追加、空白不产生空 tag、用户已配 hibiki 不重复、AnkiDroid 后端同行为。外部不可控残留：AnkiConnect/AnkiDroid 服务端处理 + 远程音频下载本身的耗时不在客户端可控范围，本修复把客户端可控的「串行 → 并行」部分提速到位。
