## BUG-283 · Bundled ffmpeg 跑起来却空输出时不回退 PATH 致内封字幕枚举静默失败
- **报告**：2026-06-15（用户：「读取不到内嵌还是内封字幕了，字幕轨里面没有」——以前能读、现在读不到，回归。TODO-372）
- **真实性**：✅ 真 bug（BUG-275 的回退覆盖不全的第三幕）。沿真实代码路径：字幕枚举 `listEmbeddedSubtitleTracks`（`hibiki/lib/src/media/video/video_subtitle_source.dart:303`）→ `resolveFfmpegBackend().run(['-hide_banner','-i',video])` → `CliFfmpegBackend.run`（`hibiki/lib/src/media/video/ffmpeg_backend.dart:194`）→ `_runCliFfmpeg`（同文件 `:119`）。解析正则 `_subtitleStreamPattern`（`video_subtitle_source.dart:58`）经无头探针实测对 ffmpeg 5/6/7.x 全部 `Stream #...[0x2](lang): Subtitle: codec` 行（含 ass/subrip/hdmv_pgs_subtitle/mov_text/forced/default、有无 `[0x2]`、有无 lang）均正确匹配——**不是**正则断点。上层 `video_hibiki_page.dart` 的枚举/菜单/恢复路径（`_subtitleSourcesForMenu` / `_restorePersistedSubtitle`）未做任何把图形轨/forced 轨滤空的操作——**不是**过滤断点。断点在 (a) **ffmpeg 跑成但产出空文本**。
- **根因**：`_runCliFfmpeg` 的「bundled→PATH」回退仍不完整。BUG-275 已覆盖 ① bundled 退出码 = STATUS_INVALID_IMAGE_FORMAT ② bundled 在 `Process.start` 阶段抛 `ProcessException`（任意 errorCode）两种。**但还有一类损坏让 `Process.start` 成功、进程真起来、随后在加载期才崩**：典型 STATUS_DLL_NOT_FOUND(0xC0000135) / STATUS_ENTRYPOINT_NOT_FOUND(0xC0000139)——bundled `ffmpeg.exe` 本体没坏，但它依赖的 `avcodec/avformat` 等 DLL 被杀软隔离或漏打包。此时退出码**不在** BUG-275 认的 invalid-image 列表里、也不抛异常，`_runCliFfmpeg` 把这条「退出码非 0 + stderr 全空」的结果**原样返回、从不回退 PATH** → `parseSubtitleStreamsFromFfmpegLog` 收到空文本 → 解析出 0 条轨 → `listEmbeddedSubtitleTracks` 既不抛异常也不回退 → **静默无任何内封字幕轨**（用户看到「字幕轨里面没有」）。同一断点也连累内封字幕抽取（`_extractAllEmbeddedSubtitles` → `extractEmbeddedSubtitlesViaFfmpeg`）、cue 动图、制卡句子音频、视频帧封面——全走同一 `_runCliFfmpeg`。
- **[x] ① 已修复**（commit `2f565301f`）：`_runCliFfmpeg`（`hibiki/lib/src/media/video/ffmpeg_backend.dart`）把回退条件从「死盯 invalid-image 退出码」扩展为一个**稳健的语义信号**——新增纯函数 `_bundledProducedNoUsableOutput(FfmpegRunResult)`：一个真正能跑的 ffmpeg 无论退出码是否为 0 都会往 stderr 写东西（banner / version / `-i` 流信息 / 真错误），所以「退出码非 0、非超时(null)、且 stderr 完全为空」就是「这个二进制没真正运行起来」的标志，回退 PATH。该条件**subsume** 了原 invalid-image 特例（那些码本就非 0 且产出空 stderr）。明确排除项（保持原契约不回退）：`returnCode==null`（超时 SIGKILL，是慢 IO 非坏二进制，回退会让用户对同一慢文件再等一遍）、`returnCode==0`（成功，哪怕 stderr 恰好为空，如某些 extract 只写输出文件）、`output` 非空（ffmpeg 确实跑了，含 `-i` 恒非 0 的正常枚举）。显式 `HIBIKI_FFMPEG` 覆盖仍走第一分支、不进 bundled 分支，旧契约不变（如实报错、不悄悄换 PATH）。一处修，字幕枚举/抽取、cue 动图、制卡音频、视频帧全链路受益。
- **[x] ② 已加自动化测试**（`hibiki/test/media/video/ffmpeg_executable_resolve_test.dart`）：新增 ① bundled 加载期崩（returnCode = STATUS_DLL_NOT_FOUND / STATUS_ENTRYPOINT_NOT_FOUND / 通用非 0，stderr 空）一律回退 PATH 取到字幕枚举输出；② 同样跨平台（非 Windows，returnCode=127 空输出）回退；③ 正常 `-i` 枚举（退出码非 0 但 stderr 满是流信息）**不**回退、只调一次 bundled；④ 超时（returnCode:null + 空）**不**回退（保持慢 IO 契约）。原 BUG-275 的 invalid-image / ProcessException 回退用例全部仍绿（新条件向后兼容）。`flutter test test/media/video/ffmpeg_executable_resolve_test.dart ffmpeg_backend_test.dart ffmpeg_backend_selection_test.dart video_subtitle_source_test.dart` = 66 passed / 1 skipped（skip 为本机有 ffmpeg、无 ffmpeg 的真跑用例按设计跳过）。`flutter analyze` 改动文件 No issues。
- **备注**：**仍需真机确认**——本机无法复现「用户那台机器上 bundled ffmpeg.exe 究竟是哪种损坏」。若用户系统 PATH 上**也没有**可用 ffmpeg，则回退到 PATH 仍取不到 ffmpeg → 字幕枚举仍空，此时是**打包侧问题**（bundled ffmpeg 的依赖 DLL 为何缺失/被杀软隔离，或 ffmpeg-min 产物在 CDN/打包阶段被破坏），不属本运行时修复范围。需用户用其失败的 Windows 安装包（程序目录里的 ffmpeg.exe + 一个有内封字幕的真实视频）复测字幕轨现在能否枚举/加载；若仍空，请提供该视频的 `ffmpeg -i <file>` 完整 stderr + 程序目录里 `ffmpeg.exe -version` 的退出码与输出，以判断是「bundled DLL 缺失（本修已覆盖回退）」还是「PATH 也无 ffmpeg（需装/重打包）」。
