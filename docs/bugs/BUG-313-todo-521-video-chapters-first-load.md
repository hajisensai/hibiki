## BUG-313 · TODO-521: 视频章节首次加载缺失
- **报告**：2026-06-18（用户：`视频的章节经常要第二次加载才能加载出来，修复`）
- **真实性**：✅ 真 bug。根因在 `hibiki/lib/src/media/video/video_player_controller.dart:749` 附近：修复前 `load()` 在 `player.open()` 后立刻 `unawaited(refreshChapters())` 读取 libmpv `chapter-list`，但 media_kit/libmpv 此时常还没解析出 duration/章节元数据，首次读到空列表后没有再基于真实就绪信号重读。页面层还有一个竞态：`VideoHibikiPage` listener 先挂、`_controller` 后赋值，章节通知早到时会读到 null/旧 controller；进度条 marker 依赖 `durationMs`，controller 之前也没有订阅 `player.stream.duration` 触发重绘。
- **[x] ① 已修复** — `VideoPlayerController.load()` 每次换片递增 `_loadToken`，先清空旧章节并通知，取消上一片 duration 订阅；随后用 `_refreshChaptersWhenDurationReady()` 等 `player.state.duration` 或 `player.stream.duration` 出现 `> 0` 的真实就绪信号再读 `chapter-list`，并在 `_refreshChaptersForLoad()` 的每个异步阶段用 player identity + load token 丢弃旧媒体结果（`video_player_controller.dart:599`、`:605`、`:629`、`:761`、`:1177`、`:1682`）。`VideoHibikiPage` 改为 controller 身份闭包 listener，并在 `_applyLoad()` 的同一次 `setState` 中同时写 `_controller` 与 `_hasChapters`，之后显式同步一次，避免章节入口/marker 被 null 或旧 controller 状态污染（`video_hibiki_page.dart:1685`、`:1694`、`:1702`、`:5457`、`:5485`）。
- **[x] ② 已加自动化测试** — `hibiki/test/media/video/video_chapter_test.dart` 增加 TODO-521 源码契约，覆盖 duration ready 触发、禁止 `Future.delayed`/立即 `refreshChapters()`、订阅生命周期、load token guard、换片清旧章节；`hibiki/test/pages/video_chapter_first_load_guard_test.dart` 覆盖页面 controller/`_hasChapters` 同帧对齐和 controller 身份 listener；`hibiki/integration_test/video_chapter_first_load_test.dart` 新增可执行首次打开验证：用 ffmpeg 生成带 3 个 MKV chapters 的短视频，seed 到 `VideoHibikiPage`，首次打开后等待 duration/chapter ready，断言章节数、章节面板文本和 `VideoChapterMarkers` 出现。
- **备注**：没有使用固定延迟修章节竞态；duration 订阅在换片和 dispose 均取消；旧片章节在新 load 开始时清空，避免换集残留。
