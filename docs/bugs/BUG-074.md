## BUG-074 · 视频字幕播放完不消失，一句播完到下一句开始前一直挂着
- **报告**：2026-06-06（用户：「字幕播放完没消失，一句播完，下一句开始前一直存在」）。 采番注：原取 BUG-073，提交后发现并发 worktree（libmpv TrueHD 换源）已占 073，为避免合并碰撞顺延取 074。
- **真实性**：✅ **真 bug（沿真实代码路径定位）**。视频底部字幕 overlay（`VideoSubtitleOverlay`）监听 `VideoPlayerController.currentCue`：`currentCue==null` 时显示 `SizedBox.shrink()`（消失），非空时画黑底白字框。`VideoPlayerController.updateCueForPosition`（`hibiki/lib/src/media/video/video_player_controller.dart:313`，原 `// Gap：保持上一条 cue 不清高亮，避免闪烁` + `if (idx < 0) return;`）当播放位置落在两条字幕之间的静音 gap（`findCueIndex` 返回 -1）时**直接 return、不清 `_currentCue`**，于是 overlay 继续显示上一句——正是「一句播完到下一句前字幕一直在」。根因是当初「照搬有声书 `_updateCurrentCue` 语义」时把 gap-sustain 一并抄了过来：有声书的 `currentCue` 驱动的是正文里的**跟随高亮**（读者顺读时高亮停在上一句、不在静音里闪灭是对的），而视频的 `currentCue` 驱动的是**独立字幕条**（时间窗 `[startMs,endMs]` 结束后就该消失）。两者语义不同却共用了同一分支。`findCueIndex` docstring 本就写明 gap 返回 -1 是「让上层清高亮」，视频层却没清，违背契约。根因 `hibiki/lib/src/media/video/video_player_controller.dart:313`。
- **[x] ① 已修复** — 本提交（自含修复+测试+本条目；并发抢号致哈希随 rebase 变动故不内联记录）：`updateCueForPosition` 在 `idx < 0`（gap / 早于首句）时清空 `_currentCue` / `_currentCueIndex` 并 `notifyListeners()`，让 overlay 收到空 cue 后消失；已无字幕（`_currentCueIndex == -1`）时直接 return 避免 gap 内逐 tick 无谓 notify。**只动视频 controller，不碰有声书** `_updateCurrentCue`（它的 gap-sustain 对正文跟随高亮仍正确）。同步更新类级与方法级 docstring 标明此处与有声书的关键差异。
- **[x] ② 已加自动化测试** — `hibiki/test/media/video/video_player_controller_test.dart`：将原把错误行为固化的 `gap keeps previous` 用例改为 `gap clears subtitle`（红→绿实测）；新增 BUG-074 用例覆盖「句间 gap 清空 + 末句之后清空 + gap 内多 tick 不重复 notify」与「早于首句无字幕且不 notify」。media_kit/libmpv 测试宿主不可用，cue 选择逻辑经 `debugUpdateCueForPosition` + `setCues` 纯逻辑单测（控制器 Player 延迟构造）。`test/media/video/` + `test/media/audiobook/` 全量 +468 绿（~2 skip），有声书未受影响。
- **备注**：真机/真桌面端用带句间静音 gap 的真实视频字幕复测「一句播完→字幕消失→下一句出现」**待用户**。
