## BUG-481 · 阅读器双击原生框选打扰查词
- **报告**：2026-06-30（用户：·TODO-1028）
- **真实性**：✅ 真 bug。根因 `hibiki/lib/src/pages/implementations/reader_hibiki/webview.part.dart:866-868`（`selectstart` 抑制只挡拖动起手 400ms 内、不挡双击）+ 正文上 `_hoshiReaderMouseNativeTextStart=true` 放过双击原生选词，且无 `user-select:none` 兜底 → WebView 原生双击建立蓝色框选，盖住单击查词的 CSS Highlight、并把双击的振假名整页切换（`webview.part.dart:285-291` `_buildFuriganaJs` 'toggle' 分支带 `if(sel && !sel.isCollapsed) return` 守卫）绊住（双击自己产生的选区让守卫早退，振假名切换不触发）。单击查词走 `callHandler('onTap')→_selectTextAt` 自管 CSS Highlight，与原生选区分离，故只需清掉双击建立的原生选区即可。
- **[x] ① 已修复** — `webview.part.dart:869-878` 在 `selectstart` 监听后加 capture 阶段 `dblclick` 监听：`window.getSelection()` 非折叠则 `removeAllRanges()` 清掉。capture 让它先于振假名 handler（默认 bubble）跑：双击选区被清 → 振假名守卫的 `!sel.isCollapsed` 不再被绊住，整页振假名切换反而恢复正常。用 `removeAllRanges` 而非 `preventDefault`（原生选词在 mousedown/selectstart 已发生，dblclick 只是结果，preventDefault 拦不住）。向后兼容：单击查词不产生原生选区零影响；长按/拖选/桌面 Ctrl+C 复制/右键导出片段是单次 down→drag→up 不触发 dblclick，`nativeSelectionSentenceRange` 继续工作；翻页零影响。
- **[x] ② 已加自动化测试** — `hibiki/test/reader/reader_dblclick_clear_selection_behavior_test.dart` + 同名 `.js`：node-vm 跑从 `webview.part.dart` 抽取的真实 dblclick 监听 + 从 `_buildFuriganaJs` 抽取的 'toggle' 振假名 handler（capture→bubble 顺序派发），断言双击后 `getSelection().isCollapsed===true`（框选清）且 `show-all-rt` 仍被 toggle（振假名保留）；撤掉本修复（去掉 removeAllRanges）则振假名守卫被双击选区绊住、`show-all-rt` 未切换 → 红。+ 源码守卫 `hibiki/test/reader/reader_dblclick_clear_selection_guard_test.dart` 断言 `webview.part.dart` 含新 dblclick→removeAllRanges（capture），且仍保留振假名 dblclick→show-all-rt（防后人误删/混淆两者）。
