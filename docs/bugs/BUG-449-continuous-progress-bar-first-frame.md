## BUG-449 · 连续模式进度条初次不显示·滑动一下才出来
- **报告**：2026-06-28（用户：）· TODO-933
- **真实性**：✅ 真 bug。根因竞态：`hibiki/lib/src/pages/implementations/reader_hibiki/navigation.part.dart:65` `_onRestoreComplete` 在 `:131` 调 `_reanchorContinuousAfterRestore()`（不 await）→ `runUiScaleReanchorOrchestration`（`reader_hibiki_page.dart:493`）的 `evalBegin` 在 JS 侧同步置 `window.hoshiReader._reanchorPending = true`（清旗推迟到 postFrame 的 `evalCommit`）；紧接着 `:143` 首发 `_refreshProgress()` 执行，但 JS `stableProgressInvocation`（`hibiki/lib/src/reader/reader_pagination_scripts.dart:482`）= `!_reanchorPending ? hoshiProgressDetails() : null` → 旗为 true 返 null → `_refreshProgress` 在 `navigation.part.dart:667` 早退（`if (result == null) return;`），`_progressCurrentChars`（`reader_hibiki_page.dart:844`，初值 null）保持 null → `_showTopProgress`（`reader_hibiki_page.dart:1029`）要求其非 null 故进度条隐藏。用户滑动时旗已 commit 清掉 → 刷新成功 → 条出现。分页模式 reanchor 门控（`continuousMode==false`）整体不触发，故本 bug 仅在连续/滚动模式。系 TODO-718 reanchor（2026-06-25）引入的副作用。
- **[x] ① 已修复** — 把"首次进度 seed"与 reanchor 解耦：给编排核心 `runUiScaleReanchorOrchestration`（`hibiki/lib/src/pages/implementations/reader_hibiki_page.dart:493`）加可选 `onAfterCommit`，在 `evalCommit` 成功**清旗之后**确定性回调一次（commit 抛异常则不调；postFrame 内 try/catch 经 `onCommitError` 吞异常不外抛）。恢复路径 `_reanchorContinuousAfterRestore`（`hibiki/lib/src/pages/implementations/reader_hibiki/chrome.part.dart:446`）把它接到 `onAfterCommit: () => _refreshProgress()`，旗已清不再撞 null gate，首屏进度条确定性 seed。缩放/样式重锚（`_reanchorContinuousForUiScale` / `_reanchorForStyleChange`）不传 `onAfterCommit`，行为不变。提交：见分支 fix-933-progress-seed。
- **[x] ② 已加自动化测试** — `hibiki/test/reader/reanchor_after_commit_progress_seed_test.dart`（6 例，对编排核心做运行时断言）：commit 成功后 `onAfterCommit` 严格在 commit 清旗之后确定性回调；不传 `onAfterCommit`（缩放/样式路径）则 commit 后无补刷；commit 抛异常/begin 返回 -1/门控抑制时不补刷；补刷自身抛异常经 onCommitError 吞掉。red→green：注释掉编排里的 `onAfterCommit` 调用块 → "commit 成功后确定性回调"与"补刷自身抛异常"两例红；恢复后 6 例全绿。
- **备注**：与 BUG-380（滚动进度停下才更新，另一通道）区分；不影响分页模式/跨章/其它 `_refreshProgress` 调用点。真机：连续模式开书/退出再进首屏进度条立即可见留用户验收。
