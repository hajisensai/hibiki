## BUG-054 · 调整「界面大小」后词典查词结果文字发糊（词典 WebView 漏接界面缩放中和器，与阅读器 BUG-039 同因）
- **报告**：2026-06-05（用户：「调整界面大小以后，词典管理里面的文字会糊，修复」，并追问「根本性修复呢，为什么总有这种情况」）。采番注：动手前查 `docs/BUGS.md` 当前最大号为 BUG-053，故本条取 054（并发 agent 多，编号可能被抢，以实际提交为准）。
- **真实性**：✅ **真 bug（WebView 纹理被全局缩放拉糊），且是阅读器 BUG-039 同一根因在多个未覆盖表面上的重复**。先验真伪+定根因：全局「界面大小」是 `HibikiAppUiScale`（`lib/src/utils/app_ui_scale.dart:15-84`）用 `FittedBox(BoxFit.fill)` 把整棵渲染树当一张画布（`canvas=view/s`）拉伸 s 倍。原生矢量内容（按钮/文字/图标）被 Skia 按最终变换重栅格化=清晰；但 **WebView 是平台视图纹理**，先按逻辑尺寸渲染成纹理再被 GPU 拉大 → 丢分辨率发糊。用户口中的「词典管理」经确认是**首页「词典」标签查词结果页**（`home_dictionary_page.dart:439` 的 `DictionaryPopupWebView`），其释义内容走 WebView。（澄清：真正的「管理已安装词典」页 `dictionary_dialog_page.dart` 全是原生组件、缩放下不会糊。）移动端真 WebView 无离屏过采样旋钮（只 Windows fork 有 `put_RasterizationScale`），跨平台唯一干净解法=让词典 WebView 永远在**原生密度**渲染（净缩放=1，内容大小走 WebView 自带字号）——即必须处在 `HibikiAppUiScaleNeutralizer` 之下（与阅读器 BUG-039 完全同法）。**根因 = 中和器之前只在阅读器 route 上接了（`reader_hibiki_source.dart:115` / `reader_hibiki_history_page.dart:755`），App 里其它处在全局缩放下、挂载 `DictionaryPopupWebView` 的活跃表面全漏接**：① 首页词典标签查词结果（`home_dictionary_page.dart`，用户所报）；② 弹窗词典窗口（`popup_dictionary_page.dart`，经 `popup_main.dart:166` 同样套了 `HibikiAppUiScale`）。于是用户一个个表面撞糊来报——「为什么总有这种情况」即源于此（逐页打补丁、没把不变式强制住）。（排除项：悬浮词典 `floating_dict_page.dart:215` 走原生 `DictionaryPopupNative` 且 `floating_dict_main` 不套缩放；native 渲染链 `DictionaryResultPage→DictionaryHtmlWidget` 全仓无构造点=死代码。）
- **[x] ① 已修复** — `001c905d9`：根因修（Linus 式：不再逐页补丁，而是把「词典 WebView 必须在原生密度渲染」这条不变式**强制执行**）。改动：①给所有漏掉的活跃表面接中和器——`home_dictionary_page._buildSearchResultBody()` 把 `HibikiAppUiScaleNeutralizer` 包在 `LayoutBuilder` **外层**（含 `DictionaryPopupWebView` + 同坐标系嵌套弹窗整块；内层 `constraints/screen` 还原成真实视口，WebView 与弹窗共用净缩放=1 坐标系，`selectionRect` 定位不错位）；`popup_dictionary_page` 把 `HibikiOverlayScaffold.body` 的 `_buildOuterContainer()`（透明关闭遮罩+卡片+嵌套层）整块中和。②**装上「忘了就报错」的运行时不变式守卫**——在共享叶子 `DictionaryPopupWebView.build` 顶部加 `assert(HibikiAppUiScale.of(context) == HibikiAppUiScale.defaultScale, ...)`：`of()==defaultScale` 同时覆盖「全局未缩放(=1)」与「已被中和器中和(=1)」两种合法态，**唯一触发条件是「被全局缩放且未中和」=发糊的精确条件**；以后任何人再加一个词典 WebView 表面忘了套中和器，会在 debug/集成测试里立刻炸，而非等用户撞糊（assert 仅 debug，release 无行为变更）。**关键不变式（务必守住）**：必须整块中和「WebView + 同坐标系兄弟弹窗」，不能只中和 WebView——否则 WebView 在原生坐标、弹窗仍在缩放坐标会 factor-s 错位（被撤销的 HibikiNativeScale 老坑）。`dart format` 通过、改动文件 `flutter analyze` 0 issue。Opus 审查 0 Critical（PASS）并采纳两条增强：assert 取 `s` 一次复用、守卫加「中和器在 LayoutBuilder 外层」顺序断言。
- **[x] ② 已加自动化测试** — `001c905d9` · `hibiki/test/pages/dictionary_webview_neutralizer_wired_test.dart`（源码扫描守卫，最强可落地层——`DictionaryPopupWebView` 需完整 `appProvider`+平台 WebView 无法在单测 pump，故沿用 `reader_neutralizer_wired_test.dart` 同范式）三例：①断言 `dictionary_popup_webview.dart` 的 `build` 带 `assert(`+读 `HibikiAppUiScale.of(context)`+比对 `HibikiAppUiScale.defaultScale`（守住运行时不变式存在）；②断言 `home_dictionary_page.dart` 含中和器**且** `indexOf('HibikiAppUiScaleNeutralizer') < indexOf('child: LayoutBuilder')`（守住中和器在 LayoutBuilder 外层这一关键层级，防未来错放进内层重蹈 HibikiNativeScale 坑）；③断言 `popup_dictionary_page.dart` 含中和器且包裹 `_buildOuterContainer`。3 例全绿；相邻 `test/pages/` 全量 179 例、中和器行为测试 `app_ui_scale_neutralizer_test.dart`、阅读器接线守卫均无回归。
