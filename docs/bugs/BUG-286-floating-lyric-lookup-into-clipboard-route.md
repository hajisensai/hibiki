## BUG-286 · 悬浮字幕点词复用剪贴板查词出口而非主app内浮层
- **报告**：2026-06-15（用户：Windows 桌面悬浮字幕条点词查词时，弹窗别弹进主 app 窗口；复用剪贴板查词那套逻辑。TODO-376）
- **真实性**：✅ 真 bug（交互问题）。根因：`hibiki/lib/src/pages/implementations/reader_hibiki_page.dart` 的 `_lookupFromFloatingLyric`（TODO-354 引入）把桌面悬浮字幕（独立 always-on-top 原生窗口 `windows/runner/floating_lyric_window.cpp`，点词经 `FloatingLyricChannel.onLookupText` 回调）点的词送进**阅读器页面内的 in-app 浮层**（`searchDictionaryResult` + `_highlightAndShowPopup`，定位在 reader 屏幕中心）。而桌面「剪贴板查词那套逻辑」的出口是 `hibiki/lib/src/sync/desktop_lookup_service.dart`（剪贴板监听 + Ctrl+Shift+D 全局热键 → `submitText` → `pendingText`）→ `hibiki/lib/src/pages/implementations/home_dictionary_page.dart:84` 的 `_onDesktopLookupPending` 消费 → 主窗**查词 tab** 展示 + `bringPendingLookupToFront`（已前台时 no-op，TODO-341）。两个出口分裂：悬浮字幕点词没复用剪贴板查词出口。
- **[x] ① 已修复** — 提交 <PENDING>。
  - `desktop_lookup_service.dart` 新增显式查词入口 `triggerLookup(String text)`：清 `_lastText` 越过去重 + `submitText`（热键 `_onHotKey` 也改走它），把「悬浮字幕点词」与「热键查词」统一成同一剪贴板查词管线/出口。
  - `reader_hibiki_page.dart`：抽顶层纯函数 `floatingLyricSearchTerm(text/index/word)` 解析待查词（分词器优先、整句回退、双空 no-op）；`_lookupFromFloatingLyric` 桌面分支改为 `DesktopLookupService.instance.triggerLookup(searchTerm)` + `bringPendingLookupToFront()`，**不再走 reader in-app 浮层**；非桌面 no-op（Android 走 PopupDictActivity，不到此 handler）。
  - **返工（复核退回）**：上一轮（9d1270665）让 `home_page.dart` 根节点常驻监听 `DesktopLookupService` 自动切 tab，撞红既有守卫 `home_page no longer owns desktop clipboard lookup lifecycle`，且 `_onHotKey` 改道 `triggerLookup` 撞红 `DesktopLookupService 只排队命中词…` 守卫；同时真正的消费缺口未除（`HomeDictionaryPage` 只在 `desktopClipboardEnabled` 为真时才消费已存在 pending，悬浮字幕点词由 `floatingLyricClickLookup` 控制与之无关，关剪贴板监听的默认用户 pending 卡死）。本轮按推荐方案 (a) 重做：
    - `desktop_lookup_service.dart`：`_onHotKey` 恢复直接 `_lastText = null; submitText(text)`（满足守卫），`triggerLookup` 仅留给悬浮字幕显式查词；并把 `triggerLookup` 移出 `_onHotKey..._readClipboardText` 区间，避免其 doc 里的 `bringPendingLookupToFront` 误命中守卫切片。
    - `home_dictionary_page.dart`：`initState` 排一帧**无条件**消费一次已存在的 pending（`addPostFrameCallback`→`_onDesktopLookupPending()`，不再门控在 `desktopClipboardEnabled` 分支）；`_startDesktopLookupIfEnabled` 删除其内的消费调用。这才是根因修复：默认用户（剪贴板关 + 悬浮点词开）点词后查词 tab 真能显示。`_onDesktopLookupPending` 自带「有 pending 才消费」边界，无 pending 不乱消费。
    - `app_model.dart`：新增显式导航原语 `homeDictionaryTabRequest`（`ValueNotifier<int>`）+ `requestHomeDictionaryTab()`，与被动剪贴板正交。
    - `reader_hibiki_page.dart`：`_lookupFromFloatingLyric` 桌面分支 `triggerLookup`（排队 pending）→ `bringPendingLookupToFront()`（唤前台）→ `appModel.requestHomeDictionaryTab()`（请主窗切查词 tab）。pending 在请求切 tab 之前就位，`HomeDictionaryPage` 挂载即消费。
    - `home_page.dart`：只监听 `homeDictionaryTabRequest` 切 tab（不监听 `DesktopLookupService`，不在剪贴板被动命中时切 tab），重新满足旧守卫；并修低问题：`reader_hibiki_page.dart` 新增 import 按字母序归位（`src/sync/` 放到 `src/startup/` 之后）。

- **残留限制（需真机复测）**：reader 是 `Navigator.push` 全屏路由覆盖在 home 之上，且 `closeMedia` 会 `audioHandler.stop()`（关 reader 即停有声书）。故当用户**正前台全屏阅读 + 悬浮条**时，点词结果落在被 reader 遮挡的 home 查词 tab，需退出阅读器才可见（不能为看词强退 reader 停音频）。这是「复用剪贴板查词出口」的固有特性，与用户决策一致；典型悬浮字幕场景（主窗最小化 / 后台听书 / 停在 home）下 `bringPendingLookupToFront` 唤起主窗 + 自动切查词 tab，完全满足「不弹进主 app 阅读器浮层」。Windows 真机复测：①后台听书态点悬浮字幕词 → 主窗前台、自动切查词 tab、显示该词；②**默认用户（关剪贴板监听、只开悬浮字幕点词）**点词同样能在查词 tab 显示（返工修复重点）；③连点同一词每次都能再查；④主窗已前台时不触发任务栏闪烁（TODO-341 不回退）。
- **备注**：仅改桌面悬浮字幕点词路由 + 剪贴板查词出口常驻化；Android 悬浮词典 `PopupDictActivity` 路径与移动端剪贴板行为不变。
