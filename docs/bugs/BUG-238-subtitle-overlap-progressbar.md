## BUG-238 · 进度条出现时字幕只往上动一点点、仍被进度条遮挡（移动端）
- **报告**：2026-06-13（用户：底部进度条 / 控制条出现时字幕只「动了一点点」，还是被进度条压着；TODO-279。BUG-228 备注②已预判此后续）
- **真实性**：✅ 真 bug（BUG-228 备注②「若移动端默认 75 仍被高进度条遮，考虑把 reserve 提到移动端进度条上缘量级或按平台分值」的兑现）。根因分两层：
  - **reserve 是常量、不随界面缩放**：`hibiki/lib/src/media/video/video_subtitle_style.dart:30`（修复前）`kVideoControlsBottomReserve = _kButtonBarHeight = 56`。overlay 默认 `controlsBottomReserve = kVideoControlsBottomReserve`（`hibiki/lib/src/media/video/video_subtitle_overlay.dart:57`），视频页**没显式传**（`video_hibiki_page.dart` 构造 `VideoSubtitleOverlay` 时缺 `controlsBottomReserve` 实参）→ 永远用裸常量 56。视频页整页被 `HibikiAppUiScaleNeutralizer` 中和回 scale=1.0，控制条几何全靠各 getter ×`_videoUiScale` 手动放大，但 reserve 不在其中 → 放大界面后控制条变高、reserve 仍 56 → 盖不住。
  - **reserve < 默认基线 → 取下限失效**：避让在 `_paddingFor`（`video_subtitle_overlay.dart:433-446`）用 `controlsVisible ? max(bottomPadding, controlsBottomReserve) : bottomPadding`（取下限，非加法——加法会顶飞，BUG-226）。默认基线 `bottomPadding=75`（`video_subtitle_style.dart:88`）> reserve 56 → `max(75,56)=75`，控制条可见/不可见都是 75，**字幕完全不抬升**（用户感知「只动一点点」=实际 0）。而**移动端** `_mobileControlsTheme`（`video_hibiki_page.dart:3017-3018,3058`）把进度条整体抬到按钮行上方，其热区上缘距视频底 ≈ `离底基线 24 + 系统 inset + 按钮行 56 + 间距 8 + 热区 52 ≈ 140px`，远高于 75 → 字幕停在 75 被进度条压住。桌面进度条骑按钮行上沿（约 56px），默认 75 已在其上，故桌面观感正常（BUG-228 调好的就是桌面）。
- **[x] ① 根因修复** — reserve 改成「按平台真实控制条几何加总 + 随界面缩放」并由视频页显式传入，消除「常量 56 < 基线 75 → 不抬升」失效区间，保留 BUG-226 的取下限语义（不顶飞高位字幕）：
  - `video_subtitle_style.dart`：新增纯函数 `videoSubtitleControlsReserve({isDesktop, buttonBarHeight, seekBarButtonGap, seekBarContainerHeight, bottomChromeBaseline, bottomSystemInset})`。桌面=一个按钮行高（进度条骑按钮行上沿，保 BUG-228「只让进度条那一条」观感，但现在随缩放）；移动=`bottomChromeBaseline + bottomSystemInset + buttonBarHeight + seekBarButtonGap + seekBarContainerHeight`（=被抬高的移动进度条热区上缘，scale=1.0 ≈140 > 75 → `max(75,140)=140` 真抬升盖过进度条）。几何项由调用方传入已 ×缩放值，不二次缩放。`kVideoControlsBottomReserve`=56 保留为 overlay 默认 / 无控制条场景兜底（不破 BUG-228 守卫与既有 overlay 测试）。
