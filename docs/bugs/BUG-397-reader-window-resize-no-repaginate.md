## BUG-397 · 拖窗口边框后阅读器不重排文字错乱
- **报告**：2026-06-22（用户：）
- **真实性**：✅ 真 bug。根因 `hibiki/lib/src/pages/implementations/reader_hibiki_page.dart:1280`（`didChangeMetrics`）—— 唯一 resize→重排入口是 `didChangeMetrics → _syncPageSize`（同文件 :1302）。桌面 window_manager `onWindowResized`（`hibiki/lib/main.dart:439`）只记窗口边界、不通知 reader；且 Windows 拖窗口边框时 `didChangeMetrics` / `MediaQuery.of(context).size` 更新滞后，JS 端分页几何缓存（`--page-width/height` / `this.pageWidth` / `_contW` / `paginationMetrics`，见 `reader_pagination_scripts.dart`）无人失效 → 正文按旧分页几何渲染、文字错乱、不自动重排；下一次翻页用实时 `getScrollContext` 重算才恢复。
- **[x] ① 已修复** — 在阅读器树内（`HibikiAppUiScaleNeutralizer` 之下、WebView 子树外层）包**透明** `LayoutBuilder`（`reader_hibiki_page.dart` `build` :1460 起，零几何变换）：builder 把 `constraints` 交给新方法 `_onReaderConstraintsChanged`（:1405），用纯谓词 `readerLayoutResizeNeedsRepaginate`（:194，复用 `readerViewportNeedsRepaginate` 的 1px 容差与 `lastWidth>0` 门控）判定，超阈值起 ~50ms 尾沿防抖 `Timer`（`_resizeRepaginateDebounce` 字段 :812），回调调 `_syncPageSize()`（与 `didChangeMetrics` 路径靠 `_lastSyncedWidth/Height` 基线天然去重幂等）；timer 在 `dispose` 取消（:1228）。约束由布局系统每帧驱动，比 `didChangeMetrics` 更早更可靠。提交哈希：见本分支提交（待回填）。
- **[x] ② 已加自动化测试** — 纯谓词 `readerLayoutResizeNeedsRepaginate` 真值表：`hibiki/test/pages/reader_viewport_repaginate_test.dart`（sub-pixel 抖动不触发 / >=1px 宽高变触发 / 首帧 / 同尺寸不重复）。源码扫描守卫：`hibiki/test/reader/reader_window_resize_repaginate_guard_test.dart`（透明 LayoutBuilder 包裹且调 `_onReaderConstraintsChanged` / 纯谓词 + 50ms Timer 尾沿防抖调 `_syncPageSize` / 禁 `Future.delayed` / timer 在 dispose 取消）。
- **备注**：文字错乱本身是 WebView2 渲染产物，widget 测试照不到——**必须 Windows 真机**（或 `tool/run_windows_itest.ps1` 离屏验接线）最终肉眼/截图确认 resize 后正文立即按新尺寸重排、不溢出/裁切、连续模式不弹回章首、分页模式不跳页。
