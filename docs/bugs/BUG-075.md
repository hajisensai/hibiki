## BUG-075 · 降级（旧版装到新版之上）路径在 foreign_keys=ON 下 DROP 表崩溃 + 非原子半删毁库
- **报告**：2026-06-06（用户先报 `no such table: preferences`，恢复 v20 备份后再报 `no such table: main.epub_books / DROP TABLE IF EXISTS "book_tags"`）。采番注：动手时基线最大号 BUG-074，取 075（以提交为准，可能被并发抢）。
- **真实性**：✅ **真 bug（沿真实代码路径定位 + 在 DB 文件实证 + 测试逐字复现报错）**。`HibikiDatabase.migration` 的降级分支（`packages/hibiki_core/lib/src/database/database.dart` `if (from > to)`）在旧版本 app 打开更高 schema 的 DB 时，按设计「备份旧库到 `.bak.v<from>.<ts>` → DROP 所有表 → `createAll()` 重建空库」。两处致命缺陷：
  - **① FK 顺序崩溃**：生产经 `_openDb` 的 `createInBackground(setup:)` 开 `PRAGMA foreign_keys = ON`。删表按 `allTables`（即 `@DriftDatabase` 声明顺序）进行：`EpubBooks` 先于 `BookTags`、`BookTagMappings`。`book_tag_mappings` 同时外键引用 `epub_books(book_key)` 与 `book_tags(id)`（均 `ON DELETE CASCADE`，见 `tables.dart:259/261` 与生成 `database.g.dart`）。当 `epub_books` 已删、再 `DROP TABLE book_tags` 时，FK=ON 让 SQLite 校验仍存在的 `book_tag_mappings`，解析其 FK 需已删的 `epub_books` → `no such table: main.epub_books`，迁移中途抛错。
  - **② 非原子半删 → 永久损坏**：该 drop 循环不在事务里，每条 `DROP` 各自 auto-commit。①崩在半路 → 已删的基线表（preferences/media_items/epub_books/reader_positions/bookmarks/audiobooks/audio_cues/reading_statistics/dictionary_* 等只在 `onCreate→createAll` 建、升级阶梯里从不重建的表）回不来，而 `user_version` 没回退。之后任何构建打开它：若 `user_version==schemaVersion` 直接跳过迁移、首条 `getPref`（`SELECT * FROM preferences WHERE key = ?`，启动早期 `migrateSmbToWebDav` 触发）即 `no such table: preferences`；若 `user_version<schemaVersion` 则只跑升级阶梯里 `m.createTable()` 那些表、基线表仍缺，user_version 被推高、坑死。实证：用户机 `hibiki.db` 一度 `user_version=22` 仅 13 表（缺 preferences），剩下的恰好都是阶梯 `createTable` 建的（profiles/tags/sync_baselines/video_*）。根因 `packages/hibiki_core/lib/src/database/database.dart:85`（降级 drop 循环 FK=ON + 非原子）。
  - **测试盲区**：`migration_downgrade_test.dart` 此前 seed `user_version=17`、断言落 `16`，schemaVersion 升到 22 后 17<22 变成**升级**、降级分支根本不跑（且断言 stale），故 FK=ON 崩溃从未在 CI 暴露——「reconcile after v22 rebase」漏改了它。
- **[x] ① 已修复** — 本提交（自含修复+测试+本条目；并发抢号致哈希随 rebase 变动故不内联记录）。**根因修**降级分支：teardown 前 `PRAGMA foreign_keys = OFF`、`createAll()` 后恢复 `ON`（此迁移非事务，故 PRAGMA 生效——已由红→绿测试验证）使删表顺序无关；删表来源由 `allTables` 改为扫 `sqlite_master`（清掉降级时残留的高版本表，并堵住「构建用残缺 allTables 漏删基线表」的同类隐患）。语义不变（仍是「备份→清空→createAll 重建」），只让它在 FK=ON 下不再崩、不再半删。
- **[x] ② 已加自动化测试** — `packages/hibiki_core/test/migration_downgrade_test.dart`：① 修好 stale 用例（seed `user_version=99` 真触发降级、断言对 `db.schemaVersion` 动态比较，永不再因版本号 bump 失效）；② 新增 BUG-075 用例——`PRAGMA foreign_keys = ON` + 真实 FK 链（epub_books/book_tags/book_tag_mappings，照搬生成 FK 子句）+ 未来版本，未修时逐字复现 `no such table: main.epub_books / DROP ... book_tags`，修后断言落当前 schema、preferences 与 FK 表均重建。`packages/hibiki_core` 降级测试 2 绿。
- **备注**：① 同文件另有 3 个**预存 stale 失败**（`migration_book_key_test` / `migration_orphan_bookmark_test` / `migration_partial_v16_shape_test`，均硬编码「落 v16」，schemaVersion=22 下失效——已在基线撤改动验证非本次引入），属同类「reconcile after v22 漏改」，**未在本次修复范围内**，待单独处理。② 真机/真桌面端复测：把 DB 升到高版本后用低版本构建打开，应「干净重建空库 + 旧库进 `.bak`」而非崩溃**待用户**（后台跑不了 GUI 构建）。③ 用户机数据已由我从 `.bak.v16.1780592923530`（v15/23表/6书）恢复，损坏库留底 `hibiki.db.corrupt-downgrade-*`。
