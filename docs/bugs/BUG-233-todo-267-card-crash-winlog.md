## BUG-233 · Reader card mining fails when bundled ffmpeg is invalid
- **报告**：2026-06-13（用户：制卡闪退了，看 win 日志）
- **真实性**：真 bug。Windows Application event log 在 2026-06-13 18:32:52 记录 `hibiki.exe` 0.5.9.8 在 `GraphicsCapture.dll` 以 `0xc0000005` 崩溃；同一时间段项目 `error_log.txt` 连续记录 reader 制卡路径 `extractAudioSegmentViaFfmpeg -> onMineFromPopup` 的 `ffmpeg exit -1073741701`。本机实测 `D:\APP\Hibiki\ffmpeg.exe -version` 同样返回 `-1073741701`，而 PATH 上 `D:\APP\ffmpeg\bin\ffmpeg.exe` 正常。
- **根因**：`hibiki/lib/src/media/video/ffmpeg_backend.dart` 桌面 CLI 后端只按 `HIBIKI_FFMPEG > 程序旁 bundled ffmpeg > PATH` 解析可执行文件，未验证 bundled ffmpeg 是否为可运行的 Windows 可执行；一旦 bundled `ffmpeg.exe` 损坏/架构不匹配，所有制卡音频裁剪都直接失败。`hibiki/lib/src/pages/implementations/reader_hibiki_page.dart` 在已决定导出句子音频后，如果 `TtsChannel.extractAudioSegment` 返回 null，仍继续调用 `repo.mineEntry`，导致用户看到制卡路径异常且没有明确失败降级。
- **[x] ① 已修复**：桌面 ffmpeg CLI 后端在无显式 `HIBIKI_FFMPEG` 覆盖时，若 bundled ffmpeg 在 Windows 返回 `STATUS_INVALID_IMAGE_FORMAT`（`-1073741701` / `0xC000007B`）或 `ERROR_BAD_EXE_FORMAT`（193），记录 debug 信息并回退 PATH 的 `ffmpeg`；显式覆盖保持原契约，不自动替换。reader 制卡路径新增 `requestedSentenceAudioClip` 保护：请求了句子音频但裁剪失败时清理临时目录、写 `ReaderHibiki.mineEntry.sentenceAudio` 日志、弹出 `card_export_failed_detail` 并返回 false，不再生成伪成功无音频卡。
- **[x] ② 已加自动化测试**：`hibiki/test/media/video/ffmpeg_executable_resolve_test.dart` 覆盖 invalid bundled ffmpeg 回退 PATH、显式覆盖不回退；`hibiki/test/reader/reader_mining_audio_guard_test.dart` 守住请求句子音频裁剪失败必须在 `repo.mineEntry` 前阻断且用户可见。相邻验证包含 ffmpeg backend、desktop audio clipper、mining audio clip、video mining guard。
- **备注**：真实 Windows 事件日志、WER 摘要、项目日志和本机 ffmpeg 探测证据保存于 `.codex-test/todo-267-winlog-evidence.txt`。当前代码基线已有 BUG-209/WGC framepool teardown UAF 修复，但用户机器安装的崩溃版本是 `0.5.9.8`，低于当前 worktree `0.5.12+45`；仍需后续安装新构建复测 Windows 原生 `GraphicsCapture.dll` 崩溃签名是否消失。
