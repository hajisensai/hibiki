## BUG-045 · Windows 上导入日文命名词典 / 含日文名的推荐词典报「unsupported format or failed to open file」（native 把 UTF-8 路径当 ANSI 解码）
- **报告**：2026-06-04（用户截图：导入中 → `Exception: unsupported format or failed to open file`；追问确认 = Windows 上「选文件导入」和「推荐词典下载导入」都失败）。采番注：动手前查 `docs/BUGS.md` 当前最大号为 BUG-044（040/041/042/043/044 已被并发 agent 占用），故本条取 045。
- **真实性**：✅ **真 bug（native 跨平台边界把 UTF-8 路径当 ANSI 代码页解码，Windows 专有），本机经验性复现**。报错字符串全工程唯一来源 = `native/hoshidicts/hoshidicts_src/importer.cpp:1068`，`zip.open()` 返回 false。`zip.open()` = `memory::map_rd()` + 平台无关的 `parse_central_directory()`；前者 Windows 分支用 **`CreateFileA`（ANSI 代码页解释 `char*`）**，而 Dart 经 FFI 传的是 **UTF-8** 字节。本机 VS2022 独立复现铁证：同一文件 `複合語起源.zip`，`CreateFileA(UTF-8字节)` → `ERROR_INVALID_NAME(123)` 打不开，`CreateFileW(UTF-8→UTF-16)` → OK；ASCII 名 `JMdict_english.zip` 两法皆 OK。触发面：① 用户选日文命名的词典文件（三省堂.zip 等）；② 推荐词典里 `複合語起源`/`日・モ辞典`/`Pixiv` 等 URL 经 `Uri.parse(url).pathSegments.last`（`dictionary_downloader.dart:635`）**解码成日文/特殊字符文件名**，下载到 `download_temp/<日文名>.zip`。POSIX narrow API 本就是 UTF-8，故 Android/macOS/Linux 正常，唯独 Windows 非 ASCII 路径全崩；Windows app 产品名 "Hibiki" 为 ASCII 故 appData 根路径 ASCII，问题只出在**文件名/词典标题**非 ASCII 时（ASCII 名推荐词典如 JMdict_english 其实能过）。同源缺陷遍布 native 写出/回读侧：`std::filesystem::path(std::string)`、`std::ofstream/ifstream(std::string)`、glaze `write_file_json/read_file_json`(内部窄路径) 在 Windows 都把 UTF-8 std::string 当 ANSI，且 `std::filesystem::path::string()` 在 Windows 返回 ANSI 窄串（非 UTF-8）→ 日文**标题**的词典在建输出目录/写 index.json/回读查词时同样崩。根因点 = `memory/memory.cpp` 的 `CreateFileA`/`CreateFileMappingA`（map_rd/map_rw）+ `importer.cpp`/`query.cpp`/`stardict/stardict_reader.cpp` 全部文件系统触点用窄 std::string 路径。
- **[x] ① 已修复** — `344c5f3f9`（+ `fs_utf8.hpp` 去未用 helper 微调）：根因修在唯一正确的层——平台边界，消除「UTF-8 路径被当 ANSI 解码」这个特殊情况（非补丁）。① 新增 `hoshidicts_src/util/fs_utf8.hpp`：`hoshi::fs_path(utf8)`（C++23 `char8_t` 构造 `std::filesystem::path`，Windows 内部存正确 UTF-16）、`hoshi::fs_to_utf8(path)`（`.u8string()` 取回 UTF-8，绕开 `.string()` 的 ANSI 陷阱）。② `memory.cpp` 仅 `_WIN32` 分支改 `CreateFileW`/`CreateFileMappingW` + `MultiByteToWideChar(CP_UTF8)`；POSIX `#else` 分支零改动。③ `importer.cpp`/`query.cpp`/`stardict_reader.cpp`：18 处 fstream + 全部 `std::filesystem::*` 触点过 `fs_path`，所有 `.stem()/.filename()/.string()` 取回改 `fs_to_utf8`；glaze `write_file_json`→`glz::write_json` 到 buffer 再 `fs_path` ofstream 写盘、`read_file_json`→`fs_path` ifstream 读字节再 `glz::read_json`（绕开 glaze 的窄路径）。④ `hash::build_to_file`/`bloom::build_to_file` 经 `memory::map_rw` → 传递性修好，无需改。opus 审查 🟢（穷尽 grep 确认 native 树零残留 `CreateFileA`/裸窄路径 fstream/`glz::*_file_json`/`.string()` 往返；POSIX 零变化、ASCII 等价、glaze 序列化逐字节相同、`MultiByteToWideChar`/`char8_t` 无 UB）。
- **[x] ② 已加自动化测试** — `344c5f3f9`：双层。① **真实端到端**（最强层）`native/hoshidicts/tests/win_utf8_import_test.cpp`：本机 VS2022 用 CMake 构建真实 `hoshidicts.lib`（含全部改动 + zstd/libdeflate/glaze），对日文名 yomitan zip `複合語起源テスト.zip`（标题 `テスト複合語辞典`）跑 `dictionary_importer::import` → `import OK`，磁盘真实生成日文标题目录 `テスト複合語辞典`，再 `DictionaryQuery::add_term_dict` + `query("日本")` → 命中 `日本/にほん` 1 条 glossary → `PASS`（覆盖打开/写出日文标题目录/回读查词全链路）。配合独立 `CreateFileA` 失败 / `CreateFileW` 成功复现，红→绿闭环成立。② **源码扫描守卫**（防回归，CI 可跑）`hibiki/test/dictionary/native_utf8_path_guard_test.dart`：断言 `fs_utf8.hpp` 存在含 `fs_path`/`fs_to_utf8`、`memory.cpp` 含 `CreateFileW`+`MultiByteToWideChar` 且**不含** `CreateFileA`/`CreateFileMappingA`、importer/query/stardict 均 include `fs_utf8.hpp` 且**不含** `glz::write_file_json`/`read_file_json`。真红→绿验证过（修复前 memory.cpp 含 CreateFileA → 红）。`flutter test` 全量 **2204 绿**（guard 测试在内）。
