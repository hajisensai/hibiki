## BUG-067 · 桌面端（Windows）拖动重排必须长按左键等 ~500ms 才生效（词典 / 音频源 / 字体 / 同步 URL 列表），鼠标按住即拖不响应
- **报告**：2026-06-06（用户：「win 端要左键长按等一会才能拖动，包括词典等进行排序，修复一下，电脑只需要按住左键就应该能拖动」）。采番注：动手前查 worktree 基线最大号 BUG-065，原取 066；rebase 回 develop 时发现 066 已被并发分支占用（「一次性导入大量词典崩溃」），故改取 067。
- **真实性**：✅ **真 bug（所有重排入口对全平台一律强制「长按起拖」，桌面鼠标也得等 `kLongPressTimeout`≈500ms）**。沿真实代码路径定位到两类拖拽入口：① **自实现 `HibikiReorderableColumn`**（`hibiki/lib/src/utils/components/hibiki_reorderable_column.dart`，为祖先 `HibikiAppUiScale` 缩放下零偏移而手搓）`_buildSlot` 用 `GestureDetector.onLongPressStart/MoveUpdate/End`——只认长按，**与平台/输入设备无关**。词典查词页排序（`dictionary_dialog_page.dart:1067`）、词典设置排序（`dictionary_settings_dialog_page.dart:158`）、本地音频源排序（`local_audio_sources_dialog.dart:139`）全走它，故「包括词典」。② **SDK `ReorderableListView` 整行手柄**两处直接用 `ReorderableDelayedDragStartListener`（`custom_fonts_page.dart:1161` 字体源排序、`sync_settings_schema.dart:1813` 同步 URL 排序），同样对全平台强制延迟（长按）起拖。讽刺的是 SDK 自带的 `buildDefaultDragHandles` 本就按平台分流（桌面即时、移动长按），这两处手动钉死成 Delayed 反而退化了桌面体验。根因共性：**「起拖时机」应由输入设备（鼠标=即时 / 触摸=长按）决定，却被硬编码成所有平台都长按**。
- **[x] ① 已修复** — 提交 `c4ae8f8e5`。根因修（消除「所有平台都长按」这个特例，让输入设备/平台驱动起拖时机，非加延迟/特例分支）：① `HibikiReorderableColumn._buildSlot` 把 `GestureDetector.onLongPress*` 换成 `RawGestureDetector` + 两个**按设备过滤**的 MultiDrag 识别器：`ImmediateMultiDragGestureRecognizer(supportedDevices: {mouse,trackpad,stylus,invertedStylus,unknown})`（精确指针按下移动即拖）+ `DelayedMultiDragGestureRecognizer(supportedDevices: {touch})`（触摸屏保留长按）；新增 `_ReorderDrag extends Drag` 把识别器的 update/end/cancel 桥接回原有全局坐标拖拽逻辑（`_startDrag/_updateDrag/_endDrag` 不变，`globalToLocal` 消缩放的核心机制原样保留）。**按设备而非平台**：Windows 触摸屏用户在触屏上仍是长按起拖、鼠标即时，更正确。② 新增 `HibikiReorderDragListener`（`hibiki/lib/src/utils/components/hibiki_reorder_drag_listener.dart`，导出进 `utils.dart` barrel）：按 `Theme.of(context).platform` 选 `ReorderableDragStartListener`（桌面，即时）/ `ReorderableDelayedDragStartListener`（移动，长按），**逐字镜像 SDK `buildDefaultDragHandles` 的平台分流**；替换 `custom_fonts_page` 与 `sync_settings_schema` 两处硬编码 Delayed。（SDK 列表沿用 SDK 的「按平台」范式而非「按设备」，因为它非缩放场景、且 SDK 自身就是按平台。）`flutter analyze` 0 issue。
- **[x] ② 已加自动化测试** — ① `hibiki/test/widgets/hibiki_reorderable_column_test.dart` 新增「鼠标按下立即拖拽（不长按、不等待）即重排」用例（`startGesture(kind: PointerDeviceKind.mouse)` 后立即 move，断言 onReorder 触发）——这正是旧 `onLongPress` 做不到的；原有 3 例（触摸长按重排 / 0.5 缩放下重排 / 触摸快速滑动**不**重排）全部保留并仍绿，证明触摸长按语义未回归。② 新增 `hibiki/test/widgets/hibiki_reorder_drag_listener_test.dart`：6 例守卫 `HibikiReorderDragListener` 在 windows/linux/macOS → 即时 `ReorderableDragStartListener`、android/iOS/fuchsia → 延迟 `ReorderableDelayedDragStartListener`（`buildDefaultDragHandles:false` 排除 SDK 默认手柄干扰，`find.byType` 精确匹配 runtimeType 不误命中子类）。`test/widgets/` 全量 216 例绿；`test/pages/` 音频源对话框 18 例（含「触摸长按在 0.5 缩放下重排」「uses HibikiReorderableColumn」）无回归。
- **备注**：桌面交互/手势/重排类。**真机肉眼复测原始失败路径待用户**（Windows + 鼠标）：词典查词页/词典设置/本地音频源/字体源/同步 URL 列表，鼠标**按住左键即可立即拖动重排**、不再需要长按等待；移动端（触摸）仍是长按起拖。
- **审查修正（opus，W-1/I-2）**：提交 `84cb3aed1`。opus 审 🟢 0 Critical。采纳 W-1：`_ReorderDrag.cancel` 原复用 `_endDrag` 会在「拖拽已重排但手势被竞技场夺走/路由销毁/dispose」时误提交一次 `onReorder`、且 `_endDrag` 的 `setState` 有 dispose 期隐患——改为走独立 `_cancelDrag()`（复位到原序、**不提交** onReorder，并守 `mounted`）。采纳 I-2：补「鼠标点击行内交互子控件（按钮/开关）应触发子控件 onTap 而非重排」守卫测试，钉死「即时拖拽识别器仅在越过 slop 后接管、不抢纯 tap」这一最易回归的竞技场契约。I-1/I-3（trackpad 单指拖序列 vs 两指滚动、unknown 归即时仍受 slop 门控）已补注释。`test/widgets/hibiki_reorderable_column_test.dart` 5 例全绿。
