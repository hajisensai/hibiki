## BUG-052 · 词典弹窗里词典名（折叠条 `▼` 标签）显示成一堆乱码 `▼������`（native `add_dict` 对 glaze 零拷贝 `string_view` 的 use-after-free）
- **报告**：2026-06-05（用户截图：词典管理页搜 `ちぐはぐ`，每条结果顶部折叠条 `▼` 后的词典名全是 U+FFFD 方块问号 `�`，正文日文/英文释义正常，要求「根本性修复这种问题」）。采番注：动手前查 `docs/BUGS.md` 当前最大号为 BUG-051（被并发 agent 占用 = 嵌套弹窗横滑），故本条取 052。
- **真实性**：✅ **真 bug（native use-after-free 破坏词典名首字节，渲染成 U+FFFD）**。`▼` 折叠条 = `popup.js createGlossarySection` 的 `<details>` summary，`textContent = g.dictionary`（弹窗 JSON 的词典名）。逐层验真：① 磁盘 `blobs.bin`/`index.json` 标题全干净（`EF BF BD` 计数 0）；② 用**部署中的** `hoshidicts_ffi.dll` 直接 ctypes 跑 `lookup_popup_json('ちぐはぐ')`，小规模 2 词典时 JSON 词典名干净——因 Python ctypes 在 SEH 边界吸收了 C++ 异常，掩盖了真因；③ 用**当前源码**重新编译 DLL 全量加载 30 词典再跑，**复现**：词典名首 6~8 字节变垃圾（`JMdict`→`\xd7t\xdf\x01\x00\x00`、`明鏡(6 byte)`→垃圾+`国語辞典 第三版`），且 **2 词典坏 / 7 词典好 / 30 词典坏**——堆布局相关的非确定性，垃圾值形似堆 free-list 指针的小端低字节 = **释放块被回收后写入指针**。**根因 `native/hoshidicts/hoshidicts_src/query.cpp:103`（旧）**：`Index::title`（`yomitan_parser.hpp:9`）是 glaze 零拷贝解析出的 `std::string_view`，只在源缓冲存活期间有效；`add_dict` 把读取 `index_buf` 的 `{...}` 块**先关闭**（`index_buf` 析构释放），**之后**才 `dict.name = ... index.title`——对已释放缓冲的 string_view 拷贝 = use-after-free。分配器在释放块首写入 free-list 指针 → `dict.name` = `[指针垃圾前缀] + [标题尾部]`，首字节非法 UTF-8 → 渲染 U+FFFD。正文释义来自完好的 mmap `blobs`，故「只有名字烂、内容好」。Dart FFI 不像 Python 那样吞 C++ 异常，且其堆布局触发该 UAF，故用户端必现。（旁证：用户机另有非 ASCII 标题词典在旧 DLL 上 `add_dict` 抛 C++ 异常，留下了 `hibiki.exe` 崩溃转储——同一 string_view/路径脆弱面。）`import_yomitan`（`importer.cpp:865-895`）把 `index_content` 缓冲一直存活到所有 `index.title` 使用之后，故导入侧无此 bug、磁盘标题干净。
- **[x] ① 已修复** — `06d908169`：根因修——尊重 `Index` 零拷贝契约，在 `index_buf` 存活期内就把 `index.title` 拷进自有的 `dict.name`。`query.cpp::add_dict` 把 `Index index;` 与 `dict.name = index.title.empty() ? stem : std::string(index.title)` 一并移进 `index_buf` 的 `{...}` 块内（缓冲析构前完成拷贝），并加注释说明该字段是 glaze 视图、禁止越出缓冲作用域。非补丁、非吞错。**验证（重编译 DLL）**：修前全量 30 词典加载词典名首字节烂、`FFFD>0` 且随词典数非确定；修后全量加载 0 抛异常、`FFFD=0`，full-dict×3 连续干净、2 词典/7 词典均干净。
- **[x] ② 已加自动化测试** — `06d908169`：双层守卫。① **源码扫描守卫（CI 跑）** `hibiki/test/dictionary/dict_name_lifetime_guard_test.dart`：解析 `query.cpp` 的 `add_dict`，对每个 `index.title` 用法计算自 `index_buf` 声明处起的相对花括号深度，**深度 <0（已越出缓冲块）即判 UAF 失败**，并要求出现自有拷贝 `std::string(index.title)`。已实测：修后绿；对旧（bug）代码片段跑同一逻辑两处 `index.title` 深度 = -1 → **红**（red→green 守护）。② **native 端到端守卫** `native/hoshidicts/tests/dict_name_lifetime_test.cpp`（沿用 `win_utf8_import_test` 的手动 fixture 约定，Windows 下用 `CommandLineToArgvW` 取 UTF-8 argv）：`add_term_dict(<dir>)` → `query(<term>)` → 断言每条 `glossary.dict_name == index.json 标题`。已用修后库对用户真实词典实测：`JMdict [2026-05-30]`/`明鏡国語辞典 第三版`/`wty-ja-ja` 全 **PASS**（名字与标题逐字节一致）。
