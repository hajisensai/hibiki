## BUG-342 · 导入字幕原生崩溃 0xc0000005 flutter_windows.dll AV
- **报告**：2026-06-20（用户：Windows 桌面端点「导入字幕」时崩溃，app 内错误日志无任何记录；Windows 事件查看器：错误应用 `hibiki.exe` 0.9.28.5304，错误模块 `flutter_windows.dll`，异常码 `0xc0000005`（访问违例），偏移 `0x00000000008e4f41`，进程崩溃无 Dart 堆栈）。TODO-603。
- **真实性**：✅ **真 bug（原生崩溃，Dart 错误区结构性捕不到）**。分两层定性：
  - **A 层（Dart 侧 libmpv use-after-free 隐患，已根因修复）**：导入字幕链路里选「内嵌图形字幕轨」（PGS/DVD 位图）走 `VideoPlayerController.selectEmbeddedGraphicTrack`（`hibiki/lib/src/media/video/video_player_controller.dart:467`）。该方法先 `await _waitUntilSubtitleTracksReady(player)`（最长 5s），其后还有 3 次连续原生下发：`await player.setSubtitleTrack(...)`+ 两次 `await applySubtitleMpvPropertiesToPlayer(player, ...)`。修复前**只在第一处 await 用 `if (_player != player)` 校验一次**，后续 3 处 await 后**从不重校验**，且用的是**调用时捕获的局部 `player` 引用**。
  - **load() 主路径（本轮核心·更高频·与用户崩溃更贴合）**：`VideoPlayerController.load()`（`:632`）开头捕获 `final Player player = _player ?? Player()`（`:710`）+ `final int loadToken = ++_loadToken`（`:653`）后，**8 个早期 await FFI 下发全裸用捕获 player、零重校验**：`player.open`（`:724`）/ `applyNetworkCachePropertiesToPlayer`（`:730`）/ `setSubtitleTrack(no)`（`:737`）/ `applySubtitleMpvPropertiesToPlayer(suppression)`（`:746`）/ `applyShadersToPlayer`（`:756`）/ `applyMpvConfigToPlayer`（`:761`）/ `setVolume`（`:767`）/ `setRate`（`:771`）。修复前**只有 `resolvedStartMs>0` 分支的 seek 前挂了一次 `_isCurrentLoad`（`:786`）**——从头播放 / 换集从头路径（`resolvedStartMs==0`）连这一次都没有。另外 autoPlay 的 `play()` 守卫用的是**单判据** `if (autoPlay && _player == player)`（`:825` 旧），换集复用同一 player 实例时单判据仍成立，会向已被新 load 接管的 player 误发 `play()`。
  - **根因机制**：这些 await 缺口内用户退出页面 / 换集触发 `dispose()`（`:1947`：`_loadToken++` → `unawaited(_player?.dispose())` 异步释放底层 libmpv `NativePlayer` → `_player=null`）。dispose 后局部 `player` 仍指向正被 / 已被释放的 native handle，继续 `open`/`setProperty`/`setSubtitleTrack`/`play` 即原生 use-after-free → 访问违例（0xc0000005）。这与代码库其它异步 mpv 路径（`_refreshChaptersForLoad` 等，`:836`）的不变量不一致——那些已在每个 await 后用 `_isCurrentLoad(player, loadToken)` 双判据（player identity + loadToken，定义见 `:851`）重校验，唯独 `load()` 主路径与图形选轨路径漏了。注意：此路径的 FFI 最可能崩在 `media_kit`/`libmpv` 模块，**不一定就是用户报告的 `flutter_windows.dll`**；但它是导入 / 播放链路上真实、可修、与现有不变量统一的缺陷，必修。
  - **B 层（用户报告的 `flutter_windows.dll` AV 真正崩点，需 Windows 符号调试器精确取证，不在本轮范围）**：`flutter_windows.dll`（Flutter 引擎本体）内的 `0xc0000005` 强烈指向**引擎层外部纹理投递**路径（`FlutterDesktopTextureRegistrar` → `MarkExternalTextureFrameAvailable`），而非字幕业务 Dart 逻辑。此签名与本仓库一个修了 16 轮、仍以「待真机收口」记录的 WGC 纹理桥 UAF **逐字吻合**（见 `BUG-163`/`BUG-197`/`BUG-209`，WebView2 常驻热槽查词纹理桥 + media_kit 视频纹理）。机制推断：`file_picker` 弹原生模态对话框时跑自己的消息循环、抢占 Flutter 主线程消息泵 → 对话框开/关迫使引擎窗口失活/重新合成 → 引擎向已注册 external texture 投递帧；若此刻某纹理源正处 teardown 中间态，一个迟到帧 `MarkTextureFrameAvailable()` 打进正被注销的 registrar → 崩在 `flutter_windows.dll`。即「点导入字幕」是**触发器/暴露器**，不是字幕业务逻辑本身崩。**仅凭偏移 `0x8e4f41` 无法定位到 `flutter_windows.dll` 的具体导出函数**——需配该 Flutter 版本的符号才能精确坐实，故 B 层标「需进一步取证」，不在本轮做无证据的原生改动。

- **[x] ① 已根因修复（A 层 + load() 主路径）** — `hibiki/lib/src/media/video/video_player_controller.dart`：
  - **A 层**：`selectEmbeddedGraphicTrack` 在第一条 await 前捕获 `final int loadToken = _loadToken;`，把原先唯一的 `if (_player != player)` 单次校验替换为在**每个 await 之后**都用 `_isCurrentLoad(player, loadToken)` 双判据重校验（共 3 处：等就绪后、`setSubtitleTrack` 后、第一次 `applySubtitleMpvPropertiesToPlayer` 后），过期立即 `return false`。
  - **load() 主路径**：在 8 个早期 await FFI 下发（`open` / `applyNetworkCachePropertiesToPlayer` / `setSubtitleTrack(no)` / `applySubtitleMpvPropertiesToPlayer(suppression)` / `applyShadersToPlayer` / `applyMpvConfigToPlayer` / `setVolume` / `setRate`）后各补一句 `if (!_isCurrentLoad(player, loadToken)) return;`，过期立即 `return` 干净放弃（return 点都在订阅/tick 挂起之前，不留半初始化）。从头播放 / 换集从头路径（`resolvedStartMs==0`）现也被覆盖。
  - **autoPlay 双判据**：`if (autoPlay && _player == player)` 升级为 `if (autoPlay && _isCurrentLoad(player, loadToken))`，换集复用同一 player 实例时不再误向被新 load 接管的 player 发 `play()`。
  - 全程复用代码库既有不变量（`_loadToken` / `_isCurrentLoad`），与 `_refreshChaptersForLoad` 等异步 mpv 路径范式统一，非新增特例分支、非吞异常、非加延迟。提交见本 worktree 分支 `worktree-agent-aaa45de8a82b04c81`（commit 哈希填于下）。**B 层未修**：用户报告的 `flutter_windows.dll` 纹理桥 AV 不是纯 Dart 可根治，需先取证（见备注「需进一步取证」），不硬编码绕过。提交：分支 `worktree-agent-aaa45de8a82b04c81` 单 commit（含本文件，哈希见 `git log`；本轮交付时为 `f0a797a73`）。

  - A 层 3 例：①`selectEmbeddedGraphicTrack` 第一条 await 前捕获 `loadToken`；②三处原生下发后都 `_isCurrentLoad(player, loadToken)` 重校验（计数 ≥3）且禁止回退到 `if (_player != player) return false;`；③结构契约——`setSubtitleTrack`/两次 `setProperty` 之间各插一次重校验。
  - load() 主路径 3 例：①`load()` 第一处原生下发前捕获 `loadToken`；②**8 个早期 await 下发后各自紧跟一次 `_isCurrentLoad(player, loadToken)` 重校验**（逐对断言「下发→重校验」落在下一处下发之前，杜绝整段共用一处；并断言 `if (!_isCurrentLoad(...)) return;` 干净放弃守卫 ≥8 处）；③autoPlay 的 `play()` 用 `_isCurrentLoad` 双判据，禁止回退到 `if (autoPlay && _player == player)`。
