## BUG-458 · 句子音频gap词残留
- **报告**：2026-06-29（用户：）
- **真实性**：✅ 真 bug — 根因 `hibiki/lib/src/media/audiobook/mining_audio_clip.dart:75`（修前）。书籍/有声书查词制卡时，DOM 抽句模式下 `data.sentenceNormalizedOffset/Length` 为空（`lookup.part.dart:239` / `chrome.part.dart:372` 把 `_cachedSentenceRange` 置 null），查词 token 又落在句子**外**的 plain-selector 边界 cue（标点/邻句 fragment，本地有声书无 sasayaki 位置）。此时 `_rangeFromSentencePosition` 的 span 兜底块旧守卫 `if (cue != null || textFallback.isEmpty) return null` 在 `cue != null` 时**直接放弃纯文本兜底**，回主函数走 cue-anchored 的 `_expandAroundCue`：它要求文本匹配覆盖锚 cue / 相邻 cue 文本是句子子串，锚 cue 在句外 → 锚定匹配被拒、扩展立即停 → 坍缩成单个边界 cue `_cueRange(cue)`，丢掉整句其余音频（残留：句子音频只剩边界那一个 cue 或错段）。
- **[x] ① 已修复** — `mining_audio_clip.dart:_rangeFromSentencePosition` span 兜底块改为：只要有句子文本就**无条件先试** `CollectionAudioMatcher.findPlaybackRange(text:)` 非锚定整句匹配（不再被 `cue != null` 挡死），命中返回完整区间；未命中才回 null 让主函数 fall through 到 cue==null 无线索 / cue!=null 的 `_expandAroundCue`。cue 可用 / span 可用的既有路径（956 已修对的）逐字不变；纯 EPUB 无 audioFiles 不进音频分支不受影响。提交：见本轮 commit 哈希。
- **[x] ② 已加自动化测试** — `hibiki/test/media/audiobook/mining_audio_clip_test.dart` 新增 `recovers the full sentence via text when span is empty and the cue is outside the sentence`：span 空（offset/length=null）+ cue 为句外 plain-selector 边界 cue + 句子跨三 cue，断言返回完整句子区间（startMs=1000/endMs=4300）而非边界 cue（startMs=100）。红→绿实证：修前 `Actual: <100>`（坍缩成边界 cue）失败，修后通过；原 11 个用例全保持绿（含「无线索回 null」「section 无匹配 cue 回 null」两个负向用例）。
