## BUG-483 · 导入有声书封面竞态被吞 (m4b 内嵌封面异步抽取未 await)
- **报告**：2026-06-30（用户：）· TODO-1034
- **真实性**：✅ 真 bug — 导入时刻竞态。根因 `hibiki/lib/src/media/audiobook/book_import_dialog.dart`：内嵌封面抽取 `_tryExtractAudioCover()`（原 :503）在 `initState` postFrame（:117）与拖放（:206）触发点是 **fire-and-forget**，抽取经 `desktop_audio_clipper.dart` 的 `extractEmbeddedCover`（spawn ffmpeg probe m4b 内嵌封面，数百 ms~数秒延迟）才在 :516 `setState(_audioCoverPath)`；而导入路径 `_applyBestCoverToEpub`（原 :825，`else if (_audioCoverPath != null && !_epubHasCover)`）同步读 `_audioCoverPath`。用户在 ffmpeg 未返回时点「导入」→ `_audioCoverPath` 仍 null → 封面被吞，写进 `EpubBooks.coverPath` 持久列的是 NULL（书架经 `reader_hibiki_source.dart:289/311` 渲染该列）。重启后 m4b 已被磁盘缓存预热，probe 瞬回，封面在点导入前写好 → 「重启重导入又恢复」，证明是导入时刻竞态而非渲染未刷新。受影响场景=epub 自身无封面、靠 m4b 内嵌封面兜底。
- **[x] ① 已修复** — `book_import_dialog.dart`：新增字段 `Future<void>? _coverExtraction`；`_tryExtractAudioCover()` 把在途抽取 Future 存进该字段（内部 `_runCoverExtraction()` 实际干活），各触发点（postFrame / 拖放 / pickAudio）经此统一赋值；`_applyBestCoverToEpub` 开头 `if (extraction != null) await extraction;` 再读 `_audioCoverPath`——抽取已完成则零等待，未完成正好补真实耗时，零回归、零 delay。提交：3efaee9de（本文件随该提交，哈希指 commit 自身）
