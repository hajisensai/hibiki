## BUG-212 · Android 悬浮字幕条查词退化：点哪都查句首+弹键盘
- **报告**：2026-06-12（用户：）TODO-166（拆自 TODO-160 子 c）
- **真实性**：✅ 真 bug。根因 `hibiki/android/app/src/main/java/app/hibiki/reader/FloatingLyricService.java:295-305`（修复前指向 `new Intent(this, PopupDictActivity.class)` + `putExtra("charIndex", index)`）。0.5.0 的 commit `0248a0260`（refactor(android): rewrite PopupDict from Java to Kotlin）把弹窗词典从 Java `PopupDictActivity` 重写为 Flutter 弹窗 `PopupDictFlutterActivity`，并把系统 PROCESS_TEXT / SEND 的 intent-filter 迁到新 Activity（`AndroidManifest.xml:98-101` 把旧 `PopupDictActivity` 标注「失活：暂留作回退」），但 `FloatingLyricService.handleTap` 这条悬浮字幕条/歌词点击查词的入口没跟着迁移，仍显式启动失活的旧 native WebView Activity。后果两重：①旧 native Activity 打开预填整句的 EditText 搜索框 → 弹软键盘（用户报「弹键盘」）；②旧 native 路径整套丢掉 `charIndex`，只把整句喂给 HoshiDicts，最长匹配恒从句首 offset 0 开始 → 「点哪都查句首」。系统文本选择菜单查词不受影响（它走已迁移的 intent-filter 进 `PopupDictFlutterActivity`），所以退化只发生在悬浮字幕条这一条入口。与 TODO-160 其它子问题（a 透明 / b 自动阅读 / d 开关放置）不同源。
- **[x] ① 已修复** — 切回 Flutter 弹窗并打通 charIndex 透传链（根因修复，非补丁）：`FloatingLyricService.java` 把 tap 改 `new Intent(this, PopupDictFlutterActivity.class)` 并 `putExtra(PopupDictFlutterActivity.EXTRA_CHAR_INDEX, index)`；`PopupDictFlutterActivity.kt` 新增 `companion const val EXTRA_CHAR_INDEX` + `extractCharIndex(intent, 默认 -1)`，onCreate（冷启）/onNewIntent（热复用）把真实 charIndex 透传给 `PopupEngineHolder.setPendingText` / `pushProcessText`；`PopupEngineHolder.kt` 新增 `pendingCharIndex` 字段，`getInitialProcessText`（冷启轮询）/ `onNewProcessText`（热复用推送）回传真实 charIndex（替换原硬编码 `-1`）。链路终点 `hibiki/lib/popup_main.dart:14-21` 的 `_extractWord`：`charIndex >= 0` 时 `wordFromIndex` 从该位置切出被点词；`charIndex < 0`（系统整句查词的默认 -1）直接 `return text` 走整句查词 → 系统菜单查词路径零回归。切回 Flutter 弹窗本就无强制 EditText 搜索框 → 自然消除「弹键盘」。提交：见下方 commit。
- **[x] ② 已加自动化测试** — 两层守卫：①源码扫描 `hibiki/test/build/android_popup_charindex_guard_test.dart`（钉死 wire 契约：`handleTap` 必须 route 到 `PopupDictFlutterActivity` 且不再引用 `PopupDictActivity` + 必带 `EXTRA_CHAR_INDEX`；Activity 必须从 intent 读 charIndex 并在冷启/热复用两路径转发；Holder 必须回传 `pendingCharIndex` 而非硬编码 `-1`）。②Dart 行为测试 `hibiki/test/pages/popup_floating_lyric_charindex_test.dart`（`JapaneseLanguage.wordFromIndex` 在无 C++ FFI 的降级模式下返回被点字符：不同 index 锚定不同字符，证明 index 是切词锚点——撤销修复回到忽略 charIndex 喂整句时每次都锚句首；负/越界 index 返回空，让调用方回退整句查词，对齐系统 PROCESS_TEXT 路径）。本机实测 5 例全绿。提交：见下方 commit。
