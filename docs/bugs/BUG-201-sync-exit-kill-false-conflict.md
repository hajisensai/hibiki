## BUG-201 · 退出书后杀后台重开总弹假冲突(baseline 与远端进度传输非原子)
- **报告**：2026-06-11（用户：TODO-132 诉求A）
- **真实性**：✅ 真 bug。根因 `hibiki/lib/src/sync/sync_manager.dart`：旧实现把进度 baseline 写在 `_handleImport`/`_handleexport` 返回**之后**（旧 `:356-359` import 分支、`:385-389` export 分支），而权威进度传输（export 的 `updateProgressFile` / import 的 `upsertReaderPosition`）在方法**内部**就已落地。两者之间还隔着 stats/audio/content 等若干远端网络调用。
- **复现路径**：退出书 `hibiki/lib/src/pages/base_source_page.dart:121` 的 `onWillPop` 触发 `triggerAutoSyncAfterClose`（fire-and-forget 单本双向同步，`sync_auto_trigger.dart:_runAutoSync`）。auto 路径走三方 gate：`resolveProgressSync(local, remote, base)`。export 把本地进度上传到远端后，进程在「远端进度已落地 but baseline 未写」之间被杀（杀后台），baseline 停在旧的共同祖先。下次开 app 全量 sweep 再走 auto gate，看到 local 与 remote 都偏离旧 baseline → `sync_progress_resolver.dart:46-47` 判 `isConflict` → 假「本地远端冲突」。「书里面直接杀」正常是因为还没触发退出 export，baseline/remote 一致。
- **[x] ① 已修复** — `hibiki/lib/src/sync/sync_manager.dart`：把 progress baseline 的写入下沉进 `_handleImport`（紧跟 `upsertReaderPosition` 后）与 `_handleExport`（紧跟 `updateProgressFile` 返回后），并把 `assetKey` 作为参数传入两个 handler；删掉 `_syncBookOnce` 内方法返回后的两处 baseline 写。这样一旦权威进度落地，新的共同祖先时间戳就在同一处、紧接着的本地 Drift 写里被记录（两次相邻本地写之间无 `await` ≈ 原子），不可恢复的杀进程窗口从「权威进度 + 所有尾部远端传输」收缩到「单条本地 baseline 写」。行为对旧 gate 等价：import 早返回（无远端进度）仍不写 baseline；export 仅在 `localPosition != null` 时写（等价旧 `localPosition?.updatedAt != null`），content-only 导出（无进度）照旧不写。提交：见 commit。
