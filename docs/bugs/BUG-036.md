## BUG-036 · 手机端谷歌云盘「立刻同步」开有声书文件/词典同步时 OOM 闪退 + UI 卡死
- **报告**：2026-06-04（用户：手机端用谷歌云盘「立刻同步」，开了内容同步 + 有声书文件同步 + 词典同步，触发卡死/闪退）。采番注：计划暂定 BUG-036，动手前查 `docs/BUGS.md` 当前最大号为 BUG-035（035 已被并发 agent 占用 Hibiki 互联根目录死锁），故本条取 036。
- **真实性**：✅ **真 bug（整文件 + 整 zip 进内存 + 跑 UI isolate）**。沿真实代码路径逐行验实：`sync_asset_package_service.dart` 导出时把整个资源文件 `await file.readAsBytes()` 读进内存（音频 :99 / 词典经 `_addDirectoryFiles` :295）再 `ZipEncoder().encode(archive)!` 把整个 zip 在内存里编码（`_writeZip` :272）；导入时 `ZipDecoder().decodeBytes(await packageFile.readAsBytes())` 把整包字节读进内存（词典 :45 / 有声书 :127）。词典/有声书包数百 MB → Android 低堆 native OOM 闪退；且整条编解码链跑在 UI isolate（无 `compute`/`Isolate`），deflate/inflate 的 CPU 同步阻塞 event loop → UI 卡死。根因点 = `sync_asset_package_service.dart:99/272/45/127/295`（整文件 + 整 zip 进内存，跑 UI isolate）。
- **[x] ① 已修复** — `c4f9270cd`（导出流式化）+ `25252a524`（导入流式化、删内存路径）+ 审查修正轮（措辞订正 + 音频 STORE 流式 + 导入真流式 + 异常清理 + 大文件测试）：根因修，整条 zip 编解码链放进 `Isolate.run`（离开 UI isolate）、不再整 zip 入内存，包格式 100% 不变（向后兼容）。**导出**：主 isolate 只先读 DB 建 manifest + 收集「zip 内路径 → 磁盘路径」清单（不读文件内容），再把纯文件 zip 交 `_zipPackageInIsolate`（`ZipFileEncoder.create/addArchiveFile(小 manifest)/addFile/closeSync`，外裹 try/catch：中途失败 `closeSync` 释放句柄 + 删半截包后 `rethrow`，补回旧 `writeAsBytes(flush:true)` 的原子性，不留半截 zip 被当成有效包）。**按包类型选压缩策略**：音频包 `storeResources:true` 用 `ZipFileEncoder.STORE`——archive 3.6.1 的 STORE 分支对 `ArchiveFile.stream` 真流式（`getFileCrc32` 分块 + `writeInputStream` 转发，不 `toUint8List`），**单个几百 MB 的音频文件不会整入内存**，且音频/图片本是压缩格式、deflate 纯浪费 CPU；词典包 `storeResources:false` 用 deflate（JSON 可压缩省上传体积，词典资源文件较小，单文件入内存可接受——这是 archive 3.6.1 deflate 分支 `toUint8List` 整文件入内存的**已知残留限制**，如需进一步降内存可后续把词典也换 STORE）。**导入**：`_readManifestInIsolate`（`decodeBuffer(InputFileStream)` 只读中央目录 + 惰性流，仅取小 manifest 进内存）在主 isolate 解析后写 DB；资源解压交 `_extractResourcesInIsolate`，每条经 `_streamArchiveFileTo` 逐块落盘——**关键订正：不能用 `ArchiveFile.decompress(out)`**（`decodeBuffer` 把条目 `_content` 设为 `ZipFile`(FileContent) 非 null，`decompress` 的前置 `_content==null` 不成立 → no-op 写 0 字节，实测会回归往返测试），也不能用 `writeContent(out)`（先 `_content.content` 整文件解压进内存 = 导入侧 OOM 源）；真流式是直接消费条目原始压缩流 `rawContent`（从 `InputFileStream` 解出时是惰性窗口文件流）：STORE→`out.writeInputStream(raw)`、DEFLATE→`Inflate.stream(raw,out)`，均不整文件入内存。保留与旧 `_extractArchivePrefix` 等价的 zip-slip 路径安全校验。DB 读写全部留主 isolate（`Isolate.run` 闭包只捕获 String/Map<String,String>/bool，不捕获 `_db`/Flutter 对象）。删除被取代的私有件 `_writeZip`/`_jsonFile`/`_addDirectoryFiles`/`_extractArchivePrefix`/`_readManifest`，并移除因全部符号已由 `archive_io.dart` 提供而冗余的 `archive.dart` import。`dart analyze` 0 issue、`dart format` 0 改动。（备注：A1 准备提交 `8e3d5f5e1` 仅加 3 个 isolate 辅助函数。）
- **[x] ② 已加自动化测试** — `d7ab815e6`（源码守卫）+ 审查修正轮（大文件行为测试 + 守卫加流式断言）：① 行为级 `hibiki/test/sync/sync_asset_package_service_test.dart` 新增「large audio file survives STORE export + streaming import intact」——造 **3MB 伪随机音频资源**（覆盖 0..255 全字节值，小文件走不到分块/流式分支），走 `exportAudioDatabasePackage`→`importAudioDatabasePackage` STORE 往返，断言导入后磁盘文件**长度相等 + sha256 哈希完全一致**；这层行为测试证明 STORE 导出 + `rawContent` 流式导入对大文件字节级正确、格式兼容（比源码子串守卫可靠，也正是它揪出 `decompress(out)` no-op 写 0 字节的回归）。既有词典/有声书小文件往返测试（含 DEFLATE→`Inflate.stream` 导入路径）保持绿。② 源码守卫 `hibiki/test/sync/sync_asset_package_no_inmemory_guard_test.dart` 仍断言不含 `readAsBytes`/`ZipEncoder()`/`decodeBytes`、含 `Isolate.run`/`ZipFileEncoder`/`InputFileStream`，**新增**含 `Inflate.stream(`/`writeInputStream(`（导入两分支真流式）/`ZipFileEncoder.STORE`（音频 STORE）。`test/sync/` 全量 **266 绿**无回归。
