## BUG-083 · 同步进行中弹出/打开「本地 vs 远端」对比会打断同步、并卡加载甚至连接超时
- **报告**：2026-06-06（用户：「同步的时候点击对比数据会打断」「立刻同步的时候，同步一半弹出本地 vs 远端，还要加载，甚至能连接超时」；平台：Windows 桌面 + Google Drive）。
- **真实性**：✅ **真 bug（沿真实代码路径定位）**。云后端（GoogleDrive/Dropbox/OneDrive/WebDAV…）是**进程级单例**，持有可变共享状态（access token / cached api / `_rootFolderId` / `_titleToFolderId` folder 缓存）。`sync_auto_trigger.dart` 早已用一把全局 `AsyncMutex _autoSyncMutex` 串行**所有自动/手动同步**（见 `sync_auto_trigger.dart:24-33` 注释，正是为「避免并发改 singleton backend 状态」），但**对比/冲突对话框的网络操作完全绕过了这把锁**：
  - `SyncCompareDialog._load()`（`sync_compare_dialog.dart`，`initState` 触发）调 `_fetchCompareData` → `backend.listBooks` 重新列举远端 + `backend.cacheBookFolderIds(...)` / 写 `cachedRootFolderId`·`cachedFolderIds`（`_fetchCompareData` 末尾 `repo.setFolderCache`），**直接改写同一单例的 folder-id 缓存**；
  - `_applyChoices()` 经 `SyncManager` 做真实下载/上传/删除，也在同一单例上；
  - `showSyncCompareDialog` 的前置 `restoreAuth`+`isAuthenticated` 同样裸调。
  - 冲突解决弹窗（`SyncConflictPrompter.present`，由「立即同步」跑完发现冲突后、或后台自动同步 `onReport` 触发）**自动弹出** `SyncCompareDialog(conflictsOnly:true)`，其 `_load` 又是一次全量远端重列举——这正是用户看到的「同步一半弹出本地 vs 远端、还要加载」。当这次重列举与一个仍在飞的同步（后台自动同步 / 或用户手点对比时尚未结束的同步）**并发抢同一个 Google Drive 连接 + 改同一份 folder 缓存**时：搅乱在飞同步对 folder id 的视图 → 同步报错被打断；两路请求争用单连接 → 对比 load 变慢、甚至 `连接超时`。根因 `hibiki/lib/src/sync/sync_compare_dialog.dart`（`_load` / `_applyChoices` / `showSyncCompareDialog` 三处绕过 `_autoSyncMutex`）。
- **[x] ① 已修复** — 本提交（自含修复+测试+本条目；并发抢号致哈希随 rebase 变动故不内联记录）。**根因修（让对比加入既有锁，非补丁）**：`sync_auto_trigger.dart` 暴露公共 `runExclusiveWithSync<T>(body)`（包 `_autoSyncMutex.withLock`）；`sync_compare_dialog.dart` 三处触碰单例 backend 的网络操作全部进锁——① `_load` 的远端 `Future.wait([_fetchCompareData, _fetchDictEntries])`、② `_applyChoices` 的整段网络写、③ `showSyncCompareDialog` 的 `restoreAuth`+读 `isAuthenticated`。效果：同步在跑时打开/弹出对比，对比的网络阶段**等待**同步放锁后再跑（对话框期间显示 loading），而非并发争抢——既不打断同步，也不再因争用单连接而超时。`AsyncMutex` 非重入，进锁的 body 均不再调用任何会取此锁的同步入口（已核对 `SyncManager`/`importRemoteBookFolder`/`restoreAuth` 均不取 `_autoSyncMutex`），无死锁。
- **[x] ② 已加自动化测试** — `hibiki/test/sync/sync_compare_mutex_test.dart`（真 widget 行为）：持锁模拟一次在飞同步（`runExclusiveWithSync(() => gate.future)`）→ `pumpWidget(SyncCompareDialog)` → 断言持锁期间记录型 backend 的 `listBooks` **零调用**（不并发列举远端）→ 放锁后断言 `listBooks` 被调用（对比继续完成）。已实测红（去掉 `_load` 的 `runExclusiveWithSync` 包裹 → 持锁期间即列举，`Expected:<0>` 失败）→ 绿。同步既有 BUG-047 源码守卫（`google_drive_mobile_auth_test.dart`「compare dialog restores auth before gating」）随结构调整更新匹配模式（restore→read→gate 顺序不变）。`test/sync/` 全量 +303 绿。
- **备注**：sync/网络类修复。代码正确 + `test/sync/` 无回归；**Windows 桌面 + Google Drive 真机复测原始失败路径**（立即同步进行中弹出/打开对比 → 不打断、不超时）**待用户**。本条目针对的是「对比与同步并发争用」的根因；冲突弹窗每次都全量重列举远端（即使无并发也偏慢）属独立性能优化，可后续让冲突弹窗复用同步已取的数据，不在本次范围。
