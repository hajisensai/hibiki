## BUG-035 · Hibiki 互联：主机可达却报「No reachable Hibiki server address」（新开启服务器从未创建 WebDAV 根目录）
- **报告**：2026-06-04（用户：「No reacable Hibiki server address」。澄清确认：场景 A —— **主机服务器确认开着且可达**（浏览器/ping 能连），客户端仍报不可达 = 真 bug）。
- **真实性**：✅ **真 bug（鸡生蛋死锁，根因确定性）**。沿真实代码路径定位：`HibikiSyncServer` 把 `<databaseDirectory>/sync-data` 当 WebDAV 根对外服务（构造器 `syncDataDir = p.join(syncDataDir, 'sync-data')`，`hibiki_sync_server.dart:73`），但 `start()`（`:104`）**从不创建该目录**，只在 `_handlePut`/`_handleMkcol` 懒创建。客户端 `HibikiClientSyncBackend` 的所有网络操作都先过 `_ensureResolved`→`resolveReachableHibikiUrl`→`_defaultHibikiProbe`（`hibiki_client_sync_backend.dart:21/39`）这道**可达性探测**闸门，而探测做的是 `WebDavOps.testConnection`（`webdav_ops.dart:60`）= 对根发 `PROPFIND`。服务端 `_handleRequest` 把根映射到 `syncDataDir`，`_handlePropfind`（`hibiki_sync_server.dart:381`）见目录不存在 → `FileSystemEntity.typeSync==notFound` → 返回 **404**；`testConnection` 见 `statusCode>=400` 抛 `SyncBackendError('Server returned 404')`（非 `SyncAuthError`），被 `_defaultHibikiProbe` 的 `catch(e)` 吞成 `return false`（判不可达）→ 所有候选耗尽 → 抛 `SyncBackendError('No reachable Hibiki server address')`。而唯一能创建 `sync-data` 的 MKCOL（`findOrCreateRootFolder`）又被同一探测闸门挡在后面 → **死锁：探测要根目录存在，只有写操作能创建它，写操作又过不了探测**，新开启的主机对客户端永远「不可达」。**冒烟枪**：既有 `test/sync/hibiki_p2p_roundtrip_test.dart:22-24` 手动预建 `sync-data` 并注释「it must exist so the client's connection probe (PROPFIND on root) succeeds」——测试把生产缺失的前置条件自己补上了，于是 CI 绿、生产坏。根因点 = `HibikiSyncServer.start()` 不创建作为 WebDAV 根的 `syncDataDir`。
- **[x] ① 已修复** — `4abdd31a2`：Linus 式消除特殊情况——WebDAV 服务的根集合必须恒存在（不变式）。`start()` 在 `shelf_io.serve` 之前 `await Directory(syncDataDir).create(recursive: true)`（`create(recursive:true)` 对已存在目录是幂等 no-op，重启安全；父目录 `databaseDirectory` 必存在因 DB 落在那里）。这样新开启的服务器对根 `PROPFIND` 恒返回 207（空集合）而非 404，探测有意义、客户端能 bootstrap；"404-on-empty-root" 这个特殊情况从源头消失。**不动**探测/客户端逻辑（根因在服务端缺失的根目录，不是探测把 404 误判——探测把「服务器返回错误」当不可达本身是合理的；真正的洞是「服务器不该对自己的根返回 404」）。
- **[x] ② 已加自动化测试** — `4abdd31a2` · `hibiki/test/sync/hibiki_server_root_bootstrap_test.dart`：端到端复现——起一个真 `HibikiSyncServer` 但**故意不预建 `sync-data`**（区别于 p2p roundtrip 的预建，正是预建掩盖了此 bug），断言 ① `start()` 后 `sync-data` 目录存在（服务端不变式，修复前此处即红）② 真 `HibikiClientSyncBackend` 经 `restoreAuth`+`ensureResolved` 能解析到该服务器（`activeBaseUrl` == 服务器 URL，探测不再 404）。修复前红（实测：invariant 断言 `Expected true Actual false`）、修复后绿。`test/sync/` 全量 **264 绿**无回归（含既有 p2p roundtrip）。
- **备注**：同步/Hibiki 互联/WebDAV 服务端类。代码 + 单测绿、`flutter analyze` 0 issue、`dart format` 0 改动。**真机复测原始失败路径待用户**：设备 A 开「Hibiki 服务器/允许局域网」（全新、从未被同步过）→ 设备 B 配对后触发同步 → 应能连上并完成首次同步（修复前必报「No reachable Hibiki server address」）。本修复让首个客户端无需任何前置写入即可 bootstrap；对已有过同步（`sync-data` 已存在）的主机无行为改变。
