## BUG-236 · 安卓大屏在设置 tab 按返回键直接退出 app（应切回来源 tab）
- **报告**：2026-06-13（TODO-285，用户：安卓大屏在设置界面按返回键会直接关掉 app）
- **真实性**：真 bug。设置是 home 的一个内容 tab（无独立路由层级）。`hibiki/lib/src/pages/implementations/home_page.dart` 的 `_buildDesktopLayout` settings 分支与 `buildBody` 的 `HomeTab.settings` 分支都只渲染 `HibikiSettingsContent`，没有任何就近的 `PopScope` 拦截系统返回键；宽屏设置内部用 `MaterialSupportingPaneLayout` 且 `pushRoutes:false`（master-detail 不 push 路由），返回也只靠屏幕左上箭头。于是硬件/手势返回冒泡到 home 顶层 `PopScope`（`home_page.dart:346` `canPop: !syncing`）→ 非同步时 `canPop:true` → pop 掉 home 路由 = 退出 app。窄屏移动布局（`_buildMobileLayout` → `buildBody`）走同一个 settings 分支，因此同样会被退出。
- **根因**：设置 tab 内容外壳缺少自己的 `PopScope`，系统返回事件没有就近消费者，直接命中顶层 `PopScope(canPop:!syncing)`（`home_page.dart:345-361`）退出 app。`home_page.dart` 旧实现：`_buildDesktopLayout` settings 分支 `HibikiSettingsContent(onBack: () => _selectTab(_previousTab))`（仅左上箭头可返回），`buildBody` settings 分支 `const HibikiSettingsContent()`（连箭头都没有）。
- **[x] ① 已修复**：抽出共用外壳 `HomeSettingsTabContent`（`home_page.dart:583`），用 `PopScope(canPop: false, onPopInvokedWithResult: ...)`（`home_page.dart:603`）拦截系统返回键，消费后调用 `onReturnToPreviousTab`（= `_selectTab(_previousTab)`，复用现有返回逻辑）切回进入设置前的来源 tab，而非冒泡退出。home 两处渲染路径（`_buildDesktopLayout` settings 分支 `home_page.dart:464` `showBackButton:true` 显示页头箭头；`buildBody` settings 分支 `home_page.dart:559` `showBackButton:false` 走侧栏/底栏切回）都改用 `_buildSettingsTabContent`，同时覆盖大屏与窄屏移动端。内层设置 PopScope 离焦点更近，先于顶层同步态 PopScope 收到事件，不破坏顶层同步告警/退出逻辑；非设置 tab 不构造此外壳，退出行为不变。用 Flutter 3.44 的 `PopScope`（非废弃的 `WillPopScope`），新增 helper 带完整类型签名。
- **[x] ② 已加自动化测试**：`hibiki/test/pages/home_settings_back_intercept_test.dart`（widget 行为）——大屏约束（1024×768）下构造 `HomeSettingsTabContent` 注入轻量占位 child，`tester.binding.handlePopRoute()` 模拟系统返回，断言事件被消费（`handled==true`，即不退出 app）且回调 `onReturnToPreviousTab` 被调用（切回来源 tab）；另断言 `PopScope.canPop==false`；并覆盖 `showBackButton:false`（移动底栏/宽屏侧栏在侧）路径同样拦截。3 例全绿。
- **备注**：构造完整 `HomePage` 需要 AppModel/Drift DB/WebView 预热等重依赖，故把 PopScope 拦截逻辑下沉为可独立测试的 `HomeSettingsTabContent`（生产路径始终用默认 `HibikiSettingsContent`，测试用 `child` 注入占位），既是根因修复也是可测设计。残留风险：仅单测层用 `handlePopRoute()` 验证拦截，未在真安卓大屏设备复测原始失败路径，待真机回归。
