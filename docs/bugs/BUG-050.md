## BUG-050 · Windows 批量导入词典中途报 `PathAccessException: Rename failed … (OS Error: 拒绝访问。, errno = 5)`（AV/索引器瞬时锁住刚写盘的词典目录，发布 rename 失败）
- **报告**：2026-06-05（用户截图：导入中 4/14，`PathAccessException: Rename failed, path = 'C:\Users\wrds\Documents\dictionaryResources\import_temp\故事ことわざの辞典' (OS Error: 拒绝访问。, errno = 5)`，要求「根本性修复拒绝访问」）。采番注：动手前查 `docs/BUGS.md` 当前最大号为 BUG-049，故本条取 050。
- **真实性**：✅ **真 bug（导入发布阶段的目录 rename 在 Windows 上被外部句柄瞬时占用，Windows 专有、间歇）**。沿真实代码路径定位：批量导入 `importFromDirectory` → 对每个 zip 调 `importFromFile`（`dictionary_import_manager.dart`）：① 清空并重建暂存 `import_temp` → ② `importDictionaryViaHoshidicts` 把词典写进 `import_temp/<标题>`（native `import_` 经 `hoshidicts.dart:335` 的 `Isolate.run` **同步**执行，返回即用 RAII/CloseHandle 关掉自身文件句柄）→ ③ `innerDataDir.renameSync(finalDir.path)`（旧 `:296`）把 `import_temp/故事ことわざの辞典` move 到 `dictionaryResources/故事ことわざの辞典`。Dart `Directory.renameSync` 在 Windows 走 `MoveFileExW`，要求源目录**及其内容**无任何打开句柄；报错 `path` 报的是**源**目录，说明 move 时源里某文件仍被占用。**根因 = 外部不可控平台行为**：Windows Defender 实时保护 / 搜索索引器在文件刚落盘后**异步**打开扫描，短暂(几十~几百 ms)持有句柄，撞上紧接着的 rename → ERROR_ACCESS_DENIED(5)（或 SHARING_VIOLATION(32)）。**判据**：若是本进程同进程句柄泄漏（可控根因），同一进程内句柄永不自动关闭 → 每次都会**确定性**在 rename 失败；而用户是 **4/14 间歇**失败（越大的词典文件越多、扫描窗口越长、越易撞），正是外部瞬时锁的签名 → 排除 native 句柄泄漏。POSIX(Android/macOS/Linux) 允许 rename 打开中的目录，故仅 Windows 中招。根因点 = `dictionary_import_manager.dart` 的发布 rename 直接用 `renameSync`，对 Windows AV/索引器瞬时锁零容忍。
- **[x] ① 已修复** — `a786846df`：根因层修——这是无法从 app 侧消除的外部平台限制（不能阻止 Defender/索引器扫描刚写入的文件），按 CLAUDE.md「外部系统/不可控平台限制才允许临时兼容层」开有界重试 + 复制回退，并说明清理条件。① 新增 `@visibleForTesting static publishImportedDir({rename, copyThenDelete, sleep, isWindows, maxAttempts})` 纯逻辑：**仅当 `isWindows` 且 OS 错误码为瞬时锁(5/32)** 时按 50/100/…ms 退避有界重试（默认 10 次≈2.25s），用尽后回退 `copyThenDelete`；其余错误（非 Windows、或非瞬时码如 13）**原样抛出**（不掩盖真错）。② 私有 `_publishImportedDir(source, destPath)` 注入真实 FS 实现：rename = `source.rename(destPath)`（异步，不冻 UI）；回退 = 新增异步 `_copyDirectoryAsync`（copy 用共享读打开源，绕过 `MoveFile` 的句柄限制）再 `source.delete`（删源失败的残留 temp 由下一轮 import 的 `import_temp` 重建覆盖清理）。③ `importFromFile` 与 `importFromDirectory` 两处发布点（旧 `renameSync` 的 inner/temp 分支共 4 处）统一改 `await _publishImportedDir(...)`。重试同时覆盖「replace 场景下 finalDir 删除处于 delete-pending 致 rename 目标仍存在」的同源瞬时态。`flutter analyze`（改动文件）0 issue、`dart format` 通过。
- **[x] ② 已加自动化测试** — `a786846df` · `hibiki/test/models/dictionary_import_publish_test.dart`（行为守卫，最强可落地层——发布逻辑已抽成依赖注入的可测纯函数，绕开真实 AV 时序）：6 例覆盖 happy path（rename 1 次成功、0 重试 0 复制）、Windows 瞬时 errno 5 两次后成功（rename 3 次、退避 2 次、不复制）、Windows 持续 errno 5 用尽 maxAttempts 后回退复制+删源（`DictPublishMethod.copied`）、errno 32 同样被当瞬时重试、**非 Windows errno 5 直接抛出**（POSIX 不该走重试）、**Windows 非瞬时 errno 13 直接抛出**（不掩盖真错）。修复前 `publishImportedDir` 不存在 → 测试不编译即红；修后 `flutter test test/models/ test/dictionary/` **262 绿**无回归。
- **备注**：词典/导入/Windows/文件系统类。代码 + 单测绿、`flutter analyze` 0 issue。说明：这是对**不可控外部平台限制**的兼容层而非补丁掩盖——根因（Defender/索引器异步扫描持句柄）在 app 侧无法根除，故在发布 rename 处做有界重试+复制回退是正确的最小处理；非瞬时/非 Windows 错误仍原样上抛不被吞。**真机肉眼复测原始失败路径待用户**（Windows）：一次性批量导入十几本词典（含日文标题如「故事ことわざの辞典」），应全部导入成功、不再中途报 `拒绝访问 errno=5`；若个别因 AV 长占用走了复制回退，词典内容仍完整可查。
