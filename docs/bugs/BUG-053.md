## BUG-053 · 导入本地音频后「没了」/不生效（「管理音频来源」对话框导入只入内存，仅底部「关闭」按钮才落盘，点遮罩/返回退出即丢失）
- **报告**：2026-06-05（用户：「现在本地音频真的生效了吗，我导入以后怎么没了」）。采番注：动手前查 `docs/BUGS.md` 当前最大号为 BUG-052（被并发 agent 占用 = 词典名乱码），故本条取 053。
- **真实性**：✅ **真 bug（导入持久化时机错位：拷文件 + 弹「成功」但不落盘，靠一个可被轻易绕过的关闭按钮才保存）**。沿真实代码路径定位：设置「管理音频来源」入口（`settings_schema.dart:804-849`，`'lookup.audio_sources'`）经 `showSettingsDialog`→`showAppDialog`（`show_app_dialog.dart:9` 默认 `barrierDismissible: true`）弹出 `AudioSourcesDialog`。点「添加本地音频数据库」→ `_addLocalDb`（`dictionary_settings_dialog_page.dart:312-327`）→ `onPickLocalDb`（`settings_schema.dart:817-832`）→ `appModel.importLocalAudioDbFile` → `LocalAudioManager.importFile`（`local_audio_manager.dart:132-149`）：**只把外部文件拷进库目录、返回一条 `enabled:true` 的 entry，但明确不写 prefs、不通知 native**（注释自陈「持久化交给 setEntries/setAudioSourceConfigs」）。`_addLocalDb` 把它插进对话框**内存** `_sources` 并立刻弹「导入成功」toast（`:318-319`）——此刻**尚未落盘**。真正持久化（`onSave`=`appModel.setAudioSourceConfigs`，`app_model.dart:2606-2637`：写 `audio_source_configs` + `local_audio_dbs` + 推 native）**只发生在点底部「关闭」按钮时**（旧 `dictionary_settings_dialog_page.dart:136-138`：`onSave(_sources)` 然后 `pop`），`dispose()` 不保存。于是用户若用**点遮罩 / 系统返回 / Esc**（对 `barrierDismissible:true` 对话框全是合法关闭路径）退出 → `onSave` 从不被调用 → 导入丢失。雪上加霜：拷进库目录的 `local_audio_*.db` 副本成了「拷了但没持久化」的孤儿，下次任意一次 `setAudioSourceConfigs` 跑 `pruneOrphans`（`app_model.dart:2634` → `local_audio_manager.dart:154-167`，正则 `^local_audio_\d+\.db$`）会把它物理删除，连恢复都没了。该对话框无「取消」概念（只有「重置」+「关闭」），用户心智=改了就生效——故这是真 bug。根因点 = `dictionary_settings_dialog_page.dart` 把持久化只挂在底部「关闭」按钮上，存在一条（甚至多条）不落盘的关闭路径。
- **[x] ① 已修复** — `7baeb34ef`（兜底）+ `fb36559be`（导入即落盘）：根因修（Linus 式：消除「存在丢数据的关闭路径」这个特殊情况，而非给单个入口打补丁）。① **导入即落盘**：`_addLocalDb` 在拷贝+插入内存列表后**当场**调 `widget.onSave(_sources)`，让「导入成功」toast 名副其实、导入后即便直接杀进程或经任何关闭路径退出都不丢（用户反馈「为什么非得关闭再写入」即指此——离散导入动作不该延迟到关闭）。② **兜底保存**：再把 `onSave(_sources)` 从「关闭」按钮专属下沉到 `_AudioSourcesDialogState.dispose()`，覆盖所有关闭路径（关闭按钮 / 点遮罩 / 系统返回 / Esc），让此后的排序/开关/URL 等批量编辑无论怎么退出都持久化；底部「关闭」按钮改为只 `Navigator.pop(context)`。`onSave`=`appModel.setAudioSourceConfigs` 作用于长生命周期 provider、不依赖 widget context，调用安全；保存幂等（导入即存 + 关闭再存等值），孤儿副本随即被纳入引用、`pruneOrphans` 不再回收。`flutter analyze` 改动文件 **0 issue**、`dart format` 通过。
- **[x] ② 已加自动化测试** — `fb36559be` · `hibiki/test/pages/audio_sources_dialog_page_test.dart`（widget 行为，最强可落地层）两例：①「persists immediately at import time, before any dialog close」——导入本地库后**不做任何关闭动作**即断言 `onSave` 已被调用且收到该 localAudio 源（守护导入即落盘）；②「persists when the dialog is dismissed by the barrier (not the close button)」——导入后**点遮罩**关闭（非「关闭」按钮）仍断言已落盘（守护 dispose 兜底）。red→green 实测：临时去掉 `_addLocalDb` 的 `widget.onSave(_sources)` → 导入即落盘测试**红**；恢复后该文件 **11 测试全绿**。（注：此前曾被并发 v16 book-key 迁移致 `lib` 不编译而暂不可执行，迁移收口后已正常运行。）
- **备注**：音频/本地音频导入/对话框持久化类。改动文件 `flutter analyze` 0 issue、`dart format` 通过、该测试文件 11 例全绿（含 red→green 验证）。**真机肉眼复测原始失败路径待用户**（任意平台）：设置→管理音频来源→添加本地音频数据库选一个库→**用点遮罩/系统返回退出**（不点「关闭」按钮）→ 再次打开对话框，本地库应仍在列表且生效（查词点 ♪ 能播本地音频）；修复前这样退出会导致导入丢失。
