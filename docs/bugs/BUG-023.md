## BUG-023 · 阅读器内调整字体大小后页面错位、最上一行被裁切
- **报告**：2026-06-04（用户，附截图：分页阅读器两列正文，两列最上一行的文字顶部被横向切掉一截）。
- **真实性**：✅ **真 bug（live 样式变更后不重锚分页）**。字体大小经 `ReaderHibikiSource.setTtuFontSize`（`reader_hibiki_source.dart:547`）触发 `onSettingsChangedLive` → `_ReaderHibikiPageState._applyStylesLive`（`reader_hibiki_page.dart:1387`）。旧实现只把新 CSS 写进 `#hoshi-reader-style` 的 `el.textContent`（`:1422`）就完事。但分页排版里 `body` 的 `font-size` 与 `column-gap`（gap 公式含 `fontSize px`，`reader_content_styles.dart:94/317`）一变，整页就重新分栏：`paginationMetrics`（progressStops/maxScroll）失效、列距 pitch 改变、`body` 的 `scrollLeft` 仍停在重排前的像素值 → 落在新 pitch 的页中间；且 `html,body{overflow:hidden}` 下重排过程把 `documentElement` 的 scrollTop 顶起后无人清（`registerSnapScroll` 只在 `scroll` 事件里 `lockRootViewport`，改 CSS 不发 scroll）→ 内容整体上移，最上一行被裁。对照组：`updatePageSize`（`reader_pagination_scripts.dart`）与 `setChromeInsets`（HBK-REG-004）这类同样会重排的入口都有「重排前捕捉进度 → 换布局 → 失效 metrics → rAF 重锚到分页边界」机制，唯独外部 live CSS 变更没有。根因 = `_applyStylesLive` 缺少与 `updatePageSize` 对称的重排后重锚。行间距 / 余白等同样经 `onSettingsChangedLive` 的设置同源同病。
- **[x] ① 已修复** — 在 `reader_pagination_scripts.dart` 的分页 + 连续两模式各加 `window.hoshiReader.reanchorAfterStyleChange(styleEl, css)`，精确镜像各自 `updatePageSize` 的重锚序列：未 `didInitialize` 时仅 `styleEl.textContent = css`（初始 restore 尚未发生，无需重锚）；已初始化则「`calculateProgress()` 捕捉进度（旧布局/旧 metrics）→ 换 textContent → 分页版 `paginationMetrics = null` + 两模式都按新内容盒重算 `--hoshi-image-max-*` → 复用 `_reanchorPending` in-flight 串行 → rAF 内分页版 `scrollToProgressPaged` / 连续版 `scrollToProgressContinuous` 重锚」。`scrollToProgressPaged`→`setPagePosition`→`assignPagePosition`→`lockRootViewport` 顺带清掉残留的 root scrollTop/Left，同治错位与裁切。Dart 侧 `_applyStylesLive` 注入的 JS 改成：`hoshiReader` 存在且有该方法时走 `reanchorAfterStyleChange(el, css)`，否则（pagination 未就绪 / 非 reader 页）回退裸 `el.textContent`。复用既有机制、消除「外部 CSS 变更无重锚」这一特殊情况，非补丁式绕过。
- **[x] ② 已加自动化测试** — ① `test/reader/reader_pagination_scripts_test.dart` 新增「reanchorAfterStyleChange (BUG-023)」组：断言两模式 shell 都定义 `reanchorAfterStyleChange`，且分页版方法体含 `calculateProgress()` / `this.paginationMetrics = null` / `_reanchorPending` / `scrollToProgressPaged`，连续版含 `calculateProgress()` / `_reanchorPending` / `scrollToProgressContinuous`。② `test/reader/reader_live_settings_guard_test.dart` 新增源码扫描守卫：`reader_hibiki_page.dart` 必须含 `reanchorAfterStyleChange`（谁把 `_applyStylesLive` 退回只 `el.textContent = css` 即红）。两文件 5 条断言修复前全红、修复后全绿（非同义反复）。全量 `flutter test` 2082 绿无回归，`flutter analyze` 0 issue，`dart format` 0 改动。
- **备注**：reader/WebView/布局类。代码 + 单测已绿；最强可落地层是 JS 生成器断言 + 源码扫描守卫（reader 页含真实 `InAppWebView` 平台视图无法 widget 挂载，分页重排/裁切是 WebView 内渲染、无法单测真实像素）。**真机肉眼复测原始失败路径**（开分页 EPUB → 阅读中途调大/调小字体 → 确认正文不错位、最上一行不被裁、停在原阅读位置；并复测连续模式、行间距/余白调整、有声书跟随）待用户后补。`reanchorAfterStyleChange` 的 `calculateProgress` 在 `textContent` 替换前调用（对旧布局/旧 metrics 取进度），rAF 在重排后重锚，与 `updatePageSize` 时序一致。
