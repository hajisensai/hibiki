## BUG-187 · Anki 字段映射「选值」弹窗太小（选项区被死压在屏高 24% / 封顶 320px）
- **报告**：2026-06-11（用户 TODO-103：「视频的 anki 制卡能自定义吗」→ 澄清「字段映射，还有这个选择 X 的值的弹窗好小，小的可怜」）
- **真实性**：✅ 真 bug（UI 可用性）。
  - 字段映射本身**已可自定义**：Anki 设置页 `hibiki/lib/src/pages/implementations/anki_settings_page.dart:260` `_buildFieldMappings` 遍历所选 note type 的每个字段，逐字段一行可点开 `_showHandlebarPicker`（`:275`）选「值来源」（handlebar 模板 `{expression}`/`{reading}`/`{sentence}`/`{audio}`/`{book-cover}` 等）。映射存全局 `AnkiSettings.fieldMappings`（`packages/hibiki_anki/lib/src/anki_models.dart:84`，`Map<anki字段名, handlebar模板>`），reader/book/video 制卡共用一套（渲染走 `AnkiHandlebarRenderer.render` `anki_models.dart:276`）。所以「视频能自定义吗」答案是已能；视频不需要也没有独立字段映射路径。
  - 真正的缺陷是那个「选择 X 的值」弹窗 `AnkiHandlebarPickerDialog`（`anki_settings_page.dart:377`）太小。根因在其 `build`（原 `:413`）：`final maxHeight = (MediaQuery.of(context).size.height * 0.24).clamp(56.0, 320.0);`，再用 `ConstrainedBox(maxHeight: maxHeight)`（原 `:438`）把整个 body（搜索框 + 十几~三十个选项的 ListView）死压在屏高 24% / 封顶 320px。
  - 这与外层 `HibikiDialogFrame(maxHeightFactor: 0.96)`（`:418`，弹窗可达屏高 96%）彻底矛盾：内层 320px 封顶让弹窗实际只有一点点高。`forTermDictionaries`（`anki_models.dart:406`）的核心 18 项 + 每个词典一项，选项常 20+；在 800 高的设备上选项区仅 ~192px（0.24×800），所以「小得可怜」。
- **[x] ① 已修复** — 删掉反常的 `(height*0.24).clamp(56,320)` 内层 ConstrainedBox，弹窗高度改由外层 `HibikiDialogFrame(maxHeightFactor: 0.96)` + `HibikiModalSheetFrame` 的 `Flexible` body 自然填满并滚动（sheet **不设** `maxHeightFactor`，留 null 不叠加内层约束，见 `hibiki_material_components.dart:868-869`）。搜索框固定，选项 ListView 在更大空间滚动；小窗口（如 320×240）因不再叠加内层封顶约束而不溢出。只动这个弹窗 widget，字段映射机制 / 两 backend 渲染 / video 路径均不变。文件 `hibiki/lib/src/pages/implementations/anki_settings_page.dart`。提交：090097f16。
- **[x] ② 已加自动化测试** — `hibiki/test/pages/anki_settings_page_test.dart`：保留「320×240 小窗口不溢出 + 有 TextField」回归用例；新增「800×1200 高窗口下 `HibikiModalSheetFrame` 渲染高度 > 500px」断言（远超旧 320px 上限），撤修复（恢复 320 封顶）则该断言转红。真实 AnkiConnect/AnkiDroid 不参与本弹窗（纯本地选项列表），widget 测试可稳定复现。
- **备注**：真机/真模拟器复测「Anki 设置→选 note type→点字段→选值弹窗变大、选项好点」待用户设备验证（CLAUDE.md 布局类验证纪律）。「字段映射能自定义」无代码改动（本就存在），本条只放大其「选值」子弹窗。
