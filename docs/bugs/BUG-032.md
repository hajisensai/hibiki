## BUG-032 · 歌词模式播放中进程被杀，音频进度归零
- **报告**：2026-06-04（用户：「歌词模式播放的时候，如果进程被杀了，音频进度会归零，修复」）。
- **真实性**：✅ **真 bug（生命周期 flush 不对称）**。沿真实路径验证，控制器层本身没坏——用 fake just_audio 平台实测两条基线均绿：`load(initialPositionMs: 65000)` 后 `controller.position == 65000`（恢复正确，无 0 覆盖）、播放中周期保存把推进位置经 `onPositionWrite` 写出新值。根因在「退到后台→被杀」这条生命周期：① 正常关闭走 `dispose()` → `_audiobookController.dispose()` → `_maybeSavePosition(force:true)` 会强制保存音频位置；② 硬杀（侧滑清后台 / OOM）`dispose` 根本不执行，只剩 `_updateCurrentCue` 里每整秒一次的周期保存，而 `onPositionWrite` 是 **fire-and-forget**（写库 Future 没人 await，可能在进程被回收前还没 commit），且退到后台后 `createPositionStream` 的 Dart 周期 timer 挂起、周期保存本身也停了；③ 后台生命周期 `didChangeAppLifecycleState(paused/inactive)` 共享的 `_syncAndFlushPosition`（HBK-AUDIT-122）**只 flush 了阅读文本位置 `reader_pos`，从不 flush 音频位置**。歌词模式独中招：阅读模式的文本位置由独立的 `reader_pos` 恢复（HBK-AUDIT-122 已在后台 flush 它），掩盖了音频位置丢失；而歌词模式整屏可视进度 = 音频 cue 位置（入口 cue = `allBookCueIdx`，由 `_currentCue` 推出），音频位置一丢就是「归零」。根因点 `reader_hibiki_page.dart` 的 `_syncAndFlushPosition`（缺音频 flush）+ `AudiobookPlayerController.onPositionWrite`（无 await 不可落库保证）。
- **[x] ① 已修复** — `5757d5b19`：对称地把 HBK-AUDIT-122 的后台 flush 扩展到音频位置（消除「reader_pos 后台被 flush、audio_pos 不被 flush」这个不对称特殊情况，非补丁）：(a) `AudiobookPlayerController` 把 `onPositionWrite` 签名改为返回 `Future<void>`，周期路径 `_maybeSavePosition` 仍 `unawaited(...)` fire-and-forget，新增公开的 `flushPosition()` 强制读当前位置并 **await 到真正落库**（绕过整秒节流，保证退到后台那一刻的进度写穿）；(b) `reader_hibiki_page` 两条控制器路径的 `onPositionWrite` 改用 `=>` 转发 `updatePositionMs` 的 Future（原来 `{ ... }` 吞返回值无法 await），并在 `_syncAndFlushPosition` 末尾 `await _audiobookController?.flushPosition()`，与既有的 `_flushPosition()`（reader_pos）并列。`flutter analyze` 0 issue、`dart format` 0 改动。
- **[x] ② 已加自动化测试** — 两层守卫：① `hibiki/test/media/audiobook/audiobook_position_flush_test.dart`——用 fake just_audio 平台跑真控制器：两条基线（load 恢复 savedMs / 预热 cue 不用 0 覆盖）+ `flushPosition` 即使整秒未变也 force 写最新位置 + `flushPosition` 返回的 Future **await 到写库完成**（Completer 卡住验证 durability，旧的 fire-and-forget 写法这条会红）；② `hibiki/test/media/audiobook/audio_lifecycle_flush_wiring_static_test.dart`——源码守卫钉 `_syncAndFlushPosition` 体内含 `_audiobookController?.flushPosition()`、两条路径都用 `=> ...updatePositionMs(` 箭头转发返回值。`test/media/audiobook/` + `test/pages/` 全量 442 绿无回归，`hibiki_audio` 包测试绿。
- **备注**：reader/播放/生命周期类。代码 + 单测已绿；真机复测原始失败路径（歌词模式播放 → 侧滑清后台杀进程 → 重开书确认音频进度续上而非归零）待用户后补。同时也覆盖普通有声书模式被杀后音频位置的可靠落库（之前同样依赖 dispose/未落库的周期保存）。
