## BUG-453 · Windows 全局查词弹窗渲染与 app 内不一致(竖排级联硬编码)
- **报告**：2026-06-28（用户：）
- **真实性**：✅ 真 bug（范围已收窄）。先纠正前提：popup.js 的**渲染样式入参**（主题/ColorScheme、内容 zoom=appUiScale×字号、词典字体、最大高度、pitch/freq 去重、折叠/隐藏过滤、自定义 CSS、外字嵌入）两端早已经 TODO-895 收口到单一真相源 `buildPopupSettingsJs`（`hibiki/lib/src/pages/implementations/popup_settings_injection.dart:138`），app 内/外**已一致**，不再缺失。真正残留的硬编码只剩级联布局层的竖排标志：`global_lookup_controller.dart:624` 写死 `isVertical: false`（默认值同样写死 `global_lookup_render.dart:76`）。该值唯一去向是 `computeFrameRect`（决定多级查词卡片往上下还是左右弹），竖排书查词时子卡片级联方向永远按横排算。注意 popup.js 本身不做竖排排版，故词条内部渲染不受影响，受影响的仅是嵌套卡片弹出方位。
- **[x] ① 已修复** — 新增纯函数 `isVerticalFromWritingMode(String?)`（`hibiki/lib/src/lookup/global_lookup_render.dart`，`vertical-rl`/`vertical-lr`→true，横排/ null→false，与阅读器 `reader_content_styles.dart:150` 同判据）；`global_lookup_controller.dart` `_renderStack` 从 `ReaderHibikiSource.readerSettings?.writingMode`（与阅读器同一接口）取竖排，替换硬编码 `false`（无活动阅读器/无书时 null→false 兜底，行为不回退）。提交哈希：见 commit。
- **[x] ② 已加自动化测试** — `hibiki/test/lookup/global_lookup_popup_style_guard_test.dart` 新增 `group('TODO-938 — vertical cascade wired from writingMode')`：①纯函数行为（vertical-rl/vertical-lr→true、horizontal-tb/null/空→false）②源码守卫断言 `global_lookup_controller.dart` 不再 `isVertical: false` 硬编码且读取 `readerSettings?.writingMode` 经 `isVerticalFromWritingMode`。red→green 见提交。
