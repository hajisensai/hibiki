## BUG-068 · Windows（及所有 Material 平台）中文等非日语 UI 字体发怪：整个 app 界面被钉死用日语字体 + ja locale 渲染，汉字显示成日文字形
- **报告**：2026-06-06（用户：「win 的中文的语言的字体很怪，用了日语字体然后 fallback？为什么字体能不一样」）。采番注：动手前查 develop 当前最大号 BUG-067，本条取 068。
- **真实性**：✅ **真 bug（app-chrome 字体/渲染 locale 与显示语言解耦，被永久钉死成日语）**。沿真实代码路径定位：① `app_model.dart:622` 旧 `textStyle` 写死 `fontFamily: appFontFamily ?? targetLanguage.defaultFontFamily` + `locale: targetLanguage.locale` + `textBaseline: targetLanguage.textBaseline`，而 `targetLanguage`（`:1492` `languages.values.first`）**永久是 `JapaneseLanguage`**（仓库只剩日语 Language，中文/英文类早删），`JapaneseLanguage.defaultFontFamily='NotoSansJP'`（`japanese_language.dart:32`）、locale=`ja`。这个 `textStyle` 灌进整个 Material `textTheme`（`:631` 全 15 个命名样式）→ ThemeNotifier（`:1111`）。② 界面**显示语言** `appLocale`（`:1509`，Slang 用它，能到 zh-CN/zh-HK）与**字体/渲染 locale**（targetLanguage=ja）是两套独立值——切中文界面，字体仍是日语。③ `NotoSansJP` **从未被 bundle/注册**：`pubspec.yaml` 无 `fonts:` 声明、无 `GoogleFonts.notoSansJp()` 加载（`custom_fonts_page.dart:98` 只是可选 CDN 下载 URL），故是**幽灵 family** → 引擎落平台 fallback，而 `locale:ja` 提示让 Windows DirectWrite/Skia 把汉字码点解析到**日文系统字体**（Yu Gothic/Meiryo）。叠加 **CJK 汉字统一**（中日同码点不同字形），中文 UI 被日文字形渲染。iOS/macOS 的 Cupertino 路径（`main.dart:591` 只传 `appFontFamily`，无 ja 钉死）本就规避了此 bug，故症状是 Material 平台（Android/Windows/Linux）特有、Windows 最明显。阅读器/词典查词的日语正文走 WebView 自带 CSS 字体，**不经**这个 textTheme，故不受影响也不应被波及。
- **[x] ① 已修复** — 提交 `0418a0359`。根因修（消除「整个 UI 永远日语」这个特例，让 app-chrome 字体跟显示语言走、内容字体各自声明）：`app_model.dart` `textStyle` 改为 `locale: appLocale`（显示语言，带 region 故能区分简繁字形）、`fontFamily: appFontFamily`（用户自定义字体优先，否则 null → 由平台按 UI locale 解析正确区域字形，不再钉 NotoSansJP）、`textBaseline` 由新纯函数 `_isIdeographicLocale(uiLocale)`（CJK→ideographic，其余→alphabetic）驱动而非永远 ideographic。**不动** `main.dart:621` 的 `MaterialApp.locale=targetLanguage.locale`（保持 LTR，避免 ar 显示语言意外触发整 app RTL 翻转，超出本 bug 范围）——15 个命名 textTheme 样式已显式带 `locale: appLocale`，字形选择以显式 TextStyle.locale 为准，足以修复症状。时序安全：`textStyle` 原就依赖 `languages`(init `:1152`)，新增的 `locales`(`:1153`) 在同一初始化窗口紧邻填充，theme 在 init 后才读，无新增 LateInit 风险。WebView 日语正文不受影响（自带 CSS 字体）。`flutter analyze` 0 issue。
- **[x] ② 已加自动化测试** — `hibiki/test/models/app_ui_font_locale_guard_test.dart`（源码扫描守卫，3 例）。`AppModel` 需活 Drift DB 难廉价实例化，源码扫描是钉住此 wiring 的最强可落地层：断言 `textStyle` getter 体内 `locale: uiLocale` + `final Locale uiLocale = appLocale;`、`fontFamily: appFontFamily,`、`_isIdeographicLocale(uiLocale)`，且**不含** `targetLanguage.locale` / `targetLanguage.defaultFontFamily` / `targetLanguage.textBaseline` / `NotoSansJP`（防回退到日语钉死）。**red→green 实测**：stash 掉修复跑旧实现 → 全红（旧 getter 是 `=> TextStyle(` 不含新标记，且内容断言命中 `targetLanguage.locale`），还原 → 3 绿。`test/models/` 全量 226 例绿（`test/goldens/`+`test/models/` 混跑偶现 1 例 `app_model_audio_sources_test` 失败，单独跑 3 例全绿 → 跨套件共享状态污染，与本改动无关）。
- **备注**：UI 字体/i18n/平台渲染类。修复纯 Dart、CI 可测。**真机肉眼复测原始失败路径待用户**（Windows + 界面语言设为简体中文）：app 界面（按钮/菜单/设置/书架）汉字应显示为**中文字形**而非日文字形；切 zh-HK 应见繁体字形；切回日语界面仍是日文字形；阅读器内日语正文不变。
- **遗留（opus 审登记，非阻断，单 ambient locale 的固有对称取舍）**：① **以 Flutter widget（非 WebView）themed Text 渲染的日语内容**，在非日语 UI 下会跟随 UI locale 取 SC/TC 字形。审查确认全 app 唯一**活跃**点是**书架书名** `reader_hibiki_history_page.dart:822` 的 `_titleOverlay`（`textTheme.labelSmall` 渲染 `book.title`）——中文界面下日文书名汉字可能偏中文字形，仍同字、可读，用户已确认「用系统的就行」；`getTermReadingOverrideWidget`/`getPitchWidget`/`RubyText`/`DictionaryTermPage`/`DictionaryResultPage` 整条经核实是**死代码**（零调用点）故不受影响。若日后要书名等保留精确日文字形，给这些特定 widget 单独钉 ja content style 即可（独立小改）。② **未动 `main.dart:621` `MaterialApp.locale=targetLanguage.locale`（ja）**：故 `Localizations.localeOf=ja`，凡用裸 `TextStyle()` 不带 locale 的 themed Text 仍按 ambient ja 解析字形——审查确认当前唯一这样的渲染器 `getPitchWidget` 是死代码，无活跃命中；不改是为避免 ar 显示语言意外触发整 app RTL 翻转（超出本 bug 范围），15 个命名 textTheme 样式已显式带 `appLocale` 足以修复症状。
