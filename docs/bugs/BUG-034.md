## BUG-034 · 谷歌云盘（Google Drive 同步）桌面端重启后掉登录
- **报告**：2026-06-04（用户：「我的谷歌云盘为什么会掉登录，修复一下」。澄清确认：**Windows 桌面端、重启时掉**）。采番注：初记为 BUG-032，与并发 agent 的歌词模式 BUG-032（`8fdca9f4d` 先占号）冲突，让号改记 BUG-034。
- **真实性**：✅ **真 bug，主因确定性、根因在桌面 OAuth 流程从不申请 offline 访问**。沿真实代码路径定位：桌面端（`useMobileAuth==false`）`GoogleDriveAuth.authenticate`（`google_drive_auth.dart`，修复前）用 `auth_io.clientViaUserConsent` 完成授权。核 googleapis_auth 2.3.1 源码：`clientViaUserConsent → AuthorizationCodeGrantServerFlow → authenticationUri`（`authorization_code_grant_abstract_flow.dart:46`）调 `createAuthenticationUri` 时**从不传 `offline`**，全库唯一能加 `access_type=offline` 的地方（`auth_code.dart:38`）无任何 `offline:true` 调用点 → 授权 URL 不带 `access_type=offline` → **Google 根本不下发 refresh token**。于是 `authenticate` 拿到的 `credentials.refreshToken == null`，持久化存的也是 null。会话内靠 access token 的 ~1 小时寿命能用；**一重启内存 `_desktopClient` 没了，`restoreDesktopAuth`（`google_drive_auth.dart:149`）命中 `if (credentials.refreshToken == null) return false` → 每次重启必然恢复失败 = 掉登录**（与用户「重启时候掉的」完全吻合，确定性而非概率）。`refreshCredentials` 本身正确保留 refresh token（`auth_functions.dart:190`），所以问题不在刷新而在「从一开始就没拿到 refresh token」。**次因（加重项）**：`restoreDesktopAuth` 的 catch 块对**任何**异常都 `clearDesktopSession()` 把持久化凭据永久擦除——包括纯网络超时（GFW 直连 Google 常见，`authenticate` 注释自己都写了 "blocked-direct-connection timeout"），一次启动瞬时直连失败就把仍有效的会话抹掉。根因点 = 桌面授权 URL 缺 `access_type=offline`（主）+ restore 把瞬时网络错误误判为凭据失效（次）。
- **[x] ① 已修复** — `a1123f470`（OAuth 根因修）+ `3b1be016a`（opus 审查修复）：根因修，复用 app 自有的 OneDrive/Dropbox 同款 loopback 流程，不再走不申请 offline 的 googleapis_auth 便捷封装。(a) 桌面 `authenticate` 改用 `runDesktopOAuthLoopback`（`desktop_oauth.dart`）+ 自建授权 URL `_buildDesktopAuthUrl`（`Uri.https('accounts.google.com','o/oauth2/v2/auth', …)` 带 `access_type=offline` + `prompt='consent select_account'` + PKCE S256），换码用导出的 `auth_io.obtainAccessCredentialsViaCodeExchange`（其响应解析 `refresh_token`）→ 拿到**带 refresh token** 的 `AccessCredentials`，再喂给原有正确的持久化/`restoreDesktopAuth`/`refreshCredentials` 机制 → 重启可恢复；`prompt=consent` 保证账号已授权过时也仍下发 refresh token，`select_account` 让多账号用户可选号；并对 `refreshToken==null` 早退不落库（消除「持久化一个永远无法恢复的会话」这个特殊情况）。(b) `restoreDesktopAuth` 的 catch 把无条件 `clearDesktopSession()` 改为 `if (_isCredentialsRejected(e))` 门控——只在 Google 真拒（`ServerRequestFailedException` 400/401/403 或 `AccessDeniedException`）时清会话，瞬时网络/5xx 保留凭据待下次启动重试（与 OneDrive `restoreAuth` 不擦持久化的范式一致）。顺手移除不再使用的 `url_launcher` import。**opus 审查修复**：(C-1) `auth.authenticatedClient` 默认 `closeUnderlyingClient:false`，原代码 `_desktopClient?.close()` 只关壳、底层 `http.Client` 泄漏（`refreshAuth` 每小时漏一个）→ 三处 `authenticatedClient` 均加 `closeUnderlyingClient: true`（`http.IOClient.close` 幂等，无双关闭）；(W-2) `_isCredentialsRejected` 把 403（撤销/项目禁用）也纳入致命，避免「每次启动刷新失败但不清库」的轻微死循环；(W-1) prompt 加 `select_account`。`flutter analyze` 0 issue、`dart format` 0 改动。
- **[x] ② 已加自动化测试** — `hibiki/test/sync/google_drive_desktop_auth_test.dart`（12 例，全绿）：OAuth 网络往返无法单测，故钉两个决策点 + 源码守卫——① `debugBuildDesktopAuthUrl` 断言 URL 命中 `accounts.google.com/o/oauth2/v2/auth`、`access_type=offline`、`prompt` 含 `consent`、`code_challenge_method=S256`、携 loopback redirect 与 drive.file scope（任何回退到无 offline 的授权 URL 即红）；② `debugIsCredentialsRejected` 断言 400/401/403/AccessDenied 判为「真失效→清会话」，`SocketException`/503 判为「瞬时→保留会话」（旧的 catch-all 擦库语义下这两条会红）；③ 源码守卫钉 `clientViaUserConsent(` 不再出现、`'access_type': 'offline'`/`'prompt': 'consent`/`obtainAccessCredentialsViaCodeExchange`/`runDesktopOAuthLoopback`/`if (_isCredentialsRejected(e)) {` 均在位。`test/sync/` 全量 264 绿无回归。
- **备注**：同步/OAuth/桌面类。代码 + 单测绿。**真机复测原始失败路径待用户**：Windows 上设置→同步选 Google Drive→登录（会弹一次同意屏）→**完全退出 app 再重开→仍保持登录、能继续同步**（修复前必掉）；并复测断网启动一次再联网启动不被登出。**移动端（Android/iOS）不受本改动影响**：走 google_sign_in 原生路径、refresh token 由系统托管，本次只改桌面分支。**另注**：若用户报「全平台每隔约 7 天必掉」，则属 Google Cloud「OAuth 同意屏」仍是 Testing 发布状态（refresh token 7 天强制过期）的云端配置问题，需在 Google Cloud Console 把同意屏发布到 Production，非本代码可修——本轮未观测到该规律，未处理。
