# 持续审查与回归记录

> [CLAUDE.md](../../CLAUDE.md) 的子文档。

## 持续审查模式

- 用户要求审查项目、继续审查、风险审计或类似任务时，默认进入**持续审查模式**；不要只在聊天里输出一次性总结。
- 审查报告写入 `docs/reviews/YYYY-MM-DD-project-review.md`。目录不存在先创建 `docs/reviews/`。
- 每轮审查**追加**到同一报告文件，不覆盖历史。每轮至少包含：
  - `Scope`：本轮检查的文件、路径、提交范围或用户路径。
  - `Findings`：按 `HBK-AUDIT-XXX` 编号列问题；每个含 `severity`、`status`、文件/行号、根因、影响、修复建议、验证方式。
  - `Next Scope`：下一轮继续审查的范围。
- 审查顺序默认**按风险**走，而不是按提交或文件名散步：数据库/迁移 → 启动初始化 → 阅读器状态 → 字典导入/native FFI → 音频 cue → WebView/缓存 → UI 假状态。
- 审查阶段**只写报告和修复建议，不改业务代码**；除非用户明确要求「开始修」「逐条修」或等价指令。
- 报告结论必须区分「代码路径审查发现的风险」「已经复现的 bug」「已验证通过的修复」。没跑过验证，不要写成已通过。

## 回归记录

- 测试发现的 bug 必须记录到 `docs/REGRESSION_BUGS.md`（本地文件，已 `.gitignore`，不入库）；以后安装包测试和阅读器手工测试要先复测其中所有 `open` 项。
- 审查或手工验证发现已复现回归，必须同步更新 `docs/REGRESSION_BUGS.md`，并把截图、UI XML、logcat 或 bounds 证据放到 `.codex-test/` 后在报告中引用。
- 手工测试的证据留存规范见 [integration-testing.md](integration-testing.md) 的「手工验证与证据留存」。

## 代码审查（code review）

- 审查关注实现是否符合计划、代码质量、边界情况、向后兼容性。审查发现问题，修复后重新提交审查。
